# Pages, Layouts, and Templates - Teaching Notes

## Topic 5: Layout Persistence, Templates, and Advanced Routing Patterns

---

# 1. Why This Topic Matters

Layouts are one of Next.js's most powerful features. They let you share UI — navigation, sidebars, footers — **without re-mounting** when users navigate between pages.

Students often confuse **layouts** with **templates**, misunderstand **route groups**, or over-nest layouts. This topic clarifies:

* How the root layout forms the HTML shell
* Why layouts persist but templates re-mount
* How route groups organize code without affecting URLs
* When parallel and intercepting routes are useful (overview)

Mastering layouts prevents duplicated nav bars and broken UX on navigation.

---

# 2. Root Layout and HTML Shell

Every App Router application **must** have a root layout at `app/layout.tsx`. It is the outermost wrapper and the only place where `<html>` and `<body>` tags belong.

```tsx
// app/layout.tsx
import type { Metadata } from 'next'
import { Inter } from 'next/font/google'
import './globals.css'

const inter = Inter({ subsets: ['latin'] })

export const metadata: Metadata = {
  title: {
    default: 'Training Portal',
    template: '%s | Training Portal',
  },
  description: 'Next.js training course',
}

export default function RootLayout({
  children,
}: {
  children: React.ReactNode
}) {
  return (
    <html lang="en">
      <body className={inter.className}>
        {children}
      </body>
    </html>
  )
}
```

| Root layout responsibility | Detail |
| -------------------------- | ------ |
| `<html>` and `<body>` | Required — only here |
| Global CSS import | `import './globals.css'` |
| Font loading | `next/font` applied to body |
| Metadata defaults | `export const metadata` |
| Providers (optional) | Theme, auth context wrappers |

Compare with react-project-study:

```tsx
// react-project-study — no single HTML shell in React
// index.html provides <div id="root">
// App.tsx wraps routes but does not own <html>/<body>
```

Next.js root layout **is** the HTML document wrapper.

# 3. Nested Layouts with children

Layouts nest automatically. Each layout wraps its child layout or page through the **`children`** prop.

```tsx
// app/dashboard/layout.tsx
import Link from 'next/link'

export default function DashboardLayout({
  children,
}: {
  children: React.ReactNode
}) {
  return (
    <div className="flex min-h-screen">
      <nav className="w-56 bg-gray-100 p-4">
        <ul className="space-y-2">
          <li><Link href="/dashboard">Overview</Link></li>
          <li><Link href="/dashboard/settings">Settings</Link></li>
          <li><Link href="/dashboard/analytics">Analytics</Link></li>
        </ul>
      </nav>
      <main className="flex-1 p-6">{children}</main>
    </div>
  )
}
```

```tsx
// app/dashboard/settings/layout.tsx
export default function SettingsLayout({
  children,
}: {
  children: React.ReactNode
}) {
  return (
    <div>
      <h1 className="text-2xl font-bold mb-4">Settings</h1>
      <div className="border rounded-lg p-4">{children}</div>
    </div>
  )
}
```

```tsx
// app/dashboard/settings/page.tsx
export default function SettingsPage() {
  return <p>Manage your account preferences here.</p>
}
```

### Render tree for `/dashboard/settings`

```text
RootLayout
  └── DashboardLayout (sidebar nav)
        └── SettingsLayout (settings heading + border)
              └── SettingsPage (content)
```

| Layout level | Persists when navigating to... |
| ------------ | ------------------------------- |
| Root | All pages |
| Dashboard | `/dashboard`, `/dashboard/settings`, `/dashboard/analytics` |
| Settings | Only `/dashboard/settings/*` child pages |

# 4. Layout Persistence Across Navigation

The key difference between layouts and regular components: **layouts do not unmount** when navigating between sibling routes.

```text
User on /dashboard/settings
  → DashboardLayout mounted (sidebar visible)
  → SettingsLayout mounted
  → SettingsPage mounted

User clicks "Analytics" → /dashboard/analytics
  → DashboardLayout STAYS mounted (sidebar still there, no flicker)
  → SettingsLayout unmounts
  → AnalyticsPage mounts
```

| Behavior | Layout | Regular component in page |
| -------- | ------ | ------------------------- |
| State preserved on sibling navigation | ✅ Yes | ❌ No — page remounts |
| Shared chrome (nav, sidebar) | ✅ Ideal use case | ❌ Would re-render fully |
| Fetch data once for section | ✅ Can fetch in layout | ❌ Re-fetch per page |

### Example: sidebar state persists

```tsx
// app/dashboard/layout.tsx
'use client' // Only if you need interactive state in layout

import { useState } from 'react'
import Link from 'next/link'

export default function DashboardLayout({
  children,
}: {
  children: React.ReactNode
}) {
  const [isCollapsed, setIsCollapsed] = useState(false)

  return (
    <div className="flex">
      <nav className={isCollapsed ? 'w-12' : 'w-56'}>
        <button onClick={() => setIsCollapsed(!isCollapsed)}>
          Toggle
        </button>
        <Link href="/dashboard">Home</Link>
      </nav>
      <main>{children}</main>
    </div>
  )
}
```

If the user collapses the sidebar and navigates from Settings to Analytics, the sidebar **stays collapsed** because the layout did not unmount.

In react-project-study, achieving this requires lifting state to a parent route component or context — layouts make it natural.

# 5. Templates vs Layouts

**Templates** (`template.tsx`) look like layouts but behave differently: they **re-mount on every navigation**, creating a fresh component instance.

```tsx
// app/dashboard/template.tsx
export default function DashboardTemplate({
  children,
}: {
  children: React.ReactNode
}) {
  return (
    <div className="animate-fade-in">
      {children}
    </div>
  )
}
```

| Feature | `layout.tsx` | `template.tsx` |
| ------- | ------------ | -------------- |
| Persists on navigation | ✅ Yes | ❌ Re-mounts each navigation |
| Preserves state | ✅ Yes | ❌ State resets |
| Creates new DOM instance | ❌ No | ✅ Yes |
| Use case | Nav, sidebar, providers | Enter animations, per-page effects |
| Can coexist | ✅ Both wrap the same segment | ✅ Renders inside layout |

### Render order with both

```text
Layout
  └── Template
        └── Page
```

```tsx
// app/dashboard/layout.tsx — persists
export default function DashboardLayout({ children }) {
  return (
    <div>
      <aside>Sidebar (persists)</aside>
      {children}
    </div>
  )
}
```

```tsx
// app/dashboard/template.tsx — re-mounts
'use client'

import { useEffect } from 'react'

export default function DashboardTemplate({ children }) {
  useEffect(() => {
    console.log('Template mounted — runs on every navigation')
  }, [])

  return <div className="fade-in">{children}</div>
}
```

### When to use templates

| Scenario | Use template? |
| -------- | ------------- |
| Page enter animation on every visit | ✅ Yes |
| Reset form state when leaving and returning | ✅ Yes |
| Shared navigation sidebar | ❌ Use layout |
| Auth provider wrapping section | ❌ Use layout |

Most apps use **layouts only**. Templates are for specific UX patterns.

# 6. Route Groups with (folder)

**Route groups** organize routes **without affecting the URL**. Wrap a folder name in parentheses:

```text
app/
├── (marketing)/
│   ├── layout.tsx       # Marketing layout — no /marketing in URL
│   ├── page.tsx         # → /
│   └── about/
│       └── page.tsx     # → /about
├── (shop)/
│   ├── layout.tsx       # Shop layout
│   ├── products/
│   │   └── page.tsx     # → /products
│   └── cart/
│       └── page.tsx     # → /cart
└── layout.tsx           # Root layout
```

| Without route groups | Problem |
| -------------------- | ------- |
| `app/marketing/about/page.tsx` | URL becomes `/marketing/about` — unwanted |
| Multiple root layouts needed | Cannot have two `app/layout.tsx` files |

| With route groups | Solution |
| ----------------- | -------- |
| `app/(marketing)/about/page.tsx` | URL is `/about` — group name omitted |
| `app/(shop)/layout.tsx` + `app/(marketing)/layout.tsx` | Different layouts per section |

### Different layouts for different sections

```tsx
// app/(marketing)/layout.tsx
export default function MarketingLayout({ children }) {
  return (
    <div>
      <header className="bg-blue-600 text-white p-4">
        Marketing Site
      </header>
      {children}
    </div>
  )
}
```

```tsx
// app/(shop)/layout.tsx
export default function ShopLayout({ children }) {
  return (
    <div>
      <header className="bg-green-600 text-white p-4">
        Shop — Cart | Account
      </header>
      {children}
    </div>
  )
}
```

Both groups share the root `app/layout.tsx` but have different section headers.

### Multiple root layouts (advanced)

Route groups can define **different root-level layouts** by using groups at the top level:

```text
app/
├── (main)/
│   ├── layout.tsx    # Root layout variant A
│   └── page.tsx
└── (auth)/
    ├── layout.tsx    # Root layout variant B (minimal — no nav)
    ├── login/
    │   └── page.tsx
    └── register/
        └── page.tsx
```

Each group has its own layout that includes `<html>` and `<body>` when using multiple root layouts pattern.

# 7. Parallel Routes Overview

**Parallel routes** render multiple pages in the same layout simultaneously using named **slots**.

```text
app/
└── dashboard/
    ├── layout.tsx
    ├── page.tsx
    ├── @analytics/
    │   └── page.tsx
    └── @team/
        └── page.tsx
```

```tsx
// app/dashboard/layout.tsx
export default function DashboardLayout({
  children,
  analytics,
  team,
}: {
  children: React.ReactNode
  analytics: React.ReactNode
  team: React.ReactNode
}) {
  return (
    <div>
      <div>{children}</div>
      <div className="grid grid-cols-2 gap-4">
        <section>{analytics}</section>
        <section>{team}</section>
      </div>
    </div>
  )
}
```

| Concept | Detail |
| ------- | ------ |
| Slot folder | `@analytics` — named with `@` prefix |
| Prop name | `@analytics` → `analytics` prop in layout |
| Independent loading/error | Each slot can have its own `loading.tsx` and `error.tsx` |
| URL | Still `/dashboard` — slots do not add URL segments |

Use cases: dashboards with independent panels, split views, modals alongside main content.

This is an **overview** — full parallel routes patterns are advanced and covered in later production topics.

# 8. Intercepting Routes Overview

**Intercepting routes** show a route inside the current layout — like a modal overlay — while keeping the URL of the intercepted route.

```text
app/
├── feed/
│   └── page.tsx
├── photo/
│   └── [id]/
│       └── page.tsx          # Full page at /photo/123
└── @modal/
    └── (.)photo/
        └── [id]/
            └── page.tsx      # Modal when navigating from /feed
```

| Intercept pattern | Meaning |
| ----------------- | ------- |
| `(.)` | Match segment on same level |
| `(..)` | Match one level above |
| `(..)(..)` | Match two levels above |
| `(...)` | Match from root |

Example UX: clicking a photo on the feed opens a **modal** with the photo detail. Navigating directly to `/photo/123` shows the **full page**.

Common for: photo galleries, quick-view product modals, login dialogs.

Students should know this exists; implementation details come with project practice.

# 9. Layout Composition Patterns

### Pattern A — Global shell + section layout

```text
app/layout.tsx              → html, body, global nav
app/(app)/layout.tsx        → authenticated app shell
app/(app)/dashboard/...     → dashboard pages
app/(marketing)/layout.tsx  → public marketing pages
```

### Pattern B — Breadcrumb from nested layouts

```tsx
// app/products/layout.tsx
export default function ProductsLayout({ children }) {
  return (
    <div>
      <nav aria-label="Breadcrumb">
        <ol>
          <li><a href="/">Home</a></li>
          <li><a href="/products">Products</a></li>
        </ol>
      </nav>
      {children}
    </div>
  )
}
```

### Pattern C — Conditional layout content (server)

```tsx
// app/admin/layout.tsx
import { getSession } from '@/lib/auth'
import { redirect } from 'next/navigation'

export default async function AdminLayout({
  children,
}: {
  children: React.ReactNode
}) {
  const session = await getSession()

  if (!session) {
    redirect('/login')
  }

  return (
    <div>
      <p>Signed in as {session.user.email}</p>
      {children}
    </div>
  )
}
```

# 10. Layout vs Page Responsibilities

| Responsibility | Belongs in layout | Belongs in page |
| -------------- | ----------------- | --------------- |
| Site navigation | ✅ | ❌ |
| Sidebar for a section | ✅ | ❌ |
| Page title and main content | ❌ | ✅ |
| Data fetching for page content | ❌ | ✅ (usually) |
| Metadata specific to one URL | ❌ | ✅ |
| Auth guard for a section | ✅ | Sometimes |
| Footer | ✅ (root layout) | ❌ |

Avoid fetching heavy data in layouts if it blocks all child page navigations within that segment.

# 11. Comparison with react-project-study

react-project-study uses a single `App.tsx` with nested routes but no layout persistence concept:

```tsx
// react-project-study/src/App.tsx — simplified
<Routes>
  <Route path="/" element={<Home />} />
  <Route path="/products" element={<Product />} />
  <Route path="/login" element={<Login />} />
</Routes>
```

To share a sidebar across pages, you would wrap routes manually:

```tsx
<Route path="/dashboard" element={<DashboardShell />}>
  <Route path="settings" element={<Settings />} />
  <Route path="analytics" element={<Analytics />} />
</Route>
```

Next.js layouts are the **built-in equivalent** of nested route shells — but defined by file convention, not JSX configuration.

| react-project-study | Next.js App Router |
| ------------------- | ------------------ |
| `<Route element={<Layout />}>` | `layout.tsx` in folder |
| Outlet for child routes | `{children}` prop |
| Layout remount behavior manual | Layout persistence automatic |
| Route groups not built-in | `(folder)` convention |

# 12. Best Practices

✔ Keep **one root layout** with `<html>` and `<body>` unless using multiple root layout groups intentionally

✔ Use **nested layouts** for section-specific chrome (dashboard sidebar, docs nav)

✔ Prefer **layouts over templates** unless you need re-mount behavior

✔ Use **route groups** `(name)` to organize without polluting URLs

✔ Do not duplicate nav bars in every `page.tsx` — lift to layout

✔ Fetch **shared section data** in layout only if all child pages need it

✔ Add `'use client'` to layouts **only** when they need hooks or event handlers

✔ Learn parallel and intercepting routes conceptually before implementing in projects

# 14. Summary

| Concept | Key takeaway |
| ------- | ------------ |
| Root layout | Required HTML shell — `<html>`, `<body>`, global CSS |
| Nested layouts | Stack via `{children}` — share UI per section |
| Layout persistence | Does not unmount on sibling navigation — state preserved |
| template.tsx | Re-mounts on navigation — animations, fresh state |
| Route groups `(name)` | Organize folders without URL segments |
| Parallel routes `@slot` | Multiple pages in one layout — dashboard panels |
| Intercepting routes | Show route as modal overlay from another page |
| vs React Router | File-based layouts replace nested `<Route element>` shells |
