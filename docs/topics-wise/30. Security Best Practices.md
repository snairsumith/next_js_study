# Security Best Practices - Teaching Notes

## Topic 30: Protecting Next.js Apps in Production

---

# 1. Why This Topic Matters

Security failures destroy user trust and company reputation. Next.js apps span server and client — each layer has risks:

* Secrets leaked to the browser via `NEXT_PUBLIC_`
* Unvalidated Server Actions accepting malicious input
* XSS through unsanitized HTML
* CSRF on state-changing requests
* Verbose errors exposing stack traces and database details

Security is not a single library. It is a set of habits applied at middleware, Server Components, actions, Route Handlers, and deployment config.

---

# 2. Server-Only Secrets

Environment variables without `NEXT_PUBLIC_` are available only on the server.

```env
# ✅ Server-only — safe
DATABASE_URL=postgresql://...
AUTH_SECRET=abc123
STRIPE_SECRET_KEY=sk_live_...

# ⚠️ Exposed to browser bundle — never for secrets
NEXT_PUBLIC_API_URL=https://api.example.com
```

```typescript
// ❌ NEVER do this
const apiKey = process.env.NEXT_PUBLIC_STRIPE_SECRET

// ✅ Server Component or Route Handler only
const stripe = new Stripe(process.env.STRIPE_SECRET_KEY!)
```

| Variable prefix | Visible to client? |
| --------------- | ------------------ |
| `NEXT_PUBLIC_*` | Yes — bundled into JS |
| No prefix | Server only |

Use `server-only` package to prevent accidental client imports:

```typescript
// lib/db/client.ts
import 'server-only'

import { PrismaClient } from '@prisma/client'

export const prisma = new PrismaClient()
```

If a client component imports this file, the build fails — catching mistakes early.

# 3. CSRF Considerations with Server Actions

Cross-Site Request Forgery tricks a logged-in user's browser into submitting unwanted requests.

Next.js Server Actions include **built-in CSRF protection** via:

* POST-only semantics for actions
* Origin header verification
* Encrypted action IDs

| Practice | Reason |
| -------- | ------ |
| Use Server Actions for mutations | Framework CSRF checks |
| Avoid sensitive GET routes that mutate data | GET requests are CSRF-able |
| SameSite cookies (`lax` or `strict`) | Limits cross-site cookie sending |

For custom Route Handlers that mutate state:

```typescript
// Verify origin on sensitive POST handlers
export async function POST(request: Request) {
  const origin = request.headers.get('origin')
  const allowed = process.env.NEXT_PUBLIC_APP_URL

  if (origin && allowed && origin !== allowed) {
    return new Response('Forbidden', { status: 403 })
  }

  // proceed
}
```

Do not disable Next.js CSRF protections without understanding the risk.

# 4. Input Validation on the Server

**Never trust client input** — including hidden form fields and JSON bodies.

```typescript
'use server'

import { z } from 'zod'

const updateRoleSchema = z.object({
  userId: z.string().uuid(),
  role: z.enum(['user', 'editor']), // never accept 'admin' from form
})

export async function updateRoleAction(formData: FormData) {
  const session = await auth()
  if (session?.user?.role !== 'admin') {
    throw new Error('Forbidden')
  }

  const parsed = updateRoleSchema.safeParse({
    userId: formData.get('userId'),
    role: formData.get('role'),
  })

  if (!parsed.success) {
    return { error: 'Invalid input' }
  }

  await updateUserRole(parsed.data.userId, parsed.data.role)
}
```

| Validation layer | Validates |
| ---------------- | --------- |
| Client (Zod/RHF) | UX — instant feedback |
| Server Action / API | Security — authoritative |
| Database constraints | Last line of defense |

Client validation is for convenience. Server validation is mandatory.

# 5. Avoiding XSS (Cross-Site Scripting)

XSS injects malicious scripts into pages viewed by other users.

### React escapes by default

```tsx
// Safe — React escapes {user.name}
<p>{user.name}</p>
```

### Dangerous: dangerouslySetInnerHTML

```tsx
// ❌ Only with sanitized HTML
<div dangerouslySetInnerHTML={{ __html: userBio }} />

// ✅ Use a sanitizer library
import DOMPurify from 'isomorphic-dompurify'

<div dangerouslySetInnerHTML={{ __html: DOMPurify.sanitize(userBio) }} />
```

| Risk | Mitigation |
| ---- | ---------- |
| User-generated HTML | Sanitize with DOMPurify |
| URL params in HTML | Let React escape; avoid innerHTML |
| Third-party scripts | CSP headers (below) |
| JWT in localStorage | Prefer httpOnly cookies |

Never store session tokens in `localStorage` — XSS can steal them.

# 6. Content Security Policy (CSP) Headers

CSP tells browsers which scripts, styles, and connections are allowed.

```typescript
// next.config.ts
const cspHeader = `
  default-src 'self';
  script-src 'self' 'unsafe-eval' 'unsafe-inline';
  style-src 'self' 'unsafe-inline';
  img-src 'self' blob: data: https:;
  font-src 'self';
  object-src 'none';
  base-uri 'self';
  form-action 'self';
  frame-ancestors 'none';
  upgrade-insecure-requests;
`

const securityHeaders = [
  {
    key: 'Content-Security-Policy',
    value: cspHeader.replace(/\n/g, ''),
  },
  {
    key: 'X-Frame-Options',
    value: 'DENY',
  },
  {
    key: 'X-Content-Type-Options',
    value: 'nosniff',
  },
  {
    key: 'Referrer-Policy',
    value: 'strict-origin-when-cross-origin',
  },
]

export default {
  async headers() {
    return [{ source: '/(.*)', headers: securityHeaders }]
  },
}
```

Tighten `script-src` in production — remove `'unsafe-inline'` when using nonces or strict CSP strategies.

# 7. Rate Limiting Overview

Rate limiting prevents abuse: brute-force login, spam forms, API scraping.

```typescript
// lib/rate-limit.ts (simplified in-memory — use Redis in production)
const hits = new Map<string, { count: number; resetAt: number }>()

export function rateLimit(key: string, limit = 10, windowMs = 60_000): boolean {
  const now = Date.now()
  const entry = hits.get(key)

  if (!entry || now > entry.resetAt) {
    hits.set(key, { count: 1, resetAt: now + windowMs })
    return true
  }

  if (entry.count >= limit) return false

  entry.count++
  return true
}
```

```typescript
'use server'

export async function loginAction(formData: FormData) {
  const ip = headers().get('x-forwarded-for') ?? 'unknown'
  if (!rateLimit(`login:${ip}`, 5, 60_000)) {
    return { error: 'Too many attempts. Try again later.' }
  }
  // verify credentials
}
```

| Endpoint | Suggested limit |
| -------- | --------------- |
| Login | 5/minute per IP |
| Password reset | 3/hour per email |
| Public API | 100/minute per key |

Use Vercel KV, Upstash Redis, or middleware-based limits at the edge for production.

# 8. Safe Error Messages for Users

Detailed errors help attackers map your system.

```typescript
// ❌ Leaks internals
catch (error) {
  return { error: error.message } // might expose SQL or file paths
}

// ✅ Generic user message + server log
catch (error) {
  console.error('[createPost]', error)
  return { error: 'Something went wrong. Please try again.' }
}
```

| Audience | Message |
| -------- | ------- |
| User | Friendly, non-technical |
| Developer | Full stack in server logs |
| API client | Structured error code, no stack trace |

Configure `error.tsx` to show generic UI — log details with Sentry or similar (Topic 34).

# 9. Authentication and Authorization Checklist

✔ Verify session in every Server Action and protected Route Handler

✔ Check resource ownership (`post.authorId === session.user.id`)

✔ Role checks on admin routes — server-side only

✔ Hash passwords with bcrypt (cost factor 10+) or Argon2

✔ Use httpOnly, secure, sameSite cookies for sessions

✔ Implement logout that clears session server-side

✔ Rotate `AUTH_SECRET` and API keys on compromise

# 10. Dependency and Supply Chain Security

Run `npm audit` regularly. Use Dependabot for automated updates. Never commit `package-lock.json` changes without reviewing.

---

# 11. Best Practices Summary

✔ `server-only` for DB and secret modules

✔ Validate all input with Zod on the server

✔ Never expose secrets via `NEXT_PUBLIC_`

✔ Sanitize HTML before `dangerouslySetInnerHTML`

✔ Set CSP and security headers in `next.config.ts`

✔ Rate limit auth and public write endpoints

✔ Generic errors to users; detailed logs server-side

✔ Keep dependencies updated and audited

# 13. Summary

| Threat | Defense |
| ------ | ------- |
| Secret leak | No `NEXT_PUBLIC_` for secrets; `server-only` imports |
| CSRF | Server Actions + SameSite cookies; no mutating GET |
| XSS | React escaping; sanitize HTML; httpOnly cookies |
| Invalid input | Zod on server; never trust client |
| Abuse | Rate limiting on auth and write endpoints |
| Info disclosure | Generic user errors; log details server-side |

Security is layered. No single fix covers everything — apply these habits across the stack.
