# Monitoring and Maintenance - Teaching Notes

## Topic 34: Keeping Production Next.js Apps Healthy

---

# 1. Why This Topic Matters

Deploying is not the finish line. Production apps need:

* Error visibility when users hit bugs
* Performance tracking as traffic grows
* Dependency updates and security patches
* A plan when things break at 2 AM

Monitoring and maintenance turn a shipped app into a **reliable product**. This lesson covers error tracking, logging, analytics, updates, and incident response basics.

# 2. Runtime Error Monitoring

Users encounter errors you never see locally. Error monitoring captures stack traces, user context, and frequency.

### Popular tools

| Tool | Strength |
| ---- | -------- |
| **Sentry** | Excellent Next.js SDK, source maps, releases |
| **Datadog** | Full observability platform |
| **Bugsnag** | Simple error grouping |

### Sentry setup (example)

```bash
npm install @sentry/nextjs
npx @sentry/wizard@latest -i nextjs
```

Wizard creates:

* `sentry.client.config.ts`
* `sentry.server.config.ts`
* `sentry.edge.config.ts`
* Updates `next.config.ts` for source maps

```typescript
// sentry.server.config.ts
import * as Sentry from '@sentry/nextjs'

Sentry.init({
  dsn: process.env.SENTRY_DSN,
  tracesSampleRate: 0.1,
  environment: process.env.NODE_ENV,
})
```

### Error boundaries integration

```tsx
// app/error.tsx
'use client'

import * as Sentry from '@sentry/nextjs'
import { useEffect } from 'react'

export default function Error({
  error,
  reset,
}: {
  error: Error & { digest?: string }
  reset: () => void
}) {
  useEffect(() => {
    Sentry.captureException(error)
  }, [error])

  return (
    <div>
      <h2>Something went wrong</h2>
      <button onClick={reset}>Try again</button>
    </div>
  )
}
```

| Monitor | Catches |
| ------- | ------- |
| Sentry client | Browser/React errors |
| Sentry server | Server Component and action failures |
| Sentry edge | Middleware errors |

# 3. Logging Strategy for Server and Client

Structured logs help debug production without exposing secrets to users.

### Server logging

```typescript
// lib/logger.ts
type LogLevel = 'info' | 'warn' | 'error'

function log(level: LogLevel, message: string, meta?: Record<string, unknown>) {
  const entry = {
    level,
    message,
    timestamp: new Date().toISOString(),
    ...meta,
  }
  console[level](JSON.stringify(entry))
}

export const logger = {
  info: (msg: string, meta?: Record<string, unknown>) => log('info', msg, meta),
  warn: (msg: string, meta?: Record<string, unknown>) => log('warn', msg, meta),
  error: (msg: string, meta?: Record<string, unknown>) => log('error', msg, meta),
}
```

```typescript
'use server'

export async function createOrderAction(formData: FormData) {
  try {
    const order = await createOrder(/* ... */)
    logger.info('Order created', { orderId: order.id, userId: order.userId })
    return { success: true }
  } catch (error) {
    logger.error('Order creation failed', {
      error: error instanceof Error ? error.message : 'Unknown',
    })
    return { error: 'Could not create order' }
  }
}
```

### What to log vs not log

| Log | Do not log |
| --- | ---------- |
| Request IDs, user IDs (non-PII) | Passwords, tokens, credit cards |
| Action outcomes | Full request bodies with secrets |
| Performance timings | Personal health or financial data |

Use Vercel Logs, AWS CloudWatch, or Datadog to aggregate server logs.

### Client logging

Minimize `console.log` in production client code. Use analytics events for user behavior, Sentry for errors.

# 4. Performance Monitoring

Track Core Web Vitals and server response times in production.

### Vercel Speed Insights

```tsx
// app/layout.tsx
import { SpeedInsights } from '@vercel/speed-insights/next'

export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <html lang="en">
      <body>
        {children}
        <SpeedInsights />
      </body>
    </html>
  )
}
```

### Sentry performance

```typescript
Sentry.init({
  dsn: process.env.SENTRY_DSN,
  tracesSampleRate: 0.1, // 10% of transactions
})
```

### Custom server timing

```typescript
export async function GET() {
  const start = Date.now()
  const data = await fetchExpensiveData()
  logger.info('API /reports timing', { durationMs: Date.now() - start })
  return Response.json(data)
}
```

| Metric | Tool |
| ------ | ---- |
| LCP, INP, CLS | Speed Insights, Google CrUX |
| API latency | Sentry, Datadog APM |
| Build time | CI logs, Vercel dashboard |

Set alerts when p95 latency or error rate exceeds thresholds.

# 6. Versioning and Release Notes

Semantic versioning communicates change impact.

| Version part | When to bump | Example |
| ------------ | ------------ | ------- |
| MAJOR | Breaking changes | 2.0.0 |
| MINOR | New features | 1.1.0 |
| PATCH | Bug fixes | 1.0.1 |

### Release notes template

```markdown
## v1.2.0 — 2026-06-07

### Added
- Dashboard export to CSV
- Dark mode toggle in settings

### Fixed
- Login redirect loop on expired session

### Changed
- Upgraded Next.js to 15.1
```

Tag releases in git:

```bash
git tag -a v1.2.0 -m "Release 1.2.0"
git push origin v1.2.0
```

Link Sentry releases to git tags for easier regression tracking.

# 7. Dependency Updates and Security Checks

```bash
npm outdated
npm audit
npm update
```

| Tool | Purpose |
| ---- | ------- |
| Dependabot | Automated dependency PRs on GitHub |
| Renovate | Configurable update bot |
| `npm audit` | Known vulnerability scan |

### Update strategy

| Dependency type | Update frequency |
| --------------- | ---------------- |
| Security patches | Immediately |
| Next.js minor | Within 1–2 sprints, test thoroughly |
| Major versions | Planned migration, read changelog |

Before upgrading Next.js:

1. Read [Next.js upgrade guide](https://nextjs.org/docs/app/building-your-application/upgrading)
2. Run `npm run build` and full test suite
3. Deploy to preview environment first

# 8. Incident Response Basics

When production breaks:

```text
1. Detect   — alert fires or user report
2. Triage   — severity, scope, who's affected
3. Mitigate — rollback, feature flag, hotfix
4. Fix      — root cause patch
5. Review   — post-incident notes, prevent recurrence
```

### Severity levels

| Level | Example | Response |
| ----- | ------- | -------- |
| P1 | Site down, payments failing | Immediate, all hands |
| P2 | Major feature broken | Same business day |
| P3 | Minor bug, workaround exists | Next sprint |
| P4 | Cosmetic | Backlog |

### Rollback on Vercel

Vercel dashboard → Deployments → previous deployment → **Promote to Production**.

Keep rollback one click away — do not debug in production unless necessary.

### Post-incident notes

Document duration, impact, root cause, fix, and prevention steps for P1/P2 issues.

# 9. Health Checks and Uptime

```typescript
// app/api/health/route.ts
import { NextResponse } from 'next/server'

export async function GET() {
  try {
    // Optional: ping database
    // await prisma.$queryRaw`SELECT 1`
    return NextResponse.json({ status: 'ok', timestamp: Date.now() })
  } catch {
    return NextResponse.json({ status: 'error' }, { status: 503 })
  }
}
```

External uptime monitors (UptimeRobot, Better Stack) ping `/api/health` every 1–5 minutes and alert on failure.

Document runbooks: "If login fails, check AUTH_SECRET and OAuth redirect URIs."

# 10. Best Practices

✔ Install error monitoring before launch — not after first outage

✔ Structured JSON logs on server; no secrets in logs

✔ Tag releases and connect to error tracking

✔ Automate dependency PRs with Dependabot

✔ Maintain rollback path — preview deploy before production

✔ Write short post-incident notes for P1/P2 issues

✔ Health check endpoint for uptime monitoring
