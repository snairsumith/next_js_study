# Testing - Teaching Notes

## Topic 31: Unit, Integration, and E2E Testing in Next.js

---

# 1. Why This Topic Matters

Manual clicking does not scale. Every deploy risks breaking login, checkout, or data mutations. Automated tests give confidence that:

* Components render correctly
* Server Actions validate and persist data
* Route Handlers return correct status codes
* Critical user flows work end-to-end

Testing in Next.js spans server code (actions, handlers) and client UI (forms, hooks). This lesson covers tools, patterns, and what is worth testing.

---

# 2. Testing Tools Overview

| Tool | Purpose |
| ---- | ------- |
| **Vitest** or **Jest** | Unit and integration test runner |
| **React Testing Library (RTL)** | Test components like users interact |
| **@testing-library/user-event** | Realistic clicks, typing, keyboard |
| **MSW (Mock Service Worker)** | Mock HTTP APIs |
| **Playwright** or **Cypress** | End-to-end browser tests |

### Recommended stack for Next.js

```bash
npm install -D vitest @vitejs/plugin-react jsdom
npm install -D @testing-library/react @testing-library/jest-dom @testing-library/user-event
```

```typescript
// vitest.config.ts
import { defineConfig } from 'vitest/config'
import react from '@vitejs/plugin-react'
import path from 'path'

export default defineConfig({
  plugins: [react()],
  test: {
    environment: 'jsdom',
    setupFiles: ['./vitest.setup.ts'],
  },
  resolve: {
    alias: {
      '@': path.resolve(__dirname, './src'),
    },
  },
})
```

```typescript
// vitest.setup.ts
import '@testing-library/jest-dom/vitest'
```

Add script to `package.json`:

```json
"scripts": {
  "test": "vitest",
  "test:run": "vitest run"
}
```

Jest works similarly — Vitest is faster and has native ESM support; either is acceptable for training.

# 3. React Testing Library for Components

Test behavior, not implementation details.

```tsx
// components/ui/button.test.tsx
import { render, screen } from '@testing-library/react'
import userEvent from '@testing-library/user-event'
import { Button } from './button'

describe('Button', () => {
  it('renders label and calls onClick', async () => {
    const handleClick = vi.fn()
    render(<Button label="Save" onClick={handleClick} />)

    const button = screen.getByRole('button', { name: 'Save' })
    await userEvent.click(button)

    expect(handleClick).toHaveBeenCalledTimes(1)
  })

  it('is disabled when disabled prop is true', () => {
    render(<Button label="Save" disabled />)
    expect(screen.getByRole('button', { name: 'Save' })).toBeDisabled()
  })
})
```

### RTL queries priority

| Priority | Query | Example |
| -------- | ----- | ------- |
| 1 | Role | `getByRole('button', { name: 'Submit' })` |
| 2 | Label | `getByLabelText('Email')` |
| 3 | Text | `getByText('Welcome')` |
| 4 | Test ID | `getByTestId('cart-count')` — last resort |

### Testing client components

Client components with `'use client'` test normally in Vitest/Jest with jsdom.

```tsx
'use client'

// components/counter.test.tsx
import { render, screen } from '@testing-library/react'
import userEvent from '@testing-library/user-event'
import { Counter } from './counter'

it('increments count on click', async () => {
  render(<Counter />)
  await userEvent.click(screen.getByRole('button', { name: 'Increment' }))
  expect(screen.getByText('Count: 1')).toBeInTheDocument()
})
```

# 4. Testing Server Components

Server Components are async functions — test extracted logic separately, or use integration tests.

### Strategy 1 — Test pure logic

```typescript
// lib/format-post.test.ts
import { formatPostDate } from './format-post'

describe('formatPostDate', () => {
  it('formats ISO string to readable date', () => {
    const result = formatPostDate('2026-01-15T10:00:00Z')
    expect(result).toMatch(/Jan/)
  })
})
```

### Strategy 2 — Test data layer

```typescript
// lib/db/posts.test.ts
import { vi } from 'vitest'
import { getPostsByUser } from './posts'

vi.mock('@/lib/db/client', () => ({
  prisma: {
    post: {
      findMany: vi.fn().mockResolvedValue([
        { id: '1', title: 'Hello', authorId: 'u1' },
      ]),
    },
  },
}))

it('returns posts for user', async () => {
  const posts = await getPostsByUser('u1')
  expect(posts).toHaveLength(1)
  expect(posts[0].title).toBe('Hello')
})
```

Do not over-mock Server Components themselves — test the units they compose.

# 5. Testing Server Actions

Server Actions are async functions. Call them directly in tests with mocked dependencies.

```typescript
// app/actions/posts.test.ts
import { vi, describe, it, expect, beforeEach } from 'vitest'
import { createPostAction } from './posts'

vi.mock('@/auth', () => ({
  auth: vi.fn(),
}))

vi.mock('@/lib/db/posts', () => ({
  createPost: vi.fn(),
}))

import { auth } from '@/auth'
import { createPost } from '@/lib/db/posts'

describe('createPostAction', () => {
  beforeEach(() => {
    vi.clearAllMocks()
  })

  it('returns error when unauthorized', async () => {
    vi.mocked(auth).mockResolvedValue(null)

    const formData = new FormData()
    formData.set('title', 'Test')
    formData.set('content', 'Body content here')

    const result = await createPostAction(formData)
    expect(result.error).toBe('Unauthorized')
    expect(createPost).not.toHaveBeenCalled()
  })

  it('creates post when authorized', async () => {
    vi.mocked(auth).mockResolvedValue({
      user: { id: 'user-1', name: 'Anu' },
    } as any)

    const formData = new FormData()
    formData.set('title', 'My Post')
    formData.set('content', 'Long enough content')

    const result = await createPostAction(formData)
    expect(result.success).toBe(true)
    expect(createPost).toHaveBeenCalledWith('user-1', expect.objectContaining({
      title: 'My Post',
    }))
  })
})
```

| Mock | Why |
| ---- | --- |
| `auth()` | Control logged-in state |
| Database functions | Avoid real DB in unit tests |
| `revalidatePath` | No-op in test environment |

# 6. Testing Route Handlers

Use `Request` and assert on `Response`.

```typescript
// app/api/health/route.test.ts
import { GET } from './route'

describe('GET /api/health', () => {
  it('returns 200 with status ok', async () => {
    const response = await GET()
    expect(response.status).toBe(200)

    const body = await response.json()
    expect(body.status).toBe('ok')
  })
})
```

```typescript
// app/api/posts/route.test.ts
import { POST } from './route'
import { vi } from 'vitest'

vi.mock('@/auth', () => ({ auth: vi.fn() }))
vi.mock('@/lib/db/posts', () => ({ createPost: vi.fn() }))

it('returns 401 without session', async () => {
  const { auth } = await import('@/auth')
  vi.mocked(auth).mockResolvedValue(null)

  const request = new Request('http://localhost/api/posts', {
    method: 'POST',
    body: JSON.stringify({ title: 'T', content: 'Long content' }),
    headers: { 'Content-Type': 'application/json' },
  })

  const response = await POST(request as any)
  expect(response.status).toBe(401)
})
```

# 8. E2E Testing Overview

E2E tests run a real browser against a running app.

### Playwright setup

```bash
npm install -D @playwright/test
npx playwright install
```

```typescript
// e2e/login.spec.ts
import { test, expect } from '@playwright/test'

test('user can log in and see dashboard', async ({ page }) => {
  await page.goto('http://localhost:3000/login')
  await page.getByLabel('Email').fill('test@example.com')
  await page.getByLabel('Password').fill('password123')
  await page.getByRole('button', { name: 'Sign in' }).click()

  await expect(page).toHaveURL(/dashboard/)
  await expect(page.getByRole('heading', { name: /welcome/i })).toBeVisible()
})
```

| Test type | Speed | Confidence | Cost |
| --------- | ----- | ---------- | ---- |
| Unit | Fast | Low–medium | Cheap |
| Integration | Medium | Medium | Moderate |
| E2E | Slow | High | Expensive |

Run E2E on critical paths only: login, checkout, core CRUD.

# 9. What to Test vs What to Skip

### Test these

| Target | Example |
| ------ | ------- |
| Business logic | Price calculation, permission checks |
| Server Actions | Auth guard, validation, success path |
| Route Handlers | Status codes, error responses |
| Critical UI | Login form, payment step, delete confirm |
| Utils | Date formatters, slug generators |

### Skip or deprioritize

| Target | Reason |
| ------ | ------ |
| Implementation details | Internal state variable names |
| Third-party libraries | Trust their own tests |
| Static layout markup | Low risk; changes often |
| Every page snapshot | Brittle, high maintenance |

**Testing pyramid:** Many unit tests, fewer integration tests, few E2E tests.

Run `npm run test:run` and `npm run build` in CI on every PR.

# 11. Summary

| Layer | Tool | Focus |
| ----- | ---- | ----- |
| Components | RTL + Vitest | User-visible behavior |
| Server Actions | Vitest + mocks | Auth, validation, mutations |
| Route Handlers | Vitest + Request/Response | Status codes, JSON shape |
| E2E | Playwright | Critical user journeys |

Good tests protect refactors. Start with Server Action and utility tests — highest value for effort in Next.js apps.
