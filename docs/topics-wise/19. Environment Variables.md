# Environment Variables - Teaching Notes

## Topic 19: Configuring Secrets and Public Values Safely

---

# 1. Why This Topic Matters

Every real Next.js app needs configuration: API keys, database URLs, feature flags, and public site URLs. Environment variables keep secrets **out of source code** and let you use different values in development, staging, and production.

Getting this wrong — exposing a database password to the browser — is one of the most common and dangerous mistakes in full-stack development. This topic teaches safe patterns.

---

# 2. How Environment Variables Work

Next.js loads variables from `.env` files into `process.env` at build and runtime.

```text
my-app/
  .env                  ← shared defaults (can commit)
  .env.local            ← local overrides (NEVER commit)
  .env.development      ← dev-specific
  .env.production       ← production-specific
  .env.example          ← template for team (commit this)
```

```bash
# .env.local (never commit this file)
DATABASE_URL="postgresql://user:pass@localhost:5432/myapp"
STRIPE_SECRET_KEY="sk_test_xxxxxxxx"
NEXT_PUBLIC_SITE_URL="http://localhost:3000"
```

| File | Committed to git? | Purpose |
| ---- | ----------------- | ------- |
| `.env` | Sometimes | Shared non-secret defaults |
| `.env.local` | ❌ Never | Local secrets and overrides |
| `.env.production` | Sometimes | Production defaults (no secrets) |
| `.env.example` | ✅ Always | Documents required variables |

Add to `.gitignore`:

```gitignore
.env*.local
.env.local
```

# 3. Server-Only vs Client-Exposed Variables

This is the most critical rule in Next.js environment variables.

| Prefix | Available on server | Available in browser | Use for |
| ------ | ------------------- | -------------------- | ------- |
| No prefix | ✅ Yes | ❌ No | Secrets, DB URLs, private API keys |
| `NEXT_PUBLIC_` | ✅ Yes | ✅ Yes (bundled into client JS) | Public site URL, analytics ID |

```bash
# Server-only — safe
DATABASE_URL="postgresql://..."
STRIPE_SECRET_KEY="sk_live_..."
JWT_SECRET="super-secret-key"

# Client-exposed — visible in browser bundle
NEXT_PUBLIC_SITE_URL="https://myapp.com"
NEXT_PUBLIC_GA_ID="G-XXXXXXXX"
NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY="pk_live_..."
```

```tsx
// ✅ Server Component — can access all env vars
export default async function ProductsPage() {
  const apiKey = process.env.STRIPE_SECRET_KEY // server only
  const siteUrl = process.env.NEXT_PUBLIC_SITE_URL // also fine
  // ...
}

// ⚠️ Client Component — ONLY NEXT_PUBLIC_ vars work
'use client'
export function CheckoutButton() {
  const publishableKey = process.env.NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY // ✅
  // const secretKey = process.env.STRIPE_SECRET_KEY // ❌ undefined in browser!
}
```

**Rule:** If it must stay secret, never prefix it with `NEXT_PUBLIC_`.

# 4. Using process.env in Next.js

Access variables via `process.env.VARIABLE_NAME`.

```tsx
// lib/db.ts — server only
import { PrismaClient } from '@prisma/client'

const globalForPrisma = globalThis as unknown as { prisma: PrismaClient }

export const db =
  globalForPrisma.prisma ??
  new PrismaClient({
    datasources: {
      db: { url: process.env.DATABASE_URL },
    },
  })

if (process.env.NODE_ENV !== 'production') globalForPrisma.prisma = db
```

```tsx
// lib/config.ts — typed config helper
function requireEnv(key: string): string {
  const value = process.env[key]
  if (!value) {
    throw new Error(`Missing required environment variable: ${key}`)
  }
  return value
}

export const config = {
  databaseUrl: requireEnv('DATABASE_URL'),
  stripeSecretKey: requireEnv('STRIPE_SECRET_KEY'),
  siteUrl: process.env.NEXT_PUBLIC_SITE_URL ?? 'http://localhost:3000',
  isProduction: process.env.NODE_ENV === 'production',
}
```

| Location | Server vars | Public vars |
| -------- | ----------- | ----------- |
| Server Components | ✅ | ✅ |
| Server Actions | ✅ | ✅ |
| Route Handlers | ✅ | ✅ |
| Middleware | ✅ | ✅ |
| Client Components | ❌ | ✅ only |
| `next.config.ts` | ✅ | ✅ |

# 5. Environment Variables in Route Handlers

Route Handlers run on the server — all env vars are available.

```tsx
// app/api/send-email/route.ts
import { NextRequest, NextResponse } from 'next/server'

export async function POST(request: NextRequest) {
  const apiKey = process.env.RESEND_API_KEY

  if (!apiKey) {
    console.error('RESEND_API_KEY is not configured')
    return NextResponse.json({ error: 'Email service unavailable' }, { status: 500 })
  }

  const body = await request.json()

  const res = await fetch('https://api.resend.com/emails', {
    method: 'POST',
    headers: {
      Authorization: `Bearer ${apiKey}`,
      'Content-Type': 'application/json',
    },
    body: JSON.stringify({
      from: process.env.EMAIL_FROM ?? 'noreply@myapp.com',
      to: body.to,
      subject: body.subject,
      html: body.html,
    }),
  })

  if (!res.ok) {
    return NextResponse.json({ error: 'Failed to send email' }, { status: 502 })
  }

  return NextResponse.json({ success: true })
}
```

# 6. The .env.example File

Document every required variable for team onboarding.

```bash
# .env.example — commit this file, copy to .env.local for local dev
# cp .env.example .env.local

# Database
DATABASE_URL="postgresql://user:password@localhost:5432/myapp"

# Authentication
JWT_SECRET="generate-a-random-string-here"
NEXTAUTH_SECRET="generate-another-random-string"
NEXTAUTH_URL="http://localhost:3000"

# Stripe (server)
STRIPE_SECRET_KEY="sk_test_..."
STRIPE_WEBHOOK_SECRET="whsec_..."

# Stripe (client — safe to expose)
NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY="pk_test_..."

# App
NEXT_PUBLIC_SITE_URL="http://localhost:3000"
```

| Practice | Why |
| -------- | --- |
| Commit `.env.example` | New developers know what to configure |
| Never commit `.env.local` | Prevents secret leaks |
| Use placeholder values | `"your-api-key-here"` not real keys |
| Add comments | Explain where to get each key |

# 7. Validation at Startup

Fail fast if required variables are missing — don't discover at runtime.

```tsx
// lib/env.ts
interface ServerEnv {
  DATABASE_URL: string
  STRIPE_SECRET_KEY: string
  JWT_SECRET: string
}

interface ClientEnv {
  NEXT_PUBLIC_SITE_URL: string
  NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY: string
}

function validateServerEnv(): ServerEnv {
  const required = ['DATABASE_URL', 'STRIPE_SECRET_KEY', 'JWT_SECRET'] as const

  for (const key of required) {
    if (!process.env[key]) {
      throw new Error(`❌ Missing server environment variable: ${key}`)
    }
  }

  return {
    DATABASE_URL: process.env.DATABASE_URL!,
    STRIPE_SECRET_KEY: process.env.STRIPE_SECRET_KEY!,
    JWT_SECRET: process.env.JWT_SECRET!,
  }
}

export const serverEnv = validateServerEnv()

export const clientEnv: ClientEnv = {
  NEXT_PUBLIC_SITE_URL: process.env.NEXT_PUBLIC_SITE_URL ?? 'http://localhost:3000',
  NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY: process.env.NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY ?? '',
}
```

With Zod (recommended for larger projects):

```tsx
// lib/env.ts
import { z } from 'zod'

const serverSchema = z.object({
  DATABASE_URL: z.string().url(),
  STRIPE_SECRET_KEY: z.string().min(1),
  JWT_SECRET: z.string().min(32),
})

const clientSchema = z.object({
  NEXT_PUBLIC_SITE_URL: z.string().url(),
  NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY: z.string().startsWith('pk_'),
})

export const serverEnv = serverSchema.parse(process.env)
export const clientEnv = clientSchema.parse(process.env)
```

# 8. Environment Config Per Deployment Target

Different environments need different values.

| Environment | DATABASE_URL | NEXT_PUBLIC_SITE_URL | STRIPE keys |
| ----------- | ------------ | -------------------- | ----------- |
| Development | Local PostgreSQL | `http://localhost:3000` | Test keys (`sk_test_`) |
| Staging | Staging DB | `https://staging.myapp.com` | Test keys |
| Production | Production DB | `https://myapp.com` | Live keys (`sk_live_`) |

```bash
# Development (.env.local)
DATABASE_URL="postgresql://localhost:5432/myapp_dev"
NEXT_PUBLIC_SITE_URL="http://localhost:3000"
STRIPE_SECRET_KEY="sk_test_..."

# Production (set in Vercel/hosting dashboard — not in files)
DATABASE_URL="postgresql://prod-host:5432/myapp"
NEXT_PUBLIC_SITE_URL="https://myapp.com"
STRIPE_SECRET_KEY="sk_live_..."
```

**Never put production secrets in committed files.**

# 9. Setting Variables on Hosting Platforms

### Vercel

```bash
# Via CLI
vercel env add DATABASE_URL production
vercel env add STRIPE_SECRET_KEY production

# Or via Vercel Dashboard → Project → Settings → Environment Variables
```

| Platform | Where to set |
| -------- | ------------ |
| Vercel | Dashboard → Settings → Environment Variables |
| Netlify | Site settings → Environment variables |
| AWS / Docker | ECS task definition, `.env` in container (not committed) |
| Railway | Project → Variables tab |

Variables set on hosting platforms override local `.env` files in production.

# 10. Common Mistakes

| Mistake | Consequence | Fix |
| ------- | ----------- | --- |
| `NEXT_PUBLIC_DATABASE_URL` | DB credentials in browser bundle | Remove `NEXT_PUBLIC_` prefix |
| Committing `.env.local` | Secrets leaked to GitHub | Add to `.gitignore` |
| Hardcoding API keys in code | Keys in git history forever | Use `process.env` |
| No validation at startup | Silent failures in production | Validate with Zod at boot |
| Using server vars in Client Components | Always `undefined` in browser | Use Route Handler or Server Action as proxy |
| Same secrets for dev and prod | Dev leak compromises production | Separate keys per environment |

```tsx
// ❌ NEVER do this
'use client'
const dbUrl = process.env.NEXT_PUBLIC_DATABASE_URL

// ✅ Proxy through server
'use client'
async function saveData(data: FormData) {
  await fetch('/api/save', { method: 'POST', body: data })
  // API route uses process.env.DATABASE_URL safely
}
```

# 11. next.config.ts and Env Vars

You can expose env vars through `next.config.ts` — but prefer `NEXT_PUBLIC_` prefix instead.

```tsx
// next.config.ts
import type { NextConfig } from 'next'

const nextConfig: NextConfig = {
  env: {
    CUSTOM_API_VERSION: 'v2',
  },
}

export default nextConfig
```

| Method | Recommendation |
| ------ | -------------- |
| `NEXT_PUBLIC_` prefix | ✅ Preferred for client vars |
| `next.config.ts env` | ⚠️ Legacy — avoid for new projects |
| `process.env` directly | ✅ Standard approach |

---

# 12. Best Practices

✔ Never commit `.env.local` or any file with real secrets

✔ Always commit `.env.example` with placeholder values

✔ Use `NEXT_PUBLIC_` only for values safe to expose in the browser

✔ Validate required env vars at startup with Zod or a helper

✔ Use different secrets for development and production

✔ Set production variables on the hosting platform — not in code

✔ Never log secret values — log variable names only when debugging

# 14. Summary

| Concept | Rule |
| ------- | ---- |
| `.env.local` | Local secrets — never commit |
| `.env.example` | Template — always commit |
| No prefix | Server-only — safe for secrets |
| `NEXT_PUBLIC_` | Bundled into client JS — never for secrets |
| `process.env` | Access in server code, Route Handlers, Server Actions |
| Validation | Fail fast at startup with Zod or helper |
| Production | Set vars on hosting platform, not in files |
