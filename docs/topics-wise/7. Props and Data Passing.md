# Props and Data Passing - Teaching Notes

## Topic 7: Passing Data Across the Server/Client Boundary

---

# 1. Why This Topic Matters

Props are how components communicate in React — and that does not change in Next.js. What **does** change is the **server/client boundary**: data must cross from Server Components (where fetch and DB access happen) to Client Components (where interactivity lives).

Students who ignore the **serializable props rule** hit cryptic errors. Understanding props in Next.js prevents:

* Passing functions from server to client incorrectly
* Passing non-serializable data (Dates, class instances)
* Duplicating fetch logic on the client when server already has data

This topic covers all prop-passing patterns in App Router with TypeScript.

---

# 2. Props Fundamentals Recap

Props are **read-only** inputs to a component — same as react-project-study.

```tsx
// components/Greeting.tsx
interface GreetingProps {
  name: string
  role: string
}

export function Greeting({ name, role }: GreetingProps) {
  return (
    <p>
      Hello, <strong>{name}</strong> — {role}
    </p>
  )
}
```

```tsx
// Usage
<Greeting name="Anu" role="Student" />
```

| Rule | Detail |
| ---- | ------ |
| Read-only | Never mutate `props` inside a component |
| One-way flow | Data flows parent → child |
| Type with interfaces | TypeScript interfaces for prop shapes |
| Destructure in parameters | `{ name, role }` — same as React |

# 3. Passing Props Between Server Components

Server Components can pass any data to other Server Components freely — no serialization restriction because nothing crosses the network to the client.

```tsx
// lib/users.ts
export async function getUsers() {
  const res = await fetch('https://jsonplaceholder.typicode.com/users')
  return res.json()
}
```

```tsx
// components/users/UserList.tsx — Server Component
interface User {
  id: number
  name: string
  email: string
}

interface UserListProps {
  users: User[]
}

export function UserList({ users }: UserListProps) {
  return (
    <ul>
      {users.map((user) => (
        <UserListItem key={user.id} user={user} />
      ))}
    </ul>
  )
}
```

```tsx
// components/users/UserListItem.tsx — Server Component
interface UserListItemProps {
  user: {
    id: number
    name: string
    email: string
  }
}

export function UserListItem({ user }: UserListItemProps) {
  return (
    <li>
      <strong>{user.name}</strong> — {user.email}
    </li>
  )
}
```

```tsx
// app/users/page.tsx — Server Component
import { getUsers } from '@/lib/users'
import { UserList } from '@/components/users/UserList'

export default async function UsersPage() {
  const users = await getUsers()
  return (
    <main>
      <h1>Users</h1>
      <UserList users={users} />
    </main>
  )
}
```

This mirrors react-project-study patterns — except data comes from server fetch, not `useEffect`.

# 4. Passing Props from Server to Client Components

When a Server Component passes props to a Client Component, props must be **serializable** — convertible to JSON for the client bundle.

```tsx
// components/products/AddToCartButton.tsx
'use client'

interface AddToCartButtonProps {
  productId: number
  productName: string
  price: number
}

export function AddToCartButton({
  productId,
  productName,
  price,
}: AddToCartButtonProps) {
  function handleClick() {
    console.log(`Added ${productName} (${productId}) — Rs. ${price}`)
  }

  return (
    <button onClick={handleClick}>
      Add {productName} to cart
    </button>
  )
}
```

```tsx
// app/products/page.tsx — Server Component passes serializable props
import { AddToCartButton } from '@/components/products/AddToCartButton'

export default async function ProductsPage() {
  const product = { id: 1, name: 'Keyboard', price: 1499 }

  return (
    <AddToCartButton
      productId={product.id}
      productName={product.name}
      price={product.price}
    />
  )
}
```

### Serializable vs non-sererializable props

| Serializable ✅ | Not serializable ❌ |
| --------------- | ------------------- |
| `string`, `number`, `boolean` | Functions (except Server Actions — Topic 23) |
| `null`, `undefined` | Class instances (`new Date()` as object) |
| Plain objects `{ key: value }` | `Map`, `Set`, `Symbol` |
| Arrays of serializable values | React elements passed incorrectly |
| `Date` as ISO string | Database model instances with methods |

```tsx
// ❌ Cannot pass function from Server to Client Component
<ClientButton onClick={() => console.log('click')} />

// ✅ Define handler inside Client Component
'use client'
function ClientButton() {
  function handleClick() { console.log('click') }
  return <button onClick={handleClick}>Click</button>
}
```

# 5. The Serializable Props Rule

Next.js sends props from Server Components to Client Components over the network during SSR. Only JSON-compatible values survive.

```tsx
// app/profile/page.tsx — Server Component
'use client' // hypothetical client child

// ❌ WRONG — Date object is not reliably serializable
<ProfileCard createdAt={new Date()} />

// ✅ CORRECT — pass ISO string, parse on client if needed
<ProfileCard createdAt={user.createdAt.toISOString()} />
```

```tsx
// components/ProfileCard.tsx
'use client'

interface ProfileCardProps {
  createdAt: string // ISO string
}

export function ProfileCard({ createdAt }: ProfileCardProps) {
  const formatted = new Date(createdAt).toLocaleDateString('en-IN')
  return <p>Joined: {formatted}</p>
}
```

| Data type | Server → Client strategy |
| --------- | ------------------------ |
| Date | Pass ISO string; construct `Date` on client |
| BigInt | Convert to string |
| Custom class | Map to plain object before passing |
| Function / callback | Define inside Client Component |
| Server Action | Special case — pass as form `action` prop (Topic 23) |

# 6. Passing children for Composition

The **`children`** prop is the primary pattern for nesting Server Components inside Client Components.

```tsx
// components/layout/ModalShell.tsx
'use client'

import { useState } from 'react'

interface ModalShellProps {
  triggerLabel: string
  children: React.ReactNode
}

export function ModalShell({ triggerLabel, children }: ModalShellProps) {
  const [isOpen, setIsOpen] = useState(false)

  return (
    <>
      <button onClick={() => setIsOpen(true)}>{triggerLabel}</button>
      {isOpen && (
        <div className="modal-overlay" role="dialog">
          <div className="modal-content">
            <button onClick={() => setIsOpen(false)}>Close</button>
            {children}
          </div>
        </div>
      )}
    </>
  )
}
```

```tsx
// app/products/page.tsx — Server Component
import { ModalShell } from '@/components/layout/ModalShell'
import { ProductDetails } from '@/components/products/ProductDetails'

export default async function ProductsPage() {
  const product = await getProduct(1)

  return (
    <ModalShell triggerLabel="View details">
      {/* ProductDetails is a Server Component — rendered on server, passed as children */}
      <ProductDetails product={product} />
    </ModalShell>
  )
}
```

| Pattern | Why it works |
| ------- | ------------ |
| Server page renders Server child | Server fetches data |
| Server child passed as `children` to Client shell | Client handles open/close; server content is pre-rendered HTML |
| Avoids importing Server into Client file | Respects component boundary rules |

Same pattern as react-project-study composition — `children` is still the most flexible prop.

# 7. Passing Event Handlers (Client Only)

Event handlers exist only in Client Components. Define them **inside** the client file — never pass from server.

```tsx
// ✅ Correct — handler defined in Client Component
'use client'

interface DeleteButtonProps {
  itemId: string
  itemName: string
}

export function DeleteButton({ itemId, itemName }: DeleteButtonProps) {
  async function handleDelete() {
    if (!confirm(`Delete ${itemName}?`)) return
    await fetch(`/api/items/${itemId}`, { method: 'DELETE' })
  }

  return (
    <button onClick={handleDelete} className="text-red-600">
      Delete
    </button>
  )
}
```

```tsx
// ❌ Wrong — server tries to pass onClick
export default function ServerPage() {
  return (
    <button onClick={() => alert('hi')}>Click</button> // Error in Server Component
  )
}
```

| Handler location | Allowed? |
| ---------------- | -------- |
| Inside `'use client'` component | ✅ |
| Passed as prop from Server to Client | ❌ (not serializable) |
| Server Action as form action | ✅ (Topic 11, 23) |

Later topics cover **Server Actions** as the server-safe way to handle form submissions without client handlers.

# 8. Props Are Read-Only

Never mutate props — same rule as React. If derived state is needed, compute locally or use state.

```tsx
// ❌ Wrong — mutating props
function BadComponent({ items }: { items: string[] }) {
  items.push('new item') // Never mutate props
  return <ul>{items.map(i => <li key={i}>{i}</li>)}</ul>
}
```

```tsx
// ✅ Correct — copy before modify
function GoodComponent({ items }: { items: string[] }) {
  const sortedItems = [...items].sort()
  return <ul>{sortedItems.map(i => <li key={i}>{i}</li>)}</ul>
}
```

```tsx
// ✅ Client Component — local state for edits
'use client'

function EditableName({ initialName }: { initialName: string }) {
  const [name, setName] = useState(initialName) // copy prop into state
  return <input value={name} onChange={(e) => setName(e.target.value)} />
}
```

# 9. Typing Props in Next.js Pages

Pages receive special props from Next.js: **`params`** and **`searchParams`**.

```tsx
// app/products/[id]/page.tsx
interface ProductPageProps {
  params: Promise<{ id: string }>
  searchParams: Promise<{ tab?: string }>
}

export default async function ProductPage({
  params,
  searchParams,
}: ProductPageProps) {
  const { id } = await params
  const { tab } = await searchParams

  return (
    <div>
      <h1>Product {id}</h1>
      <p>Active tab: {tab ?? 'details'}</p>
    </div>
  )
}
```

| Prop | Source | Type (Next.js 15) |
| ---- | ------ | ----------------- |
| `params` | Dynamic route segments | `Promise<{ slug: string }>` |
| `searchParams` | URL query string | `Promise<{ key?: string }>` |
| `children` | Layouts | `React.ReactNode` |

In Next.js 15, `params` and `searchParams` are **Promises** — must be awaited.

Layouts can also receive `params`:

```tsx
interface LayoutProps {
  children: React.ReactNode
  params: Promise<{ slug: string }>
}
```

# 10. Common Prop-Passing Mistakes

| Mistake | Symptom | Fix |
| ------- | ------- | --- |
| Pass function prop server → client | Serialization error | Define handler in client component |
| Pass `Date` object to client | Invalid or wrong date | Pass ISO string |
| Import Server Component in Client file | Build/runtime error | Use `children` composition |
| Mutate props array/object | Subtle UI bugs | Copy with spread first |
| Spread unknown API response to client | May include non-serializable fields | Map to plain DTO object |
| Optional props without defaults | `undefined` crashes | Use default params or `??` |

### Map API data before passing to client

```tsx
// app/users/page.tsx
export default async function UsersPage() {
  const rawUsers = await getUsersFromDB()

  // Map to plain serializable objects
  const users = rawUsers.map((u) => ({
    id: u.id,
    name: u.name,
    email: u.email,
    joinedAt: u.createdAt.toISOString(),
  }))

  return <UserTable users={users} />
}
```

# 11. Comparison with react-project-study

react-project-study passes props freely because everything runs on the client:

```tsx
// react-project-study/src/components/FilterableProductTable/ProductRow.tsx
function ProductRow({ product, onDelete }: ProductRowProps) {
  // onDelete function prop works — all client-side
}
```

```tsx
// react-project-study/src/pages/FilterableProductTable.tsx
<ProductRow product={p} onDelete={handleDelete} />
```

In Next.js:

| react-project-study | Next.js |
| ------------------- | ------- |
| Function props anywhere | Functions only inside Client Components |
| Fetch in parent, pass data down | Same — but parent can be Server Component |
| No serialization concern | Server → Client props must be serializable |
| `useParams()` for route params | `params` prop on page (awaited Promise) |

The **mental model** is the same (props down). The **constraint** is serialization at the server/client boundary.

# 12. Prop Drilling and Context (Preview)

For deeply nested client state, use React Context in a Client Component provider:

```tsx
// components/providers/CartProvider.tsx
'use client'

import { createContext, useContext, useState } from 'react'

interface CartContextValue {
  itemCount: number
  addItem: () => void
}

const CartContext = createContext<CartContextValue | null>(null)

export function CartProvider({ children }: { children: React.ReactNode }) {
  const [itemCount, setItemCount] = useState(0)

  function addItem() {
    setItemCount((c) => c + 1)
  }

  return (
    <CartContext.Provider value={{ itemCount, addItem }}>
      {children}
    </CartContext.Provider>
  )
}

export function useCart() {
  const ctx = useContext(CartContext)
  if (!ctx) throw new Error('useCart must be used within CartProvider')
  return ctx
}
```

Place providers in a Client Component wrapper in root layout — Topic 12 covers hooks in depth.

# 13. Best Practices

✔ Type all props with **TypeScript interfaces**

✔ Pass **plain objects** from server to client — map DB models first

✔ Use **`children`** to compose Server content inside Client shells

✔ Define **event handlers inside** Client Components

✔ Pass dates as **ISO strings**, not `Date` objects

✔ Treat props as **immutable** — never mutate directly

✔ Keep prop interfaces **small and focused** — avoid passing entire API responses if only 3 fields are needed

✔ **Await `params` and `searchParams`** in Next.js 15 pages

# 15. Summary

| Concept | Key takeaway |
| ------- | ------------ |
| Props | Read-only inputs — same one-way flow as React |
| Server → Server | Any data type allowed |
| Server → Client | Must be serializable (JSON-compatible) |
| Functions | Client-only — define inside `'use client'` files |
| `children` | Best pattern for Server UI inside Client shells |
| Dates / classes | Convert to plain values before crossing boundary |
| Page props | `params` and `searchParams` are Promises in Next.js 15 |
| vs React | Same props model + serialization rule at server/client edge |
