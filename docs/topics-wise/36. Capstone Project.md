# Capstone Project - Teaching Notes

## Topic 36: End-to-End Next.js Application Delivery

---

# 1. Why This Topic Matters

The capstone is the final proof of skill. Students move from guided exercises to **owning a complete product**:

* Choosing a real problem
* Designing routes, layouts, and data flow
* Implementing server and client boundaries correctly
* Shipping with auth, tests, and monitoring

Interviewers and employers care less about tutorial clones and more about **one polished app you can explain end-to-end**. This lesson provides capstone requirements, planning templates, implementation phases, and evaluation criteria.

---

# 2. Capstone Objectives

By completion, students must demonstrate:

| Objective | Evidence |
| --------- | -------- |
| App Router mastery | Nested layouts, dynamic routes, loading/error UI |
| Server Components | Async pages, direct data access |
| Server Actions | Validated mutations with revalidation |
| Auth (if applicable) | Middleware + server session checks |
| TypeScript | Typed params, actions, shared types |
| Performance | `next/image`, font optimization, code splitting |
| Testing | Unit tests for actions; optional E2E smoke test |
| Deployment | Live URL with env vars configured |
| Documentation | README with setup and architecture overview |

Minimum scope: **one primary domain** (e.g., tasks, events, listings) with full CRUD or equivalent workflow.

# 3. Step 1 — Choose a Real-World Problem

Pick a problem with a clear user and outcome. Avoid vague ideas.

### Strong capstone ideas

| Idea | User | Core value |
| ---- | ---- | ---------- |
| Team task board | Small teams | Assign and track tasks |
| Local events hub | Community | Discover and RSVP to events |
| Freelancer invoice tracker | Freelancers | Create and mark invoices paid |
| Recipe collection | Home cooks | Save and search recipes |
| Study group scheduler | Students | Propose and vote on meeting times |

### Weak ideas (avoid)

* "Social media app" — too broad for capstone timeline
* "Clone Netflix" — unrealistic scope
* Static portfolio only — no full-stack demonstration

Write one paragraph: **who**, **what pain**, **how your app helps**.

# 4. Step 2 — Plan Routes, Layouts, and Components

### Route map template

```text
app/
├── layout.tsx                  # Root: fonts, metadata, providers
├── page.tsx                    # Landing / marketing
├── (auth)/
│   ├── login/page.tsx
│   └── register/page.tsx
├── (app)/
│   ├── layout.tsx              # Authenticated shell
│   ├── dashboard/page.tsx
│   └── [resource]/
│       ├── page.tsx            # List
│       ├── new/page.tsx        # Create
│       └── [id]/
│           ├── page.tsx        # Detail
│           └── edit/page.tsx   # Update
├── api/                        # Optional Route Handlers
└── not-found.tsx
```

### Component inventory

| Component | Server / Client | Purpose |
| --------- | --------------- | ------- |
| `SiteHeader` | Server | Nav, auth state display |
| `ResourceList` | Server | Fetch and render list |
| `ResourceForm` | Client or Server Action form | Create/edit |
| `DeleteButton` | Client | Confirm + call action |
| `SearchFilters` | Client | URL search params |

Draw a simple diagram:

```text
Page (Server) ──fetch──► Service Layer ──► Database
     │
     └──► Client Leaf (button, form interactivity)
              │
              └──► Server Action ──► Service ──► revalidatePath
```

# 5. Step 3 — Data Model and Storage

Define entities before coding.

### Example: Task board

```typescript
interface User {
  id: string
  email: string
  name: string
}

interface Project {
  id: string
  name: string
  ownerId: string
}

interface Task {
  id: string
  title: string
  status: 'todo' | 'in_progress' | 'done'
  projectId: string
  assigneeId: string | null
  dueDate: string | null
}
```

| Storage option | Best for |
| -------------- | -------- |
| SQLite + Prisma | Solo capstone, easy local dev |
| PostgreSQL (Neon, Supabase) | Production-like deploy |
| JSON file | Prototype only — not for final capstone |

Add indexes on foreign keys and frequently filtered columns.

# 6. Step 4 — Implement Server and Client Components

### Rules for capstone

✔ Pages and layouts default to Server Components

✔ Fetch data in pages or service layer — not `useEffect`

✔ Client components only for: forms with rich interactivity, hooks, browser APIs

✔ Pass serializable props from server to client

### Example page composition

```tsx
// app/(app)/tasks/page.tsx
import { auth } from '@/auth'
import { redirect } from 'next/navigation'
import { getTasksForUser } from '@/features/tasks/lib/tasks-service'
import { TaskList } from '@/features/tasks/components/task-list'
import { CreateTaskForm } from '@/features/tasks/components/create-task-form'

export default async function TasksPage() {
  const session = await auth()
  if (!session?.user?.id) redirect('/login')

  const tasks = await getTasksForUser(session.user.id)

  return (
    <main className="mx-auto max-w-3xl p-6">
      <h1 className="text-2xl font-bold">My Tasks</h1>
      <CreateTaskForm userId={session.user.id} />
      <TaskList tasks={tasks} />
    </main>
  )
}
```

Document in README **why** each `'use client'` file exists.

# 7. Step 5 — Data Fetching, Caching, and Revalidation

| Data type | Strategy |
| --------- | -------- |
| User-specific lists | Dynamic fetch, `cache: 'no-store'` or auth-bound |
| Public catalog | Static or ISR with `revalidate` |
| Detail page | `generateStaticParams` if finite set; else dynamic |

### After mutations

```typescript
'use server'

import { revalidatePath } from 'next/cache'

export async function updateTaskAction(id: string, formData: FormData) {
  // auth + validation + update
  revalidatePath('/tasks')
  revalidatePath(`/tasks/${id}`)
}
```

Explain caching choices in README — interviewers ask this.

# 8. Step 6 — Authentication (If Needed)

Skip auth only if the app is fully public with no user-specific writes.

### Minimum auth requirements

* Login and logout flows
* Middleware protecting `(app)` routes
* Session check in every mutating Server Action
* Resource ownership validation

```typescript
'use server'

export async function deleteTaskAction(taskId: string) {
  const session = await auth()
  if (!session?.user?.id) throw new Error('Unauthorized')

  const task = await getTaskById(taskId)
  if (!task || task.ownerId !== session.user.id) {
    throw new Error('Forbidden')
  }

  await deleteTask(taskId)
  revalidatePath('/tasks')
}
```

# 9. Step 7 — Loading, Empty, and Error States

Every async route needs deliberate UI for all states.

| File | Purpose |
| ---- | ------- |
| `loading.tsx` | Skeleton while page data loads |
| `error.tsx` | Friendly error + retry |
| `not-found.tsx` | Invalid resource ID |
| Empty component | "No tasks yet — create one" |

```tsx
// features/tasks/components/task-list.tsx
export function TaskList({ tasks }: { tasks: Task[] }) {
  if (tasks.length === 0) {
    return (
      <p className="mt-4 text-gray-500">
        No tasks yet. Create your first task above.
      </p>
    )
  }

  return (
    <ul className="mt-4 space-y-2">
      {tasks.map((task) => (
        <li key={task.id}>{task.title}</li>
      ))}
    </ul>
  )
}
```

Demo all states in capstone presentation recording or screenshots.

# 10. Step 8 — Tests for Important Flows

Minimum test coverage:

| Test | Type |
| ---- | ---- |
| Create resource action — success | Unit |
| Create resource action — unauthorized | Unit |
| Validation rejects bad input | Unit |
| Home or login page loads | E2E smoke (optional) |

```typescript
// features/tasks/actions/create-task.test.ts
it('rejects empty title', async () => {
  const formData = new FormData()
  formData.set('title', '')
  const result = await createTaskAction(formData)
  expect(result.fieldErrors?.title).toBeDefined()
})
```

Tests prove the capstone works beyond manual clicking.

# 11. Step 9 — Build and Deploy

### Pre-deploy checklist

✔ `npm run lint` passes

✔ `npm run test:run` passes

✔ `npm run build` passes locally

✔ All env vars set on host

✔ Database migrated / seeded for demo

✔ Custom domain or Vercel URL in README

### README must include

```markdown
# Capstone: [App Name]

## Problem
[One paragraph]

## Live demo
https://your-app.vercel.app

## Tech stack
Next.js 15, TypeScript, Prisma, Auth.js, Tailwind

## Local setup
1. clone repo
2. cp .env.example .env.local
3. npm install
4. npx prisma migrate dev
5. npm run dev

## Architecture
[Brief folder structure and data flow]
```
