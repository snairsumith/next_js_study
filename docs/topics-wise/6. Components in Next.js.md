# Components in Next.js - Teaching Notes

## Topic 6: Server Components, Client Components, and Composition

---

# 1. Why This Topic Matters

In plain React (react-project-study), **every component runs in the browser**. In Next.js App Router, **Server Components are the default** — they run on the server and send HTML to the client.

Misunderstanding this split leads to the most common beginner errors:

* Using `useState` in a Server Component
* Marking entire pages `'use client'` unnecessarily
* Importing server-only code into client components

This topic teaches **when** to use each type, **how** to compose them, and **how** to organize component files — the foundation for everything else in the course.

---

# 2. Server Components by Default

Every file in the `app/` directory is a **Server Component** unless you add `'use client'` at the top.

```tsx
// app/products/page.tsx — Server Component (no directive needed)
export default async function ProductsPage() {
  const products = await getProducts()

  return (
    <ul>
      {products.map((product) => (
        <li key={product.id}>{product.name}</li>
      ))}
    </ul>
  )
}
```

| Server Component capability | Supported? |
| --------------------------- | ---------- |
| `async` / `await` | ✅ Yes |
| Direct database queries | ✅ Yes |
| Read files from disk | ✅ Yes |
| Access server env vars (no `NEXT_PUBLIC_`) | ✅ Yes |
| `useState`, `useEffect`, hooks | ❌ No |
| `onClick`, `onChange` event handlers | ❌ No |
| `window`, `document`, `localStorage` | ❌ No |

### Why Server Components exist

| Benefit | Explanation |
| ------- | ----------- |
| Zero client JS for static UI | Product list HTML sent ready — no hydration cost |
| Secure data access | API keys and DB credentials never reach browser |
| Faster initial load | Server fetches data close to the database |
| Smaller bundle | Heavy libraries stay on server |

Compare with react-project-study where **all** components ship JavaScript to the browser:

```tsx
// react-project-study/src/pages/DataFetch.tsx — everything runs client-side
useEffect(() => {
  fetch('/api/users').then(res => res.json()).then(setUsers)
}, [])
```

# 3. Client Components with 'use client'

Add **`'use client'`** at the very top of a file to make it a Client Component.

```tsx
// components/Counter.tsx
'use client'

import { useState } from 'react'

export function Counter() {
  const [count, setCount] = useState(0)

  return (
    <button
      onClick={() => setCount(count + 1)}
      aria-label={`Count is ${count}`}
    >
      Count: {count}
    </button>
  )
}
```

| Client Component capability | Supported? |
| --------------------------- | ---------- |
| `useState`, `useEffect`, all hooks | ✅ Yes |
| Event handlers (`onClick`, etc.) | ✅ Yes |
| Browser APIs | ✅ Yes |
| Third-party client libraries (charts, maps) | ✅ Yes |
| `async` component function | ❌ No |
| Direct server secrets | ❌ No |

### The directive rules

| Rule | Detail |
| ---- | ------ |
| File-level only | `'use client'` applies to the **entire file** and its imports |
| Top of file | Must be first line (before imports) |
| Propagation | Importing a Client Component into another file does NOT make that file client — only the imported component runs on client |
| Boundary | Once a file is `'use client'`, everything it imports is part of the client bundle |

# 4. When to Use Server vs Client Components

Use this decision flow:

```text
Does the component need interactivity?
  (onClick, onChange, useState, useEffect)
    YES → Client Component ('use client')
    NO ↓
Does it need browser APIs?
  (window, localStorage, geolocation)
    YES → Client Component
    NO ↓
Does it use hooks?
    YES → Client Component
    NO → Server Component (default)
```

| UI element | Component type |
| ---------- | -------------- |
| Static header, footer | Server |
| Product list from API | Server |
| Search input with live filtering | Client |
| Modal with open/close state | Client |
| Blog post content (markdown) | Server |
| Theme toggle (dark/light) | Client |
| Data table with sort (no server round-trip) | Client |
| Data table rendered from server fetch | Server |

### Default strategy

**Start as Server Component. Add `'use client'` only to the smallest interactive leaf.**

```tsx
// app/products/page.tsx — Server Component
import { ProductGrid } from '@/components/ProductGrid'
import { AddToCartButton } from '@/components/AddToCartButton'

export default async function ProductsPage() {
  const products = await getProducts()

  return (
    <ProductGrid
      products={products}
      renderAction={(product) => (
        <AddToCartButton productId={product.id} />
      )}
    />
  )
}
```

Only `AddToCartButton` needs `'use client'` — not the entire page.

# 5. Composing Server and Client Components

### Rule 1: Server Components can import Client Components

```tsx
// app/page.tsx — Server Component
import { Counter } from '@/components/Counter'

export default function HomePage() {
  return (
    <main>
      <h1>Home</h1>
      <Counter />
    </main>
  )
}
```

The server renders the static `<h1>` and a placeholder for `<Counter>`. The client hydrates `Counter`.

### Rule 2: Client Components cannot import Server Components

```tsx
// components/Dashboard.tsx
'use client'

// ❌ ERROR — cannot import Server Component into Client Component
import { ServerStats } from './ServerStats'

export function Dashboard() {
  return <ServerStats />
}
```

**Fix:** Pass Server Component output as `children` or props from a parent Server Component.

```tsx
// app/dashboard/page.tsx — Server Component
import { DashboardShell } from '@/components/DashboardShell'
import { ServerStats } from '@/components/ServerStats'

export default function DashboardPage() {
  return (
    <DashboardShell>
      <ServerStats />
    </DashboardShell>
  )
}
```

```tsx
// components/DashboardShell.tsx
'use client'

export function DashboardShell({ children }: { children: React.ReactNode }) {
  const [tab, setTab] = useState('overview')

  return (
    <div>
      <nav>{/* tab buttons */}</nav>
      {children}
    </div>
  )
}
```

### Rule 3: Server Components can be children of Client Components via composition

```text
Server Page
  └── Client Shell (use client)
        └── Server Content passed as children ✅
```

This is the **composition pattern** — the most important Server/Client boundary technique.

# 6. File Structure: One Component Per File

Follow the same convention as react-project-study — one main component per file.

```text
components/
├── ui/
│   ├── Button.tsx
│   └── Card.tsx
├── products/
│   ├── ProductCard.tsx
│   ├── ProductGrid.tsx
│   └── AddToCartButton.tsx   # 'use client'
└── layout/
    ├── Header.tsx
    └── Footer.tsx
```

| Guideline | Reason |
| --------- | ------ |
| One component per file | Easier to find, test, and tree-shake |
| Group by feature | `components/products/` for product-related UI |
| Shared UI in `components/ui/` | Buttons, inputs, cards used everywhere |
| `'use client'` only where needed | Keeps client bundle small |

Compare with react-project-study:

```text
react-project-study/src/
├── components/
│   ├── Home/
│   │   ├── Banner.tsx
│   │   └── Menu.tsx
│   └── FilterableProductTable/
│       ├── SearchBar.tsx
│       └── ProductRow.tsx
└── pages/
    └── FilterableProductTable.tsx
```

Next.js adds the rule: mark interactive components with `'use client'`.

# 7. Named Exports vs Default Exports

Both patterns work. This course follows project conventions consistently.

### Default export — pages and primary component

```tsx
// app/about/page.tsx
export default function AboutPage() {
  return <h1>About</h1>
}
```

```tsx
// components/ProductCard.tsx
export default function ProductCard({ name, price }: ProductCardProps) {
  return (
    <div>
      <h3>{name}</h3>
      <p>{price}</p>
    </div>
  )
}
```

### Named export — utilities, multiple exports, subcomponents

```tsx
// components/ui/Button.tsx
interface ButtonProps {
  label: string
  onClick?: () => void
  variant?: 'primary' | 'secondary'
}

export function Button({ label, onClick, variant = 'primary' }: ButtonProps) {
  return (
    <button className={`btn btn-${variant}`} onClick={onClick}>
      {label}
    </button>
  )
}
```

```tsx
// lib/formatPrice.ts
export function formatPrice(amount: number) {
  return `Rs. ${amount.toLocaleString('en-IN')}`
}

export const CURRENCY = 'INR'
```

| Pattern | When to use |
| ------- | ----------- |
| Default export | `page.tsx`, `layout.tsx`, one main component per file |
| Named export | UI library components, hooks, utilities, multiple exports |

### Import examples

```tsx
import ProductCard from '@/components/ProductCard'           // default
import { Button } from '@/components/ui/Button'              // named
import { formatPrice, CURRENCY } from '@/lib/formatPrice'    // named
```

Next.js `page.tsx` and `layout.tsx` **must** use default exports.

# 8. Component Naming Conventions

| Rule | Example |
| ---- | ------- |
| PascalCase for components | `ProductCard`, `AddToCartButton` |
| kebab-case for file names (optional) or PascalCase | `product-card.tsx` or `ProductCard.tsx` |
| Descriptive names | `UserProfileForm` not `Form2` |
| Suffix for client-only patterns | `ThemeToggleClient.tsx` (optional clarity) |
| Hook files prefixed with `use` | `useCart.ts`, `useDebounce.ts` |

```tsx
// ✅ Good names
function ProductCard() {}
function SearchBar() {}
function LoadingSpinner() {}

// ❌ Avoid
function Card1() {}
function Comp() {}
function MyComponent() {}
```

Pages follow the route name:

```text
app/products/page.tsx     →  ProductsPage (or default export function name)
app/about/page.tsx        →  AboutPage
```

Function names are for debugging — routing uses the **file path**, not the function name.

# 9. Practical Example: Product Page Composition

```tsx
// lib/products.ts — server-only data function
export async function getProducts() {
  const res = await fetch('https://fakestoreapi.com/products')
  return res.json()
}
```

```tsx
// components/products/ProductCard.tsx — Server Component
interface ProductCardProps {
  name: string
  price: number
  image: string
}

export function ProductCard({ name, price, image }: ProductCardProps) {
  return (
    <article className="border rounded-lg p-4">
      <img src={image} alt={name} className="w-full h-48 object-cover" />
      <h3 className="font-semibold mt-2">{name}</h3>
      <p className="text-green-600">Rs. {price}</p>
    </article>
  )
}
```

```tsx
// components/products/WishlistButton.tsx — Client Component
'use client'

import { useState } from 'react'

interface WishlistButtonProps {
  productId: number
}

export function WishlistButton({ productId }: WishlistButtonProps) {
  const [isSaved, setIsSaved] = useState(false)

  function handleToggle() {
    setIsSaved(!isSaved)
    // later: call server action
  }

  return (
    <button onClick={handleToggle} aria-pressed={isSaved}>
      {isSaved ? 'Saved ♥' : 'Save ♡'}
    </button>
  )
}
```

```tsx
// app/products/page.tsx — Server Component orchestrator
import { getProducts } from '@/lib/products'
import { ProductCard } from '@/components/products/ProductCard'
import { WishlistButton } from '@/components/products/WishlistButton'

export default async function ProductsPage() {
  const products = await getProducts()

  return (
    <main>
      <h1>Products</h1>
      <div className="grid grid-cols-3 gap-4">
        {products.map((product) => (
          <div key={product.id}>
            <ProductCard
              name={product.title}
              price={product.price}
              image={product.image}
            />
            <WishlistButton productId={product.id} />
          </div>
        ))}
      </div>
    </main>
  )
}
```

# 10. Common Mistakes

| Mistake | Error / symptom | Fix |
| ------- | --------------- | --- |
| `useState` in page.tsx without `'use client'` | Build error | Add `'use client'` or extract to client component |
| `'use client'` on entire page | Large client bundle | Move directive to smallest interactive child |
| Import server DB code in client component | Bundle error or leaked secrets | Keep data fetching in Server Components |
| Import Server Component inside Client Component | Runtime error | Use children composition pattern |
| Async Client Component | Not allowed | Fetch in Server Component; pass data as props |

### Wrong: entire page is client

```tsx
// ❌ app/products/page.tsx
'use client'

export default function ProductsPage() {
  const [products, setProducts] = useState([])
  useEffect(() => { fetchProducts().then(setProducts) }, [])
  return <ProductList products={products} />
}
```

### Right: server fetch + client interactivity

```tsx
// ✅ app/products/page.tsx — Server Component
export default async function ProductsPage() {
  const products = await getProducts()
  return <ProductList products={products} />
}
```

# 11. Comparison with react-project-study

| Aspect | react-project-study | Next.js App Router |
| ------ | ------------------- | ------------------ |
| Default runtime | Browser (client) | Server |
| Interactivity | All components can use hooks | Only `'use client'` components |
| Data in components | `useEffect` + fetch | `async` Server Components |
| Entry | `main.tsx` renders all | Framework splits server/client automatically |
| Component files | Same JSX syntax | Same JSX + `'use client'` boundary |

Components like `SearchBar.tsx` and `ProductRow.tsx` in react-project-study would remain Client Components in Next.js because they handle user input and filtering.

Static display components like `ProductCard.tsx` can become Server Components — no `'use client'` needed.

---

# 12. Best Practices

✔ **Default to Server Components** — add `'use client'` only when required

✔ Keep client components **small and leaf-level** (buttons, forms, toggles)

✔ Use the **children composition pattern** to nest server UI inside client shells

✔ Colocate components in `components/` — pages in `app/`

✔ One main component per file; named exports for shared UI libraries

✔ Never import server-only modules (`fs`, database clients) into client files

✔ Pages (`page.tsx`) should usually stay Server Components

✔ Put `'use client'` at the **first line** of the file

# 14. Summary

| Concept | Key takeaway |
| ------- | ------------ |
| Server Components | Default — run on server, no hooks, can be async |
| Client Components | `'use client'` — hooks, events, browser APIs |
| Composition | Server can import Client; Client receives Server via children |
| File structure | One component per file; feature folders under `components/` |
| Exports | Default for pages; named for shared UI and utilities |
| Minimize client | Only mark interactive leaves as client |
| vs plain React | Next.js adds server default — biggest architectural change |
