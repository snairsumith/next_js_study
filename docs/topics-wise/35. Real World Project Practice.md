# Real World Project Practice - Teaching Notes

## Topic 35: Guided Projects That Mirror Industry Work

---

# 1. Why This Topic Matters

Tutorials teach isolated concepts. Real projects combine everything:

* App Router layouts and dynamic routes
* Server Components with parallel data fetching
* Server Actions for mutations
* Auth, caching, error states, and deployment

Topic 35 bridges syllabus knowledge to portfolio-ready work. Students pick guided projects, implement incrementally, and apply Topics 1–34 in realistic combinations.

This lesson describes eight practice project archetypes with scope, routes, tech choices, and milestones.

---

# 2. How to Approach Each Project

Use the same delivery loop for every project:

```text
1. Plan routes + data model (on paper or FigJam)
2. Scaffold with create-next-app + folder structure (Topic 27)
3. Build read paths first (pages, fetch, loading UI)
4. Add write paths (Server Actions, validation)
5. Add auth if required (Topics 25–26)
6. Add tests for critical flows (Topic 31)
7. Optimize images/fonts (Topic 32)
8. Build, deploy, monitor (Topics 33–34)
```

| Phase | Deliverable |
| ----- | ----------- |
| MVP | Core read + one write flow |
| V1 | Auth, validation, error/empty states |
| V2 | Tests, SEO metadata, deploy |

Do not skip loading and error UI — interviewers notice.

# 3. Project 1 — Blog with MDX or CMS

### Description

Public blog with post list, post detail, and optional admin authoring.

### Suggested routes

```text
app/
├── page.tsx                    # Home / featured posts
├── blog/
│   ├── page.tsx                # All posts
│   └── [slug]/page.tsx         # Single post
└── admin/
    └── posts/
        ├── page.tsx            # Manage posts (auth)
        └── new/page.tsx        # Create post
```

### Tech choices

| Option | When |
| ------ | ---- |
| MDX files in repo | Simple, git-based content |
| Headless CMS (Sanity, Contentful) | Non-developer editors |
| Markdown + gray-matter | Middle ground |

### Key Next.js features

* `generateStaticParams` for post slugs
* `generateMetadata` for SEO per post
* ISR with `revalidate: 3600` for CMS content

### Milestones

1. Render MDX/markdown posts from `content/` folder
2. Add tag filtering via `searchParams`
3. Admin create/edit with Server Action + auth

# 4. Project 2 — E-commerce Storefront

### Description

Product catalog, product detail, cart (client state), checkout flow (simplified).

### Suggested routes

```text
app/
├── products/
│   ├── page.tsx                # Catalog + filters
│   └── [id]/page.tsx           # Product detail
├── cart/page.tsx               # Cart review
└── checkout/
    ├── page.tsx                # Checkout form
    └── success/page.tsx        # Order confirmation
```

### Key patterns

| Feature | Implementation |
| ------- | -------------- |
| Product list | Server Component + DB/API |
| Filters | `searchParams` for category, sort |
| Cart | Client Component + Zustand or context |
| Checkout | Server Action creates order |

### Stretch goals

* Stripe integration (payment intent on server)
* Inventory check before checkout
* Order history for logged-in users

# 5. Project 3 — Dashboard with Auth

### Description

Private dashboard showing user-specific metrics, tables, and settings.

### Suggested routes

```text
app/
├── login/page.tsx
├── (dashboard)/
│   ├── layout.tsx              # Auth guard + sidebar
│   ├── dashboard/page.tsx      # Stats overview
│   ├── reports/page.tsx        # Data table
│   └── settings/page.tsx       # Profile form
```

### Key patterns

| Feature | Implementation |
| ------- | -------------- |
| Auth | Auth.js + middleware |
| Layout guard | `auth()` in dashboard layout |
| Stats | Parallel fetch with `Promise.all` |
| Charts | Dynamic import — `ssr: false` |

### UI expectations

* Skeleton loaders in `loading.tsx`
* Empty state when no data
* Responsive sidebar collapse on mobile

# 6. Project 4 — Todo App with Server Actions

### Description

Classic todo app — ideal first full-stack Next.js project. Focus on mutations without complex domain logic.

### Suggested routes

```text
app/
├── page.tsx                    # Todo list + add form
└── todos/
    └── [id]/page.tsx           # Optional detail view
```

### Server Actions

```typescript
'use server'

export async function addTodoAction(formData: FormData) { /* ... */ }
export async function toggleTodoAction(id: string) { /* ... */ }
export async function deleteTodoAction(id: string) { /* ... */ }
```

### Key learning goals

* Optimistic UI optional (client) vs `revalidatePath` (server)
* Zod validation on todo title length
* `error.tsx` for failed mutations

### Database

SQLite + Prisma or Drizzle for zero-config local dev.

# 7. Project 5 — Weather App with External API

### Description

Search city → show current weather and forecast. Demonstrates external API integration and URL state.

### Suggested routes

```text
app/
├── page.tsx                    # Search + default city
└── weather/page.tsx            # ?city=London
```

### Key patterns

| Feature | Implementation |
| ------- | -------------- |
| API key | Server-only env var |
| Fetch | Server Component — never expose key client-side |
| Search | Form GET or `searchParams` |
| Cache | `revalidate: 900` (15 min) for weather data |

```typescript
// app/weather/page.tsx
export default async function WeatherPage({
  searchParams,
}: {
  searchParams: Promise<{ city?: string }>
}) {
  const { city = 'Mumbai' } = await searchParams
  const weather = await getWeather(city) // server-only fetch
  return <WeatherDisplay data={weather} />
}
```

### Client enhancement

Debounced search input (`useDebounce`) updating URL with `router.push`.

# 8. Project 6 — Admin Panel with Role-Based Access

### Description

Manage users, content, or inventory. Only admins and editors access `/admin`.

### Suggested routes

```text
app/admin/
├── layout.tsx                  # Role check
├── page.tsx                    # Overview
├── users/page.tsx              # User management
└── content/page.tsx            # CRUD table
```

### Key patterns

| Feature | Implementation |
| ------- | -------------- |
| RBAC | Session role from Auth.js callbacks |
| Middleware | Block `/admin` for non-staff |
| Data table | Server fetch + client pagination |
| Destructive actions | Confirm modal + Server Action |

### Security checklist

✔ Verify role in every admin Server Action

✔ Audit log for delete operations (optional)

✔ Generic 403 page — do not leak admin existence

# 9. Project 7 — Authentication-Based Mini App

### Description

Small app where **all value is behind login** — notes app, bookmark manager, or habit tracker.

### Scope (keep small)

* Register / login / logout
* CRUD on one entity owned by user
* Profile settings page

### Key patterns

```text
Middleware → session cookie check
Layout → auth() + redirect
Actions → verify session.user.id owns resource
```

### Deliverables

* Working OAuth or credentials login
* At least three Server Actions (create, update, delete)
* Protected Route Handler example

Good rehearsal for capstone if auth is required.

# 10. Project 8 — Full-Stack CRUD Application

### Description

Complete resource lifecycle — e.g., job board, event manager, or inventory system.

### Minimum features

| CRUD | Route / action |
| ---- | -------------- |
| Create | Form + Server Action |
| Read | List + detail pages |
| Update | Edit form + action |
| Delete | Confirm + action |

### Architecture (Topic 27)

```text
features/jobs/
├── components/
├── actions/
├── lib/jobs-service.ts
└── types/
```

### Quality bar

* Zod validation on all writes
* `notFound()` for invalid IDs
* Tests for create and delete actions
* Deployed with env vars documented

# 12. Practice Exercise — Pick and Plan

2. Route map (folder structure)
3. Data model (entities and fields)
