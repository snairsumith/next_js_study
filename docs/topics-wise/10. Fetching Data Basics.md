# Fetching Data Basics - Teaching Notes

## Topic 10: Server Component Data Fetching, Loading, and Errors

---

# 1. Why This Topic Matters

Data fetching is the core of most web applications. In Next.js App Router, the **default and recommended** place to fetch data is the **server** — inside Server Components, not in `useEffect`.

Students who default to client-side fetching miss:

* Faster initial page loads (HTML arrives with data)
* Better SEO (content in server-rendered HTML)
* Simpler code (no loading state boilerplate in many cases)
* Secure API keys (never exposed to browser)

This topic teaches the beginner fetch patterns that replace react-project-study's `useEffect` + fetch approach.

---

# 2. Fetching in Server Components

Server Components can be **`async`** and use **`await`** directly.

```tsx
// app/users/page.tsx
interface User {
  id: number
  name: string
  email: string
}

async function getUsers(): Promise<User[]> {
  const res = await fetch('https://jsonplaceholder.typicode.com/users')

  if (!res.ok) {
    throw new Error('Failed to fetch users')
  }

  return res.json()
}

export default async function UsersPage() {
  const users = await getUsers()

  return (
    <main>
      <h1>Users</h1>
      <ul>
        {users.map((user) => (
          <li key={user.id}>
            <strong>{user.name}</strong> — {user.email}
          </li>
        ))}
      </ul>
    </main>
  )
}
```

| Server fetch advantage | Detail |
| ---------------------- | ------ |
| Runs on server only | API keys safe in env vars |
| No `useEffect` | Data available on first render |
| No client loading flash | HTML includes data |
| Parallel with other server work | Multiple awaits in same component |

Compare with react-project-study:

```tsx
// react-project-study/src/pages/DataFetch.tsx — client-side pattern
const [users, setUsers] = useState([])
const [isLoading, setIsLoading] = useState(true)

useEffect(() => {
  fetch('https://jsonplaceholder.typicode.com/users')
    .then(res => res.json())
    .then(data => {
      setUsers(data)
      setIsLoading(false)
    })
}, [])

if (isLoading) return <Loading />
```

Next.js Server Component eliminates manual loading state for the initial fetch.

# 3. async Page Components

In App Router, **`page.tsx` can export an async default function**.

```tsx
// app/products/page.tsx
export default async function ProductsPage() {
  const products = await fetchProducts()
  return <ProductList products={products} />
}
```

| Rule | Detail |
| ---- | ------ |
| Only Server Components can be async | Client Components cannot be `async` |
| One await or many | Sequential fetches run in order |
| Extract fetch logic | Move to `lib/` or colocated `_lib/` helpers |
| Type return values | Use TypeScript interfaces for API responses |

### Extracting fetch helpers

```tsx
// lib/products.ts
export interface Product {
  id: number
  title: string
  price: number
  description: string
}

export async function getProducts(): Promise<Product[]> {
  const res = await fetch('https://fakestoreapi.com/products')

  if (!res.ok) {
    throw new Error(`Products fetch failed: ${res.status}`)
  }

  return res.json()
}

export async function getProductById(id: string): Promise<Product | null> {
  const res = await fetch(`https://fakestoreapi.com/products/${id}`)

  if (res.status === 404) return null
  if (!res.ok) throw new Error(`Product fetch failed: ${res.status}`)

  return res.json()
}
```

```tsx
// app/products/page.tsx
import { getProducts } from '@/lib/products'
import { ProductList } from '@/components/products/ProductList'

export default async function ProductsPage() {
  const products = await getProducts()
  return (
    <main>
      <h1>Products</h1>
      <ProductList products={products} />
    </main>
  )
}
```

Same pattern as react-project-study's `api/index.ts` — but called from server, not browser.

# 4. Displaying Fetched Data in UI

Pass fetched data as **props** to display components.

```tsx
// components/products/ProductList.tsx — Server Component
import { Product } from '@/lib/products'
import { ProductCard } from './ProductCard'

interface ProductListProps {
  products: Product[]
}

export function ProductList({ products }: ProductListProps) {
  if (products.length === 0) {
    return <p>No products found.</p>
  }

  return (
    <div className="grid grid-cols-1 md:grid-cols-3 gap-4">
      {products.map((product) => (
        <ProductCard key={product.id} product={product} />
      ))}
    </div>
  )
}
```

```tsx
// components/products/ProductCard.tsx
import { Product } from '@/lib/products'

interface ProductCardProps {
  product: Product
}

export function ProductCard({ product }: ProductCardProps) {
  return (
    <article className="border rounded-lg p-4">
      <h2 className="font-semibold">{product.title}</h2>
      <p className="text-gray-600 text-sm line-clamp-2">{product.description}</p>
      <p className="text-green-600 font-bold mt-2">Rs. {product.price}</p>
    </article>
  )
}
```

| UI state | Handling |
| -------- | -------- |
| Data loaded | Render list (default — server waits before sending HTML) |
| Empty array | Show empty state message |
| Error | Throw error → caught by `error.tsx` |
| Loading | Handled by `loading.tsx` (next section) |

# 5. Loading States with loading.tsx

When a page fetch is slow, show a loading UI with **`loading.tsx`**.

```tsx
// app/products/loading.tsx
export default function ProductsLoading() {
  return (
    <div className="p-8">
      <div className="h-8 w-48 bg-gray-200 animate-pulse rounded mb-6" />
      <div className="grid grid-cols-3 gap-4">
        {Array.from({ length: 6 }).map((_, i) => (
          <div key={i} className="h-40 bg-gray-200 animate-pulse rounded" />
        ))}
      </div>
    </div>
  )
}
```

```tsx
// app/products/page.tsx — slow fetch triggers loading.tsx
export default async function ProductsPage() {
  const products = await getProducts() // if slow, loading.tsx shows
  return <ProductList products={products} />
}
```

| loading.tsx behavior | Detail |
| -------------------- | ------ |
| Automatic Suspense | Next.js wraps page in Suspense boundary |
| Instant layout | Parent layouts render immediately |
| Streaming | HTML streams as data becomes ready |
| Scope | Applies to route segment and children |

```text
User navigates to /products
       │
       ▼
Layout renders immediately (header, sidebar)
       │
       ▼
loading.tsx shows in page slot
       │
       ▼
page.tsx fetch completes
       │
       ▼
Product list replaces loading UI
```

Compare with react-project-study manual pattern:

```tsx
// react-project-study — manual isLoading state
if (isLoading) return <Loading />
if (error) return <ErrorMessage />
return <DataList data={data} />
```

Next.js splits this into **convention files**: `loading.tsx`, `error.tsx`, `page.tsx`.

# 6. Error Handling for Fetch Failures

Handle fetch errors by **throwing** in Server Components. Next.js catches them in **`error.tsx`**.

```tsx
// lib/users.ts
export async function getUsers() {
  const res = await fetch('https://jsonplaceholder.typicode.com/users', {
    next: { revalidate: 60 },
  })

  if (!res.ok) {
    throw new Error(`Failed to fetch users: ${res.status} ${res.statusText}`)
  }

  return res.json()
}
```

```tsx
// app/users/error.tsx
'use client'

interface ErrorProps {
  error: Error & { digest?: string }
  reset: () => void
}

export default function UsersError({ error, reset }: ErrorProps) {
  return (
    <div role="alert" className="p-8 text-center">
      <h2 className="text-xl font-bold text-red-600">Failed to load users</h2>
      <p className="text-gray-600 mt-2">{error.message}</p>
      <button
        onClick={reset}
        className="mt-4 px-4 py-2 bg-brand-500 text-white rounded"
      >
        Try again
      </button>
    </div>
  )
}
```

| Error strategy | When to use |
| -------------- | ----------- |
| Throw in fetch helper | Unexpected API failures → `error.tsx` |
| Return null / empty | Expected missing resource → `notFound()` |
| try/catch in page | Graceful fallback UI inline (rare) |

### try/catch with inline fallback (optional)

```tsx
export default async function UsersPage() {
  try {
    const users = await getUsers()
    return <UserList users={users} />
  } catch {
    return (
      <main>
        <h1>Users</h1>
        <p className="text-red-600">Could not load users. Please try again later.</p>
      </main>
    )
  }
}
```

Prefer **`error.tsx`** for consistent error UX across a route segment.

# 7. External API vs Local Data

| Source | Example | Access pattern |
| ------ | ------- | -------------- |
| External REST API | JSONPlaceholder, FakeStore | `fetch(url)` in Server Component |
| Local JSON file | `data/products.json` | `fs.readFile` in Server Component |
| Database | PostgreSQL, MongoDB | DB client in Server Component / lib |
| Internal Route Handler | `/api/users` | `fetch('http://localhost:3000/api/users')` or direct function call |

### External API

```tsx
// app/posts/page.tsx
async function getPosts() {
  const res = await fetch('https://jsonplaceholder.typicode.com/posts')
  return res.json()
}

export default async function PostsPage() {
  const posts = await getPosts()
  return (
    <ul>
      {posts.slice(0, 10).map((post: { id: number; title: string }) => (
        <li key={post.id}>{post.title}</li>
      ))}
    </ul>
  )
}
```

### Local data (server-only)

```tsx
// lib/localData.ts
import products from '@/data/products.json'

export function getLocalProducts() {
  return products
}
```

```tsx
// app/local-products/page.tsx
import { getLocalProducts } from '@/lib/localData'

export default function LocalProductsPage() {
  const products = getLocalProducts()
  return <ProductList products={products} />
}
```

Reading local files with `fs` requires server context — works in Server Components, not in client.

### Prefer direct function over self-fetch

```tsx
// ❌ Avoid — unnecessary HTTP round-trip to own API
const res = await fetch('http://localhost:3000/api/products')

// ✅ Better — call the data function directly
const products = await getProducts()
```

Route Handlers (`route.ts`) are for external consumers and client fetch — not for server pages calling themselves.

# 8. Avoid Fetch in Client When Server Can Handle It

**Anti-pattern:** Fetching on the client data the server could fetch during render.

```tsx
// ❌ Client fetch for initial page data — avoid for static/semi-static content
'use client'

export default function ProductsPage() {
  const [products, setProducts] = useState([])
  useEffect(() => {
    fetch('/api/products').then(r => r.json()).then(setProducts)
  }, [])
  return <ProductList products={products} />
}
```

```tsx
// ✅ Server fetch — preferred for initial page data
export default async function ProductsPage() {
  const products = await getProducts()
  return <ProductList products={products} />
}
```

| Scenario | Fetch location |
| -------- | -------------- |
| Initial page content | Server Component |
| User-triggered refresh | Client (`router.refresh()`) or Server Action |
| Real-time updates (chat, live scores) | Client (WebSocket, polling) |
| Infinite scroll | Client (TanStack Query — Topic 14) |
| Form submission response | Server Action |

Client fetch is still valid for **interactive, user-driven** data needs — not for the default page load.

# 9. Fetch Options and Caching Preview

Next.js extends `fetch` with caching options. Full details in Topic 24 — basics here:

```tsx
// Default — cached (static feel)
const res = await fetch('https://api.example.com/products')

// Always fresh — SSR on every request
const res = await fetch('https://api.example.com/products', {
  cache: 'no-store',
})

// Revalidate every 60 seconds — ISR
const res = await fetch('https://api.example.com/products', {
  next: { revalidate: 60 },
})
```

| Option | Behavior |
| ------ | -------- |
| Default cache | Next.js caches fetch result |
| `cache: 'no-store'` | Dynamic — fetch every request |
| `next: { revalidate: N }` | Regenerate after N seconds |

For training exercises with live APIs, `cache: 'no-store'` ensures fresh data during development.

# 11. Comparison with react-project-study

| Aspect | react-project-study | Next.js Server fetch |
| ------ | ------------------- | -------------------- |
| Where fetch runs | Browser (`useEffect`) | Server (`async` page) |
| Loading UI | `useState(isLoading)` | `loading.tsx` |
| Error UI | `useState(error)` | `error.tsx` |
| API helpers | `src/api/index.ts` | `lib/` — called on server |
| Data on first paint | Empty until JS loads | HTML includes data |
| Secrets in fetch | Exposed if in client | Safe in server env vars |

```tsx

// Uses useEffect + useState — good for learning React
// Next.js replaces this pattern for page-level data
```

Students should recognize both patterns — but default to server fetch in Next.js.

# 12. Best Practices

✔ Fetch in **Server Components** for initial page data

✔ Extract fetch logic to **`lib/`** functions with typed returns

✔ **Throw errors** for failed fetches — let `error.tsx` handle UI

✔ Add **`loading.tsx`** for routes with slow or external API calls

✔ Use **`cache: 'no-store'`** during development for live API data

✔ Call **data functions directly** — avoid self-fetch to own Route Handlers

✔ Handle **empty arrays** with user-friendly empty states

✔ Keep client fetch for **interactive, user-driven** updates only

# 14. Summary

| Concept | Key takeaway |
| ------- | ------------ |
| Server fetch | Default in `async` Server Components |
| async page.tsx | `await fetch()` directly — no useEffect |
| Display data | Pass as props to child components |
| loading.tsx | Automatic loading UI via Suspense |
| error.tsx | Catch thrown fetch errors — Client Component |
| External vs local | fetch API, JSON imports, or DB on server |
| Avoid client fetch | Server handles initial page data |
| vs React | Replaces useEffect pattern for page loads |
