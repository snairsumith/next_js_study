# Authentication with Auth.js - Teaching Notes

## Topic 26: Production Auth with Auth.js (NextAuth v5)

---

# 1. Why This Topic Matters

Building auth from scratch works for learning, but production apps need:

* OAuth providers (Google, GitHub)
* Secure session rotation
* CSRF protection
* Callback hooks for custom logic
* Type-safe session access in Server Components

**Auth.js** (formerly NextAuth.js) is the standard auth library for Next.js. It integrates with the App Router, middleware, and Server Components.

This lesson covers setup, providers, sessions, protected routes, roles, and callbacks.

---

# 2. What Is Auth.js?

Auth.js is an open-source authentication library for Next.js and other frameworks.

| Feature | Benefit |
| ------- | ------- |
| Multiple providers | Google, GitHub, credentials, and more |
| Session strategies | JWT or database sessions |
| App Router support | `auth()` helper for Server Components |
| Middleware helper | `auth` wrapper for route protection |
| Callbacks | Customize sign-in, JWT, and session logic |

```text
Auth.js Request Flow
────────────────────
User clicks "Sign in with Google"
       │
       ▼
Redirect to Google OAuth
       │
       ▼
Google returns to /api/auth/callback/google
       │
       ▼
Auth.js creates session → sets cookie
       │
       ▼
User redirected to app → auth() reads session on server
```

# 3. Installation and Project Setup

```bash
npm install next-auth@beta
# Auth.js v5 (Auth.js for Next.js) — check docs for latest package name
```

Generate a secret for signing tokens:

```bash
openssl rand -base64 32
```

Add to `.env.local`:

```env
AUTH_SECRET=your-generated-secret-here
AUTH_URL=http://localhost:3000

# OAuth providers (optional)
AUTH_GOOGLE_ID=your-google-client-id
AUTH_GOOGLE_SECRET=your-google-client-secret
AUTH_GITHUB_ID=your-github-client-id
AUTH_GITHUB_SECRET=your-github-client-secret
```

Never commit `.env.local`. Add `.env.example` with placeholder keys for teammates.

# 4. Auth.js Configuration

Create the central auth config file.

```typescript
// auth.ts (project root)
import NextAuth from 'next-auth'
import Google from 'next-auth/providers/google'
import GitHub from 'next-auth/providers/github'
import Credentials from 'next-auth/providers/credentials'
import { verifyCredentials } from '@/lib/auth/verify'

export const { handlers, auth, signIn, signOut } = NextAuth({
  providers: [
    Google,
    GitHub,
    Credentials({
      credentials: {
        email: { label: 'Email', type: 'email' },
        password: { label: 'Password', type: 'password' },
      },
      async authorize(credentials) {
        const user = await verifyCredentials(
          credentials.email as string,
          credentials.password as string
        )
        if (!user) return null
        return { id: user.id, name: user.name, email: user.email }
      },
    }),
  ],
  pages: {
    signIn: '/login',
    error: '/login',
  },
  session: {
    strategy: 'jwt', // or 'database' with adapter
    maxAge: 30 * 24 * 60 * 60, // 30 days
  },
})
```

### Route Handler for Auth.js

```typescript
// app/api/auth/[...nextauth]/route.ts
import { handlers } from '@/auth'

export const { GET, POST } = handlers
```

# 5. Providers: Credentials, Google, GitHub

### Credentials provider

Best for email/password login. You validate credentials in `authorize()`.

| Step | Action |
| ---- | ------ |
| 1 | User submits form |
| 2 | `authorize()` checks database |
| 3 | Return user object or `null` |
| 4 | Auth.js creates session |

### OAuth providers (Google, GitHub)

User clicks provider → redirect to OAuth → callback creates session. Configure client ID/secret in `.env.local` and provider dashboard redirect URIs.

### Sign-in buttons

```tsx
// components/sign-in-buttons.tsx
import { signIn } from '@/auth'

export function SignInButtons() {
  return (
    <div className="flex flex-col gap-2">
      <form
        action={async () => {
          'use server'
          await signIn('google', { redirectTo: '/dashboard' })
        }}
      >
        <button type="submit">Continue with Google</button>
      </form>
      <form
        action={async () => {
          'use server'
          await signIn('github', { redirectTo: '/dashboard' })
        }}
      >
        <button type="submit">Continue with GitHub</button>
      </form>
    </div>
  )
}
```

# 6. Session Management

Auth.js stores session in an encrypted **httpOnly cookie** (JWT strategy) or in a database (database strategy).

### Reading session in Server Components

```typescript
// app/dashboard/page.tsx
import { auth } from '@/auth'
import { redirect } from 'next/navigation'

export default async function DashboardPage() {
  const session = await auth()
  if (!session?.user) redirect('/login')

  return (
    <main>
      <h1>Hello, {session.user.name}</h1>
      <p>{session.user.email}</p>
    </main>
  )
}
```

### Sign out

```tsx
// components/sign-out-button.tsx
import { signOut } from '@/auth'

export function SignOutButton() {
  return (
    <form
      action={async () => {
        'use server'
        await signOut({ redirectTo: '/login' })
      }}
    >
      <button type="submit">Sign out</button>
    </form>
  )
}
```

# 7. Protecting Pages and API Routes

### Middleware with Auth.js

```typescript
// middleware.ts
import { auth } from '@/auth'
import { NextResponse } from 'next/server'

export default auth((req) => {
  const isLoggedIn = !!req.auth
  const isProtected = req.nextUrl.pathname.startsWith('/dashboard')

  if (isProtected && !isLoggedIn) {
    return NextResponse.redirect(new URL('/login', req.url))
  }

  return NextResponse.next()
})

export const config = {
  matcher: ['/dashboard/:path*', '/admin/:path*'],
}
```

### Protected Route Handler

```typescript
// app/api/posts/route.ts
import { auth } from '@/auth'
import { NextResponse } from 'next/server'

export async function POST(request: Request) {
  const session = await auth()
  if (!session?.user) {
    return NextResponse.json({ error: 'Unauthorized' }, { status: 401 })
  }

  const body = await request.json()
  // Create post for session.user.id
  return NextResponse.json({ success: true })
}
```

# 8. Role-Based Access Control

Extend the session with a `role` field using callbacks.

### Type augmentation

```typescript
// types/next-auth.d.ts
import 'next-auth'

declare module 'next-auth' {
  interface User {
    role?: string
  }
  interface Session {
    user: {
      id: string
      name?: string | null
      email?: string | null
      role?: string
    }
  }
}
```

### JWT and session callbacks

```typescript
// auth.ts (callbacks section)
callbacks: {
  async jwt({ token, user }) {
    if (user) {
      token.role = user.role
      token.id = user.id
    }
    return token
  },
  async session({ session, token }) {
    if (session.user) {
      session.user.role = token.role as string
      session.user.id = token.id as string
    }
    return session
  },
},
```

### Admin-only page

```tsx
// app/admin/page.tsx
import { auth } from '@/auth'
import { redirect } from 'next/navigation'

export default async function AdminPage() {
  const session = await auth()
  if (!session?.user) redirect('/login')
  if (session.user.role !== 'admin') redirect('/unauthorized')

  return <h1>Admin Dashboard</h1>
}
```

| Role check location | Use case |
| ------------------- | -------- |
| Middleware | Block entire `/admin` section |
| Server Component | Page-level access |
| Server Action | Before destructive operations |

# 9. Callbacks and Custom Logic

| Callback | When it runs | Common use |
| -------- | ------------ | ---------- |
| `signIn` | Before user is signed in | Block banned emails |
| `jwt` | When JWT is created/updated | Add role, org id |
| `session` | When session is read | Expose safe fields to app |

Use `signIn` callback to reject users from blocked domains or unverified emails.

---

# 10. Best Practices

✔ Use `AUTH_SECRET` — required in production

✔ Prefer `auth()` in Server Components over client session hooks

✔ Extend session types with TypeScript module augmentation

✔ Store roles in JWT/session callbacks — not in client localStorage

✔ Use database adapter for sessions if you need instant revocation

✔ Configure OAuth redirect URIs correctly in provider dashboards

✔ Keep `authorize()` fast — it runs on every credentials login

✔ Re-check permissions in Server Actions even with middleware
