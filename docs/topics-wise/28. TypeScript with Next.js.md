# TypeScript with Next.js - Teaching Notes

## Topic 28: Type-Safe Pages, Actions, and Route Handlers

---

# 1. Why This Topic Matters

Next.js projects increasingly default to TypeScript. Types catch mistakes before runtime:

* Wrong `params` shape on dynamic routes
* Missing `searchParams` handling
* Non-serializable props passed to client components
* Untyped Server Action payloads

TypeScript in Next.js is not optional for production teams. This lesson covers setup, typing pages, actions, handlers, and shared types.

---

# 2. TypeScript Setup in create-next-app

When creating a project, choose TypeScript:

```bash
npx create-next-app@latest my-app --typescript
```

Key files:

| File | Purpose |
| ---- | ------- |
| `tsconfig.json` | Compiler options, path aliases |
| `next-env.d.ts` | Next.js type references (auto-generated) |
| `*.tsx` | Components and pages with JSX |

### Path alias (default)

```json
// tsconfig.json
{
  "compilerOptions": {
    "paths": {
      "@/*": ["./src/*"]
    }
  }
}
```

Import with `@/components/Button` instead of long relative paths.

### Strict mode

Enable `"strict": true` in `tsconfig.json`. It forces better null checks and aligns with Next.js examples.

# 3. Typing Page Props: params and searchParams

In the App Router, page components receive `params` and `searchParams` as props.

### Static route

```tsx
// app/about/page.tsx
export default function AboutPage() {
  return <h1>About</h1>
}
```

No props needed for static pages.

### Dynamic route

```tsx
// app/products/[id]/page.tsx
interface ProductPageProps {
  params: Promise<{ id: string }>
}

export default async function ProductPage({ params }: ProductPageProps) {
  const { id } = await params
  const product = await getProduct(id)

  if (!product) notFound()

  return <h1>{product.name}</h1>
}
```

> **Note:** In Next.js 15+, `params` and `searchParams` are Promises. Await them in async server components.

### searchParams

```tsx
// app/products/page.tsx
interface ProductsPageProps {
  searchParams: Promise<{ q?: string; page?: string }>
}

export default async function ProductsPage({ searchParams }: ProductsPageProps) {
  const { q = '', page = '1' } = await searchParams
  const products = await searchProducts(q, Number(page))

  return (
    <main>
      <h1>Products</h1>
      <ProductList products={products} />
    </main>
  )
}
```

| Prop | Type | Source |
| ---- | ---- | ------ |
| `params` | `{ id: string }` etc. | Dynamic route segments |
| `searchParams` | `{ key?: string }` | URL query string |

# 4. Typing Layouts

Layouts receive `children` and optionally `params`.

```tsx
// app/dashboard/layout.tsx
interface DashboardLayoutProps {
  children: React.ReactNode
  params: Promise<{ slug?: string }>
}

export default async function DashboardLayout({
  children,
}: DashboardLayoutProps) {
  return (
    <div className="flex">
      <aside>Sidebar</aside>
      <main>{children}</main>
    </div>
  )
}
```

Use `React.ReactNode` for `children` — it covers strings, elements, fragments, and null.

# 5. Typing Server Actions

Server Actions are async functions marked with `'use server'`.

### Inline action with FormData

```typescript
'use server'

export async function createUserAction(formData: FormData) {
  const name = formData.get('name')
  if (typeof name !== 'string' || !name.trim()) {
    return { error: 'Name is required' }
  }

  await saveUser({ name: name.trim() })
  return { success: true }
}
```

### Typed return for useActionState

```typescript
'use server'

export interface CreateUserState {
  error?: string
  success?: boolean
}

export async function createUserAction(
  _prevState: CreateUserState,
  formData: FormData
): Promise<CreateUserState> {
  const email = formData.get('email')
  if (typeof email !== 'string') {
    return { error: 'Invalid email' }
  }

  await saveUser({ email })
  return { success: true }
}
```

```tsx
'use client'

import { useActionState } from 'react'
import { createUserAction, type CreateUserState } from '@/app/actions/users'

const initialState: CreateUserState = {}

export function CreateUserForm() {
  const [state, formAction, isPending] = useActionState(
    createUserAction,
    initialState
  )

  return (
    <form action={formAction}>
      <input name="email" type="email" required />
      {state.error && <p role="alert">{state.error}</p>}
      <button disabled={isPending}>Create</button>
    </form>
  )
}
```

# 6. Typing Route Handlers

Route Handlers live in `route.ts` files and export HTTP method functions.

```typescript
// app/api/users/route.ts
import { NextRequest, NextResponse } from 'next/server'

interface CreateUserBody {
  name: string
  email: string
}

export async function GET() {
  const users = await getUsers()
  return NextResponse.json(users)
}

export async function POST(request: NextRequest) {
  const body = (await request.json()) as CreateUserBody

  if (!body.name || !body.email) {
    return NextResponse.json(
      { error: 'Name and email required' },
      { status: 400 }
    )
  }

  const user = await createUser(body)
  return NextResponse.json(user, { status: 201 })
}
```

For production, validate body with Zod (Topic 29) instead of `as` casts.

| Method | Signature |
| ------ | --------- |
| GET | `(request: NextRequest) => Response` |
| POST | `(request: NextRequest) => Response` |

# 7. Typing Metadata

```typescript
// app/blog/[slug]/page.tsx
import type { Metadata } from 'next'

interface BlogPageProps {
  params: Promise<{ slug: string }>
}

export async function generateMetadata({
  params,
}: BlogPageProps): Promise<Metadata> {
  const { slug } = await params
  const post = await getPost(slug)

  return {
    title: post?.title ?? 'Blog',
    description: post?.excerpt,
  }
}
```

`Metadata` type from `next` ensures valid SEO field names.

# 8. Shared Types Between Client and Server

Place shared interfaces in `types/` — no server imports in those files.

```typescript
// types/user.ts
export interface User {
  id: string
  name: string
  email: string
  role: 'user' | 'admin'
}

export interface PublicUser {
  id: string
  name: string
}
```

```typescript
// lib/users.ts (server only)
import type { User } from '@/types/user'

export async function getUser(id: string): Promise<User | null> {
  // database query
}
```

```tsx
// components/user-badge.tsx (client)
import type { PublicUser } from '@/types/user'

interface UserBadgeProps {
  user: PublicUser
}

export function UserBadge({ user }: UserBadgeProps) {
  return <span>{user.name}</span>
}
```

### Serializable props rule

Props passed from Server to Client Components must be JSON-serializable:

| Allowed | Not allowed |
| ------- | ----------- |
| string, number, boolean | Functions |
| plain objects, arrays | class instances |
| null, undefined | Date (pass ISO string instead) |

# 9. Avoiding any

`any` disables type checking. Use better alternatives:

| Instead of | Use |
| ---------- | --- |
| `any` | Proper interface or generic |
| `as any` | Zod parse + inferred type |
| `@ts-ignore` | Fix the type or use `unknown` + narrowing |

```typescript
// ❌ Avoid
function handleData(data: any) {
  return data.name
}

// ✅ Better
interface DataPayload {
  name: string
}

function handleData(data: DataPayload) {
  return data.name
}

// ✅ Unknown input from API
function parseUser(input: unknown): User | null {
  if (
    typeof input === 'object' &&
    input !== null &&
    'id' in input &&
    'email' in input
  ) {
    return input as User
  }
  return null
}
```

Enable ESLint rule `@typescript-eslint/no-explicit-any` in team projects.

# 11. Best Practices

✔ Enable `strict` mode in `tsconfig.json`

✔ Await `params` and `searchParams` in Next.js 15+ pages

✔ Define explicit return types for Server Actions used with `useActionState`

✔ Keep shared types in `types/` without server dependencies

✔ Validate external input — do not trust `request.json()` shape

✔ Use `React.ReactNode` for children

✔ Avoid `any`; use Zod for runtime validation (Topic 29)

✔ Run `npm run build` locally — TypeScript errors fail production builds

# 13. Summary

| Area | Typing approach |
| ---- | --------------- |
| Pages | `params: Promise<{ id: string }>`, `searchParams: Promise<{...}>` |
| Layouts | `children: React.ReactNode` |
| Server Actions | Return type interfaces; FormData narrowing |
| Route Handlers | Request/response types; validate JSON body |
| Metadata | `Promise<Metadata>` from `generateMetadata` |
| Shared types | `types/` folder; serializable props to client |

TypeScript catches integration bugs between Next.js layers early. Topic 29 adds generics, Zod, and end-to-end patterns.
