# Introduction to Next.js - Teaching Notes

## Topic 2: What Next.js Is and Why It Exists

---

# 1. Why This Topic Matters

React alone is a **UI library**. It renders components, but it does not tell you how to:

- Route between pages
- Fetch data on the server
- Optimize images and fonts
- Deploy a production app with SEO

**Next.js** is a **full-stack React framework** built on top of React. It answers those questions with conventions and built-in features so teams ship faster with fewer decisions.

Students who understand *why* Next.js exists — not just *how* to use it — make better choices about Server Components, caching, and when to stay on plain React.

---

# 2. What Is Next.js?

Next.js is an open-source framework created by **Vercel**. It extends React with:


| Feature                | What it gives you                                   |
| ---------------------- | --------------------------------------------------- |
| File-based routing     | Folders in `app/` become URLs automatically         |
| Server Components      | React components that run on the server             |
| Server Actions         | Functions that run on the server, called from forms |
| Route Handlers         | API endpoints in `route.ts` files                   |
| Built-in optimizations | Image, font, and script loading                     |
| Production tooling     | Build, deploy, and caching strategies               |


Next.js **15** (current stable for this course) uses the **App Router** (`app/` directory) as the primary routing model.

```text
React          =  UI layer (components, hooks, JSX)
Next.js        =  React + routing + server + build + deployment conventions
```

---

# 3. Next.js vs Plain React (Vite / CRA)

In **react-project-study**, the app is a client-side React app powered by Vite:

```tsx
// react-project-study/src/main.tsx
createRoot(document.getElementById('root')!).render(
  <StrictMode>
    <BrowserRouter>
      <App />
    </BrowserRouter>
  </StrictMode>,
)
```

You manually set up routing, data fetching, and deployment.


| Concern       | Plain React (Vite)                     | Next.js                                      |
| ------------- | -------------------------------------- | -------------------------------------------- |
| Routing       | React Router — manual `<Route>` setup  | File-based — `app/about/page.tsx` → `/about` |
| Entry point   | `main.tsx` mounts to `#root`           | Framework bootstraps from `app/layout.tsx`   |
| Data fetching | `useEffect` + fetch in browser         | `async` Server Components on the server      |
| SEO           | Empty HTML shell; content loads via JS | Server-rendered HTML sent first              |
| API           | Separate Express/Fastify backend       | Route Handlers + Server Actions built in     |
| Build output  | Static SPA in `dist/`                  | Optimized server + static hybrid             |
| Dev server    | Vite on port 5173                      | Next.js on port 3000                         |


### When plain React is enough

- Internal dashboards with no SEO needs
- Embedded widgets inside another app
- Learning React fundamentals (react-project-study)

### When Next.js is the better choice

- Public marketing sites and blogs
- E-commerce storefronts
- Apps that need fast first paint and SEO
- Full-stack apps where frontend and backend live together

# 4. Why Companies Use Next.js


| Reason                 | Business impact                                              |
| ---------------------- | ------------------------------------------------------------ |
| Faster time to market  | Routing, API, and deployment patterns are built in           |
| Better SEO             | Server-rendered HTML is crawlable by search engines          |
| Performance defaults   | Image optimization, code splitting, caching                  |
| Full-stack in one repo | Server Actions and Route Handlers reduce backend boilerplate |
| Large ecosystem        | Vercel hosting, Auth.js, CMS integrations                    |
| Industry adoption      | Used by Netflix, TikTok, Nike, and many startups             |


Teams hire for **React + Next.js** because most production React jobs involve a meta-framework, and Next.js is the most common choice.

# 5. Full-Stack React Framework Overview

Next.js is "full-stack" because the same project handles:

```text
┌─────────────────────────────────────────────────────┐
│                    Next.js App                       │
├──────────────┬──────────────┬───────────────────────┤
│   Frontend   │   Backend    │   Infrastructure      │
│  (React UI)  │ (Server)     │   (Build / Deploy)    │
├──────────────┼──────────────┼───────────────────────┤
│ page.tsx     │ Server       │ npm run build         │
│ layout.tsx   │ Components   │ Static generation     │
│ components/  │ Server       │ Edge middleware       │
│              │ Actions      │ Caching layers        │
│              │ route.ts     │ Vercel / Docker       │
└──────────────┴──────────────┴───────────────────────┘
```

A single `page.tsx` can fetch from a database, render HTML on the server, and stream it to the browser — no separate API server required for many use cases.

# 6. Rendering Strategies Explained Simply

How HTML reaches the browser determines speed, SEO, and freshness of data.


| Strategy                        | Abbreviation | When HTML is built               | Best for                                  |
| ------------------------------- | ------------ | -------------------------------- | ----------------------------------------- |
| Client-Side Rendering           | CSR          | In the browser after JS loads    | Dashboards, authenticated apps            |
| Server-Side Rendering           | SSR          | On each request on the server    | Personalized, always-fresh pages          |
| Static Site Generation          | SSG          | At build time                    | Marketing pages, docs, blogs              |
| Incremental Static Regeneration | ISR          | At build + revalidate on a timer | Product catalogs that update periodically |


### CSR — Client-Side Rendering (plain React default)

```text
Browser → empty HTML → download JS → React renders → content visible
```

This is how react-project-study works. Search engines see little content until JavaScript runs.

### SSR — Server-Side Rendering

```text
Browser → request page → server runs React → full HTML sent → hydrate
```

Every visit gets fresh HTML. Used for dynamic, user-specific content.

### SSG — Static Site Generation

```text
Build time → HTML files generated → CDN serves pre-built pages
```

Fastest delivery. Content is fixed until the next build.

### ISR — Incremental Static Regeneration

```text
Build time → static page → after N seconds, next request triggers rebuild
```

Combines static speed with periodic updates. Configured via `revalidate` in fetch options (Topic 24).

### How Next.js picks a strategy

In App Router, rendering is chosen **per route** and **per fetch**:

```tsx
// Static by default (SSG) unless you opt into dynamic
export default async function ProductsPage() {
  const products = await fetch('https://api.example.com/products')
  const data = await products.json()
  return <ProductList items={data} />
}
```

```tsx
// Force dynamic (SSR) — no caching
const res = await fetch(url, { cache: 'no-store' })
```

Students do not need to memorize every caching rule now. The key idea: **Next.js lets you choose per page**, not one strategy for the whole app.

# 7. App Router vs Pages Router

Next.js has two routing systems. This course focuses on **App Router only**.


| Aspect            | Pages Router (legacy)                        | App Router (current)       |
| ----------------- | -------------------------------------------- | -------------------------- |
| Directory         | `pages/`                                     | `app/`                     |
| Layouts           | `_app.js` workaround                         | Native `layout.tsx`        |
| Data fetching     | `getServerSideProps`, `getStaticProps`       | `async` Server Components  |
| Server Components | Not supported                                | Default component type     |
| Loading UI        | Manual                                       | `loading.tsx` convention   |
| Error UI          | Manual                                       | `error.tsx` convention     |
| Status            | Maintained, not recommended for new projects | Default in create-next-app |


### Pages Router example (do not use for new projects)

```tsx
// pages/products/index.tsx — old pattern
export async function getServerSideProps() {
  const res = await fetch('https://api.example.com/products')
  const products = await res.json()
  return { props: { products } }
}
```

### App Router equivalent

```tsx
// app/products/page.tsx — current pattern
export default async function ProductsPage() {
  const res = await fetch('https://api.example.com/products')
  const products = await res.json()
  return <ProductList items={products} />
}
```

Cleaner, colocated, and supports React Server Components natively.

# 8. Next.js Architecture: File-Based Routing

URLs map directly to folders and files inside `app/`:

```text
app/
├── page.tsx              →  /
├── about/
│   └── page.tsx          →  /about
├── products/
│   ├── page.tsx          →  /products
│   └── [id]/
│       └── page.tsx      →  /products/42
└── layout.tsx            →  wraps all routes above
```


| File            | Purpose                                            |
| --------------- | -------------------------------------------------- |
| `page.tsx`      | UI for a route — makes the URL publicly accessible |
| `layout.tsx`    | Shared UI wrapper that persists across navigation  |
| `loading.tsx`   | Loading UI while page content loads                |
| `error.tsx`     | Error boundary for that route segment              |
| `not-found.tsx` | Custom 404 for that segment                        |
| `route.ts`      | API endpoint (GET, POST, etc.)                     |


Compare with react-project-study where routes are declared in JSX:

```tsx
// react-project-study/src/App.tsx — manual route definitions
<Route path="/" element={<Home />} />
<Route path="/products" element={<Product />} />
<Route path="/data-fetch" element={<DataFetch />} />
```

Next.js removes the central route config file — the **folder structure is the route map**.

# 9. React Server Components (RSC) Overview

Introduced with App Router, **Server Components** run only on the server:


| Server Component                | Client Component (`'use client'`)       |
| ------------------------------- | --------------------------------------- |
| Default in App Router           | Opt-in with directive at top of file    |
| Can be `async`                  | Cannot be `async`                       |
| No `useState`, `useEffect`      | Full hooks support                      |
| Direct DB / file access         | Browser APIs (`window`, `localStorage`) |
| Zero JS sent for that component | JS bundled and sent to browser          |


```tsx
// app/products/page.tsx — Server Component (default)
export default async function ProductsPage() {
  const products = await getProductsFromDB()
  return (
    <ul>
      {products.map((p) => (
        <li key={p.id}>{p.name}</li>
      ))}
    </ul>
  )
}
```

```tsx
// components/Counter.tsx — Client Component
'use client'

import { useState } from 'react'

export function Counter() {
  const [count, setCount] = useState(0)
  return <button onClick={() => setCount(count + 1)}>{count}</button>
}
```

Server Components are the biggest architectural shift from plain React. Topics 6, 12, and 13 go deeper.

# 10. When to Choose Next.js vs Other Frameworks


| Framework              | Strengths                               | Trade-offs                              |
| ---------------------- | --------------------------------------- | --------------------------------------- |
| **Next.js**            | Full-stack React, Vercel ecosystem, RSC | Opinionated; learning curve for caching |
| **Remix**              | Web standards, forms, nested routes     | Smaller community than Next.js          |
| **Gatsby**             | Static sites, GraphQL data layer        | Less suited for highly dynamic apps     |
| **Plain React + Vite** | Simple, flexible, great for learning    | You build everything yourself           |
| **Astro**              | Content sites, multi-framework          | Not React-first for interactive apps    |


**Choose Next.js when:**

- The team already knows React
- You need SEO and performance out of the box
- You want server rendering without a separate backend for many features

**Stay on Vite/React when:**

- Building a SPA with no SEO requirements
- Learning React fundamentals first
- The app is a small widget or internal tool

# 11. Mental Model: Request Lifecycle

```text
1. User visits /products
2. Next.js matches app/products/page.tsx
3. Server runs layout.tsx → page.tsx (Server Components)
4. Data fetched on server (DB, API, file)
5. HTML streamed to browser
6. Client Components hydrate (buttons, forms, hooks)
7. Client-side navigation via <Link> for subsequent pages
```

This is fundamentally different from react-project-study where step 3–4 happen entirely in the browser after JS downloads.

---

# 12. Best Practices for Getting Started

✔ Complete React fundamentals (components, props, state, hooks) before deep Server Component work

✔ Use **App Router** for all new projects — not Pages Router

✔ Default to **Server Components**; add `'use client'` only when you need interactivity

✔ Think in **routes** (folders) not a central router config

✔ Understand rendering strategies at a high level before optimizing caching

✔ Keep react-project-study open for React pattern comparisons

✔ Read the official Next.js docs alongside this course: [https://nextjs.org/docs](https://nextjs.org/docs)
