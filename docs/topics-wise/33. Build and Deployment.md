# Build and Deployment - Teaching Notes

## Topic 33: From Local Dev to Production Hosting

---

# 1. Why This Topic Matters

A working dev server (`npm run dev`) is not production. Production requires:

* Optimized bundles and static generation
* Environment variables configured on the host
* HTTPS, CDN, and serverless or container runtime
* CI/CD so every merge can ship safely

This lesson covers `npm run build`, reading build output, preview mode, deploying to Vercel and alternatives, and CI/CD basics.

---

# 2. Production Build with npm run build

```bash
npm run build
```

Next.js performs:

| Step | Result |
| ---- | ------ |
| TypeScript compile | Fails build on type errors |
| Server/Client split | Separate bundles per route |
| Static generation | Pre-renders static pages where possible |
| Tree shaking | Removes unused code |
| Image/font optimization | Prepares assets |

Successful output ends with route table:

```text
Route (app)                              Size     First Load JS
┌ ○ /                                    5 kB         90 kB
├ ○ /about                               2 kB         87 kB
├ ƒ /dashboard                           8 kB         95 kB
├ ƒ /api/posts                           0 B          0 B
└ ○ /blog/[slug]                         6 kB         91 kB

○  (Static)   prerendered as static content
ƒ  (Dynamic)  server-rendered on demand
```

| Symbol | Meaning |
| ------ | ------- |
| ○ Static | HTML generated at build time |
| ƒ Dynamic | Rendered per request |
| ℇ Edge | Edge runtime route |

Fix all build errors before deploying — production will fail the same way.

# 3. Understanding Build Output

### .next folder

```text
.next/
├── static/          # Client JS/CSS chunks
├── server/          # Server bundles and RSC payloads
├── cache/           # Build cache (CI may cache this)
└── BUILD_ID         # Unique build identifier
```

Do not commit `.next/` to git — add to `.gitignore`.

### Route types and deployment impact

| Route type | Hosting behavior |
| ---------- | ---------------- |
| Static (○) | Served from CDN — fastest |
| Dynamic (ƒ) | Needs Node/Edge server per request |
| ISR | Static + revalidation timer |

Use static generation where data changes infrequently. Use dynamic for personalized or real-time pages.

# 4. Preview with npm start

After build, run production server locally:

```bash
npm run build
npm start
```

Open `http://localhost:3000` — behavior matches production (no hot reload, optimized bundles).

| Command | Environment |
| ------- | ----------- |
| `npm run dev` | Development — fast refresh, verbose errors |
| `npm start` | Production — optimized, strict |

Always smoke-test with `npm start` before first deploy.

# 5. Deploy on Vercel

Vercel created Next.js — deployment is streamlined.

### Steps

1. Push code to GitHub, GitLab, or Bitbucket
2. Import project at [vercel.com](https://vercel.com)
3. Vercel auto-detects Next.js
4. Add environment variables in project settings
5. Deploy — each push to main creates production deploy; PRs get preview URLs

```text
Git Push → Vercel Build → CDN + Serverless Functions → Live URL
```

### vercel.json (optional overrides)

```json
{
  "framework": "nextjs",
  "regions": ["bom1"],
  "headers": [
    {
      "source": "/(.*)",
      "headers": [
        { "key": "X-Frame-Options", "value": "DENY" }
      ]
    }
  ]
}
```

### Environment variables on Vercel

| Environment | Use |
| ----------- | --- |
| Production | Live site secrets |
| Preview | PR preview deployments |
| Development | Local `vercel dev` (optional) |

Set `DATABASE_URL`, `AUTH_SECRET`, etc. in Vercel dashboard — never in git.

# 6. Deploy on Netlify, AWS, or Docker

### Netlify

Supports Next.js with `@netlify/plugin-nextjs`. Connect repo, set build command `npm run build`, publish directory handled by plugin.

### AWS

Options:

| Service | Pattern |
| ------- | ------- |
| Amplify | Git-connected hosting similar to Vercel |
| ECS/Fargate | Docker container running `npm start` |
| Lambda + OpenNext | Serverless Next.js adapter |

### Docker deployment

```dockerfile
# Dockerfile
FROM node:20-alpine AS base

FROM base AS deps
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci

FROM base AS builder
WORKDIR /app
COPY --from=deps /app/node_modules ./node_modules
COPY . .
RUN npm run build

FROM base AS runner
WORKDIR /app
ENV NODE_ENV=production
COPY --from=builder /app/public ./public
COPY --from=builder /app/.next/standalone ./
COPY --from=builder /app/.next/static ./.next/static

EXPOSE 3000
CMD ["node", "server.js"]
```

Enable standalone output:

```typescript
// next.config.ts
const nextConfig = {
  output: 'standalone',
}
export default nextConfig
```

Run container:

```bash
docker build -t my-next-app .
docker run -p 3000:3000 --env-file .env.production my-next-app
```

# 7. Environment Variables on Hosting Platforms

Same rules as Topic 19 — different UI per platform.

| Variable | Platform setting |
| -------- | ---------------- |
| `DATABASE_URL` | Server-only, all environments |
| `AUTH_SECRET` | Server-only, unique per environment |
| `NEXT_PUBLIC_APP_URL` | Public, matches deployed URL |

### Checklist before deploy

✔ All required env vars set on host

✔ `AUTH_URL` / `NEXTAUTH_URL` matches production domain

✔ OAuth redirect URIs updated in Google/GitHub console

✔ Database accessible from host region

✔ `.env.local` not in repository

# 8. CI/CD Basics for Next.js

CI/CD = Continuous Integration + Continuous Deployment.

```text
Developer pushes code
       │
       ▼
CI runs: lint → test → build
       │
       ▼
CD deploys to staging/production if green
```

### GitHub Actions example

```yaml
name: CI
on:
  push:
    branches: [main]
  pull_request:

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: npm
      - run: npm ci
      - run: npm run lint
      - run: npm run test:run
      - run: npm run build
        env:
          AUTH_SECRET: ci-test-secret-min-32-chars-long
          DATABASE_URL: postgresql://ci:ci@localhost/test
```

| Stage | Blocks deploy if fails |
| ----- | ---------------------- |
| Lint | Code style and obvious bugs |
| Test | Regressions in logic |
| Build | TypeScript and Next.js errors |

Vercel can deploy automatically after CI passes (GitHub integration → Deployment Protection).

# 9. Domain, HTTPS, and Preview Deployments

| Topic | Practice |
| ----- | -------- |
| Custom domain | Add in Vercel/Netlify DNS settings |
| HTTPS | Automatic on Vercel/Netlify |
| Preview URLs | Every PR gets unique URL for QA |
| Production branch | Usually `main` — protect with required checks |

Test auth flows on preview URLs — update OAuth allowed redirect URIs to include preview pattern or use staging provider apps.

# 10. Common Build and Deploy Issues

| Issue | Fix |
| ----- | --- |
| `AUTH_SECRET` missing | Set in host env vars |
| Type error only in CI | Run `npm run build` locally |
| `Module not found: server-only` | Import server module in client component |
| Image domain not configured | Add `remotePatterns` in `next.config.ts` |
| Database connection timeout | Check IP allowlist and region |
| Static page shows stale data | Review caching and revalidation (Topic 24) |

---

# 11. Best Practices

✔ Run `npm run build` and `npm start` locally before first deploy

✔ Use preview deployments for QA on every PR

✔ Separate env vars per environment (dev, staging, prod)

✔ Cache `node_modules` and `.next/cache` in CI for speed

✔ Pin Node version in `.nvmrc` or `engines` in `package.json`

✔ Document deploy steps in project README

✔ Enable deployment protection — require CI pass before production

# 13. Summary

| Step | Command / action |
| ---- | ---------------- |
| Build | `npm run build` — optimize and type-check |
| Preview | `npm start` — local production server |
| Vercel | Git push → auto deploy + previews |
| Docker/AWS | Standalone output + container or adapter |
| Env vars | Set on host — never commit secrets |
| CI/CD | Lint, test, build on every PR |

Shipping is part of the job. A green local build plus CI gives confidence every deploy will succeed.
