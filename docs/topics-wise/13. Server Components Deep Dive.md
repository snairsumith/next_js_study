# Server Components Deep Dive - Teaching Notes

## Topic 13: Understanding React Server Components in Next.js

---

# 1. Why This Topic Matters

Server Components (RSC) are the default in the Next.js App Router. Every file in `app/` is a Server Component unless you add `'use client'`.

Understanding Server Components deeply is what separates a beginner Next.js developer from someone who can build fast, secure, production apps. This topic explains **where code runs**, **what you can and cannot do**, and **how to split server and client at the right boundary**.

---

# 2. Server vs Client: The Core Split

| Aspect | Server Component | Client Component |
| ------ | ---------------- | ---------------- |
| Directive | None (default) | `'use client'` at top of file |
| Runs on | Server only | Server (SSR) + browser (hydration) |
| Hooks (`useState`, etc.) | ❌ Not allowed | ✅ Allowed |
| Browser APIs (`window`, `localStorage`) | ❌ Not allowed | ✅ Allowed |
| Event handlers (`onClick`) | ❌ Not allowed | ✅ Allowed |
| Direct DB / file access | ✅ Yes | ❌ No |
| `async` component | ✅ Yes | ❌ No (use effects instead) |
| Sent to browser as JS | ❌ No — only rendered HTML | ✅ Yes — full component JS |

```text
Request for /products
        │
        ▼
  Server Component (page.tsx)
        │
        ├──► Query database directly
        ├──► Read files from disk
        ├──► Call internal APIs (no HTTP round-trip)
        │
        ▼
  HTML stream sent to browser
        │
        ▼
  Client Components hydrate for interactivity
```

# 3. What Server Components Can Do

### Direct data access

Server Components can talk to your database, ORM, or file system without exposing credentials to the browser.

```tsx
// app/products/page.tsx — Server Component (default)
import { db } from '@/lib/db'

export default async function ProductsPage() {
  // Runs on the server — connection string never reaches the client
  const products = await db.product.findMany({
    orderBy: { createdAt: 'desc' },
  })

  return (
    <main>
      <h1>Products</h1>
      <ul>
        {products.map((product) => (
          <li key={product.id}>{product.name}</li>
        ))}
      </ul>
    </main>
  )
}
```

### Async components

Server Components can be `async` and use `await` directly in the component body.

```tsx
interface User {
  id: number
  name: string
  email: string
}

async function getUsers(): Promise<User[]> {
  const res = await fetch('https://jsonplaceholder.typicode.com/users', {
    cache: 'no-store',
  })
  if (!res.ok) throw new Error('Failed to fetch users')
  return res.json()
}

export default async function UsersPage() {
  const users = await getUsers()

  return (
    <section>
      <h1>Users ({users.length})</h1>
      {users.map((user) => (
        <article key={user.id}>
          <h2>{user.name}</h2>
          <p>{user.email}</p>
        </article>
      ))}
    </section>
  )
}
```

| Capability | Example use |
| ---------- | ----------- |
| Database queries | Product lists, user profiles |
| File system reads | MDX content, config files |
| Server-only secrets | API keys, tokens |
| Heavy computation | Sorting large datasets server-side |

# 4. What Server Components Cannot Do

These restrictions exist because Server Components **do not ship JavaScript to the browser**.

```tsx
// ❌ WRONG — hooks in a Server Component
export default function Counter() {
  const [count, setCount] = useState(0) // Error!
  return <button onClick={() => setCount(count + 1)}>{count}</button>
}
```

```tsx
// ❌ WRONG — browser API in Server Component
export default function ThemeDisplay() {
  const theme = localStorage.getItem('theme') // Error — no window/localStorage on server
  return <p>Theme: {theme}</p>
}
```

```tsx
// ❌ WRONG — event handler in Server Component
export default function SaveButton() {
  return <button onClick={() => alert('Saved')}>Save</button> // Error!
}
```

| Forbidden in Server Components | Why |
| ------------------------------ | --- |
| `useState`, `useEffect`, all hooks | No client lifecycle on server |
| `onClick`, `onChange`, etc. | Needs browser event system |
| `window`, `document`, `localStorage` | Browser-only globals |
| `useSearchParams()` | Client hook — use `searchParams` prop instead |

**Fix:** Move interactive parts into a Client Component and import it from the Server Component.

# 5. Composing Server and Client Components

The recommended pattern: **Server Component wraps Client Component**.

```tsx
// components/add-to-cart-button.tsx
'use client'

interface AddToCartButtonProps {
  productId: string
  productName: string
}

export function AddToCartButton({ productId, productName }: AddToCartButtonProps) {
  function handleClick() {
    console.log(`Added ${productName} (${productId}) to cart`)
  }

  return (
    <button
      type="button"
      onClick={handleClick}
      aria-label={`Add ${productName} to cart`}
    >
      Add to Cart
    </button>
  )
}
```

```tsx
// app/products/[id]/page.tsx — Server Component
import { AddToCartButton } from '@/components/add-to-cart-button'
import { db } from '@/lib/db'

interface ProductPageProps {
  params: Promise<{ id: string }>
}

export default async function ProductPage({ params }: ProductPageProps) {
  const { id } = await params
  const product = await db.product.findUnique({ where: { id } })

  if (!product) return <p>Product not found</p>

  return (
    <article>
      <h1>{product.name}</h1>
      <p>{product.description}</p>
      <p>Rs. {product.price}</p>
      {/* Pass serializable props from server to client */}
      <AddToCartButton productId={product.id} productName={product.name} />
    </article>
  )
}
```

### Composition rules

| Rule | Detail |
| ---- | ------ |
| Server → Client | ✅ Import Client Component into Server Component |
| Client → Server | ❌ Cannot import Server Component into Client Component |
| Client → Client | ✅ Allowed |
| Server → Server | ✅ Allowed |
| Passing props | Must be **serializable** (JSON-safe) |

---

# 6. Serializable Props Rule

Data passed from Server to Client Components must survive JSON serialization.

| Allowed | Not allowed |
| ------- | ----------- |
| Strings, numbers, booleans | Functions |
| Arrays and plain objects | Class instances |
| `null`, `undefined` | `Date` objects (pass ISO string instead) |
| JSX as `children` | Symbols, Maps, Sets |

```tsx
// ✅ Good — serializable props
<ClientChart data={[{ month: 'Jan', sales: 1200 }]} title="Sales" />

// ❌ Bad — passing a function
<ClientChart onSelect={(id) => console.log(id)} />
```

If the client needs a callback, define it **inside** the Client Component.

# 7. Streaming and Suspense Boundaries

Server Components support **streaming**: send HTML to the browser as each part becomes ready.

```tsx
// app/dashboard/page.tsx
import { Suspense } from 'react'
import { RevenueChart } from '@/components/revenue-chart'
import { RecentOrders } from '@/components/recent-orders'
import { DashboardSkeleton } from '@/components/dashboard-skeleton'

export default function DashboardPage() {
  return (
    <main>
      <h1>Dashboard</h1>

      {/* Fast section renders immediately */}
      <section>
        <h2>Overview</h2>
        <p>Welcome back!</p>
      </section>

      {/* Slow sections stream in when ready */}
      <Suspense fallback={<DashboardSkeleton />}>
        <RevenueChart />
      </Suspense>

      <Suspense fallback={<p>Loading orders...</p>}>
        <RecentOrders />
      </Suspense>
    </main>
  )
}
```

```tsx
// components/revenue-chart.tsx — async Server Component
async function fetchRevenue() {
  await new Promise((resolve) => setTimeout(resolve, 2000)) // simulate slow API
  return [{ month: 'Jan', amount: 45000 }, { month: 'Feb', amount: 52000 }]
}

export async function RevenueChart() {
  const data = await fetchRevenue()
  return (
    <section>
      <h2>Revenue</h2>
      <ul>
        {data.map((row) => (
          <li key={row.month}>{row.month}: Rs. {row.amount}</li>
        ))}
      </ul>
    </section>
  )
}
```

| Concept | Benefit |
| ------- | ------- |
| Streaming | User sees page shell faster |
| Suspense | Per-section loading UI |
| Parallel async components | Independent sections don't block each other |

# 8. Performance Benefits

| Benefit | Explanation |
| ------- | ------------- |
| Zero client JS for static UI | Server Components don't add to bundle size |
| No waterfall in browser | Data fetched on server before HTML is sent |
| Closer to data source | DB queries run near your server, not user's device |
| Automatic code splitting | Client Components are split into separate chunks |

### Bundle size comparison (conceptual)

```text
Plain React SPA:
  └── Entire page logic + data fetching → shipped to browser

Next.js Server Component page:
  └── Server: fetch + render HTML
  └── Client: only interactive widgets (buttons, forms, charts)
```

# 9. Choosing the Right Boundary

Ask these questions for every component:

| Question | If yes → |
| -------- | -------- |
| Does it need `useState` or `useEffect`? | Client Component |
| Does it handle clicks or form input? | Client Component |
| Does it use browser APIs? | Client Component |
| Does it only display fetched data? | Server Component |
| Does it access the database? | Server Component |

### Leaf pattern — push `'use client'` down

```text
page.tsx (Server)
  ├── ProductHeader (Server) — static title, price
  ├── ProductGallery (Client) — image carousel with swipe
  └── ProductReviews (Server) — fetched reviews list
        └── ReviewForm (Client) — interactive form
```

Keep Client Components as **small leaves** at the bottom of the tree.

# 10. Common Mistakes

| Mistake | Fix |
| ------- | --- |
| Adding `'use client'` to entire page | Only mark interactive subcomponents |
| Using `useEffect` to fetch on client | Use `async` Server Component instead |
| Passing functions as props to client | Define handlers inside Client Component |
| Importing server-only code in client file | Keep DB calls in Server Components |
| Making everything a Client Component | Defeats the purpose of RSC |

```tsx
// ❌ Entire page is client — loses server benefits
'use client'
export default function ProductsPage() {
  const [products, setProducts] = useState([])
  useEffect(() => { fetch('/api/products').then(/* ... */) }, [])
  // ...
}

// ✅ Server page with small client island
export default async function ProductsPage() {
  const products = await db.product.findMany()
  return <ProductList products={products} /> // Client only if list needs interactivity
}
```

# 11. Best Practices

✔ Default to Server Components — add `'use client'` only when needed

✔ Keep Client Components small and at the leaves of the component tree

✔ Fetch data as close to where it's needed as possible (colocation)

✔ Use Suspense to stream slow sections without blocking the whole page

✔ Pass only serializable props across the server/client boundary

✔ Never import server-only modules (`db`, `fs`) into Client Components

✔ Use TypeScript interfaces for props on both server and client components

# 13. Summary

| Concept | Key takeaway |
| ------- | ------------ |
| Default | All App Router files are Server Components |
| `'use client'` | Marks interactive components that need hooks and events |
| Async | Server Components can `await` data directly |
| Composition | Server wraps Client; never the reverse |
| Props | Must be JSON-serializable across the boundary |
| Streaming | Suspense sends HTML progressively |
| Performance | Less JS to browser, data fetched on server |
