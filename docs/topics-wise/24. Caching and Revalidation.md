# Caching and Revalidation - Teaching Notes

## Topic 24: Data Cache, Route Cache, and On-Demand Updates

---

# 1. Why This Topic Matters

Caching makes your app fast. Revalidation keeps cached data fresh. Without understanding Next.js caching layers, you'll either serve stale data to users or disable caching entirely and lose performance.

Next.js 14+ has a multi-layer cache system: Request Memoization, Data Cache, and Full Route Cache. This topic explains each layer, how to configure them, and how to invalidate cached data when content changes.

---

# 2. The Four Caching Layers

```text
Request comes in
       │
       ▼
┌─────────────────────────┐
│  1. Request Memoization │  Same fetch in one render — deduplicated
└────────────┬────────────┘
             ▼
┌─────────────────────────┐
│  2. Data Cache          │  fetch() results stored across requests
└────────────┬────────────┘
             ▼
┌─────────────────────────┐
│  3. Full Route Cache    │  Rendered HTML + RSC payload stored
└────────────┬────────────┘
             ▼
┌─────────────────────────┐
│  4. Router Cache        │  Client-side — cached RSC payload per route
└─────────────────────────┘
```

| Layer | Scope | Duration | Controlled by |
| ----- | ----- | -------- | ------------- |
| Request Memoization | Single server render | One request | Automatic |
| Data Cache | All server requests | Until revalidated | `fetch` options, `revalidateTag` |
| Full Route Cache | Prerendered pages | Until revalidated | `revalidate`, `revalidatePath` |
| Router Cache | Client navigation | Session / configurable | `router.refresh()` |

# 3. Request Memoization

Automatic deduplication of identical `fetch` calls during a single server render.

```tsx
// lib/user.ts
export async function getUser(id: string) {
  const res = await fetch(`https://api.example.com/users/${id}`)
  return res.json()
}
```

```tsx
// app/profile/page.tsx
import { getUser } from '@/lib/user'

async function ProfileHeader() {
  const user = await getUser('123') // Network request #1
  return <h1>{user.name}</h1>
}

async function ProfileStats() {
  const user = await getUser('123') // Deduplicated — no second network call
  return <p>Posts: {user.postCount}</p>
}

export default async function ProfilePage() {
  return (
    <>
      <ProfileHeader />
      <ProfileStats />
    </>
  )
}
```

| Feature | Detail |
| ------- | ------ |
| Scope | Single server render pass |
| Requirement | Same URL + same options |
| Benefit | Avoid duplicate API calls in one page |
| Not configurable | Always on for `fetch` |

**Note:** Request memoization does NOT apply to direct database calls — only `fetch`.

# 4. Data Cache — fetch Options

Next.js extends `fetch` with caching controls that populate the Data Cache.

```tsx
// Cached indefinitely (default for fetch in Server Components)
const res = await fetch('https://api.example.com/categories')

// Time-based revalidation — ISR for data (revalidate every 60 seconds)
const res = await fetch('https://api.example.com/products', {
  next: { revalidate: 60 },
})

// Never cache — always fresh (dynamic)
const res = await fetch('https://api.example.com/cart', {
  cache: 'no-store',
})

// Tag-based cache — for on-demand revalidation
const res = await fetch('https://api.example.com/posts', {
  next: { tags: ['posts'] },
})
```

| Option | Cache behavior | Use case |
| ------ | -------------- | -------- |
| Default (no options) | Cached forever | Static reference data |
| `{ next: { revalidate: N } }` | Stale after N seconds, then re-fetched | Product catalog, news |
| `{ cache: 'no-store' }` | Never cached | User cart, live data |
| `{ next: { tags: ['x'] } }` | Cached until tag revalidated | CMS content, admin updates |

# 5. Full Route Cache and Static Generation

When a page can be fully rendered at build time, Next.js stores the HTML in the Full Route Cache.

```tsx
// app/products/page.tsx
// This page is statically generated at build time (default with cached fetch)

export default async function ProductsPage() {
  const products = await fetch('https://api.example.com/products').then((r) => r.json())

  return (
    <ul>
      {products.map((p: { id: string; name: string }) => (
        <li key={p.id}>{p.name}</li>
      ))}
    </ul>
  )
}
```

```tsx
// Force dynamic — skip Full Route Cache entirely
export const dynamic = 'force-dynamic'

// Or set page-level revalidation interval
export const revalidate = 3600 // revalidate page every hour (ISR)
```

| Export | Effect |
| ------ | ------ |
| (default with cached fetch) | Static at build time |
| `export const revalidate = 60` | ISR — regenerate every 60 seconds |
| `export const dynamic = 'force-dynamic'` | Always server-rendered per request |
| `fetch(..., { cache: 'no-store' })` | Opts that fetch out of static generation |

# 6. Time-Based Revalidation (ISR)

Incremental Static Regeneration — serve cached page, regenerate in background after interval.

```tsx
// app/blog/page.tsx
export const revalidate = 300 // page regenerates every 5 minutes

async function getPosts() {
  const res = await fetch('https://api.example.com/posts', {
    next: { revalidate: 300, tags: ['posts'] },
  })
  return res.json()
}

export default async function BlogPage() {
  const posts = await getPosts()

  return (
    <main>
      <h1>Blog</h1>
      <ul>
        {posts.map((post: { slug: string; title: string }) => (
          <li key={post.slug}>
            <a href={`/blog/${post.slug}`}>{post.title}</a>
          </li>
        ))}
      </ul>
    </main>
  )
}
```

```text
Timeline for ISR (revalidate: 60):

0s    ──► First request: render + cache page
30s   ──► Request: serve cached page (fast)
60s   ──► Request: serve stale page + trigger background regeneration
61s   ──► Next request: serve freshly regenerated page
```

| ISR benefit | Detail |
| ----------- | ------ |
| Fast response | Always serve cached version first |
| Fresh content | Background regeneration after interval |
| No rebuild needed | Content updates without full redeploy |

# 7. On-Demand Revalidation with revalidatePath

Immediately invalidate cached pages after a mutation — used in Server Actions.

```tsx
// app/actions/products.ts
'use server'

import { revalidatePath } from 'next/cache'

export async function createProduct(formData: FormData) {
  const name = formData.get('name') as string
  await db.product.create({ data: { name } })

  // Invalidate the products list page
  revalidatePath('/products')

  return { success: true }
}

export async function updateProduct(id: string, formData: FormData) {
  const name = formData.get('name') as string
  await db.product.update({ where: { id }, data: { name } })

  // Invalidate list page AND detail page
  revalidatePath('/products')
  revalidatePath(`/products/${id}`)

  return { success: true }
}
```

```tsx
// Revalidate layout and all nested pages
revalidatePath('/blog', 'layout')
```

| Call | Effect |
| ---- | ------ |
| `revalidatePath('/products')` | Next visit to `/products` re-renders |
| `revalidatePath('/products/123')` | Re-renders specific product page |
| `revalidatePath('/blog', 'layout')` | Re-renders layout + all child pages |

# 8. On-Demand Revalidation with revalidateTag

Tag-based revalidation — invalidate all fetches sharing a tag.

```tsx
// lib/posts.ts
export async function getPosts() {
  const res = await fetch('https://api.example.com/posts', {
    next: { tags: ['posts'] },
  })
  return res.json()
}

export async function getPost(slug: string) {
  const res = await fetch(`https://api.example.com/posts/${slug}`, {
    next: { tags: ['posts', `post-${slug}`] },
  })
  return res.json()
}
```

```tsx
// app/actions/posts.ts
'use server'

import { revalidateTag } from 'next/cache'

export async function createPost(formData: FormData) {
  const title = formData.get('title') as string
  await db.post.create({ data: { title } })

  // Invalidate ALL fetches tagged 'posts'
  revalidateTag('posts')

  return { success: true }
}

export async function updatePost(slug: string, formData: FormData) {
  const title = formData.get('title') as string
  await db.post.update({ where: { slug }, data: { title } })

  // Invalidate list and specific post
  revalidateTag('posts')
  revalidateTag(`post-${slug}`)

  return { success: true }
}
```

| Method | Granularity |
| ------ | ----------- |
| `revalidatePath('/blog')` | Specific route path |
| `revalidateTag('posts')` | All fetches with that tag — across routes |

**Prefer tags** when the same data appears on multiple pages.

# 9. unstable_cache for Non-fetch Data

Cache database queries and other non-fetch async functions.

```tsx
// lib/products.ts
import { unstable_cache } from 'next/cache'
import { db } from '@/lib/db'

export const getFeaturedProducts = unstable_cache(
  async () => {
    return db.product.findMany({
      where: { isFeatured: true },
      take: 6,
      orderBy: { createdAt: 'desc' },
    })
  },
  ['featured-products'],       // cache key
  {
    revalidate: 300,             // ISR — 5 minutes
    tags: ['products', 'featured'],
  }
)
```

```tsx
// app/page.tsx
import { getFeaturedProducts } from '@/lib/products'

export default async function HomePage() {
  const products = await getFeaturedProducts()
  // ...
}
```

| Tool | Caches |
| ---- | ------ |
| `fetch` with options | HTTP fetch responses |
| `unstable_cache` | Any async function (DB, computation) |
| `React.cache` | Request memoization for any function |

# 10. Opting Out of Caching

Some pages must always be fresh — user dashboards, carts, admin panels.

```tsx
// app/dashboard/page.tsx

// Option 1: Page-level dynamic export
export const dynamic = 'force-dynamic'

// Option 2: No-store fetch
async function getUserDashboard(userId: string) {
  const res = await fetch(`https://api.example.com/dashboard/${userId}`, {
    cache: 'no-store',
  })
  return res.json()
}

// Option 3: Dynamic functions force dynamic rendering
import { cookies } from 'next/headers'

export default async function DashboardPage() {
  const cookieStore = await cookies()
  const session = cookieStore.get('session')
  // Using cookies() automatically opts page into dynamic rendering
}
```

| Signal | Effect |
| ------ | ------ |
| `export const dynamic = 'force-dynamic'` | Page always server-rendered |
| `fetch(..., { cache: 'no-store' })` | That fetch always fresh |
| `cookies()` or `headers()` | Page becomes dynamic automatically |
| `export const revalidate = 0` | Same as force-dynamic |

# 11. Router Cache (Client-Side)

The client-side Router Cache stores RSC payloads during navigation within a session.

```tsx
'use client'

import { useRouter } from 'next/navigation'

export function RefreshButton() {
  const router = useRouter()

  function handleRefresh() {
    // Clears Router Cache for current route and re-fetches from server
    router.refresh()
  }

  return (
    <button type="button" onClick={handleRefresh}>
      Refresh data
    </button>
  )
}
```

| Action | Router Cache effect |
| ------ | ------------------- |
| Navigate with `Link` | May serve cached RSC payload |
| `router.refresh()` | Re-fetches current route from server |
| `revalidatePath` in Server Action | Invalidates server cache; client refreshes on next navigation |
| Full page reload | Clears all client caches |

# 12. Caching Strategy Guide

| Content type | Strategy | Config |
| ------------ | -------- | ------ |
| Static pages (about, contact) | Static generation | Default fetch cache |
| Product catalog | ISR | `revalidate: 3600` |
| Blog posts | ISR + tags | `tags: ['posts']`, revalidate on publish |
| User dashboard | Dynamic | `dynamic = 'force-dynamic'` |
| Shopping cart | No cache | `cache: 'no-store'` |
| Admin after edit | On-demand | `revalidateTag('products')` in Server Action |
| DB queries | `unstable_cache` | With tags and revalidate interval |

```text
Decision tree:

Is data user-specific?
  YES → cache: 'no-store' or force-dynamic
  NO ↓

Does data change frequently?
  YES → revalidate: 60 (ISR) + tags
  NO ↓

Is data completely static?
  YES → default cache (static at build)
```

# 13. Best Practices

✔ Set explicit cache options — don't rely on defaults without understanding them

✔ Use cache tags for data shared across multiple pages

✔ Call `revalidateTag` or `revalidatePath` in Server Actions after mutations

✔ Use `unstable_cache` for database queries, not just `fetch`

✔ Use `cache: 'no-store'` for personalized or sensitive data

✔ Document your caching strategy per route in code comments

✔ Test caching behavior with `npm run build && npm start` — dev mode behaves differently

# 15. Summary

| Layer / API | Purpose |
| ----------- | ------- |
| Request Memoization | Dedupe identical fetches in one render |
| Data Cache | Store fetch results across requests |
| Full Route Cache | Store rendered HTML at build time |
| `revalidate: N` | Time-based ISR — refresh every N seconds |
| `cache: 'no-store'` | Opt out — always fresh |
| `next: { tags: [...] }` | Tag fetches for on-demand invalidation |
| `revalidatePath()` | Invalidate cached route after mutation |
| `revalidateTag()` | Invalidate all fetches with a tag |
| `unstable_cache` | Cache non-fetch async functions (DB, etc.) |
| `dynamic = 'force-dynamic'` | Always server-render per request |
