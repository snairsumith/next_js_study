# Image, Font, and Performance Optimization - Teaching Notes

## Topic 32: Core Web Vitals and Next.js Performance Tools

---

# 1. Why This Topic Matters

Slow apps lose users and rank lower in search. Next.js provides built-in optimizations for images, fonts, and JavaScript bundles — but only if you use them correctly.

Performance topics in this lesson:

* `next/image` — responsive, lazy-loaded images
* `next/font` — zero layout shift font loading
* Code splitting — smaller initial bundles
* Re-render control in client components
* Core Web Vitals — LCP, INP, CLS

Goal: fast first paint, smooth interaction, stable layout.

# 2. next/image Component

Plain `<img>` tags hurt performance: no lazy loading, no size optimization, no modern formats.

```tsx
import Image from 'next/image'

export function HeroBanner() {
  return (
    <Image
      src="/hero.jpg"
      alt="Team collaborating on a Next.js project"
      width={1200}
      height={600}
      priority
      className="rounded-lg object-cover"
    />
  )
}
```

| Prop | Purpose |
| ---- | ------- |
| `width` / `height` | Reserve space — prevents CLS |
| `alt` | Accessibility (required) |
| `priority` | Preload above-the-fold images (LCP) |
| `fill` | Fill parent container (needs `position: relative` parent) |
| `sizes` | Tell browser which responsive size to download |
| `placeholder="blur"` | Blur-up while loading (needs `blurDataURL`) |

### Remote images

Configure domains in `next.config.ts`:

```typescript
const nextConfig = {
  images: {
    remotePatterns: [
      {
        protocol: 'https',
        hostname: 'images.unsplash.com',
      },
    ],
  },
}

export default nextConfig
```

```tsx
<Image
  src="https://images.unsplash.com/photo-123"
  alt="Workspace"
  width={800}
  height={500}
  sizes="(max-width: 768px) 100vw, 50vw"
/>
```

### Fill layout pattern

```tsx
<div className="relative h-64 w-full">
  <Image
    src="/card-bg.jpg"
    alt=""
    fill
    className="object-cover"
    sizes="(max-width: 768px) 100vw, 33vw"
  />
</div>
```

# 3. Image Sizing, Lazy Loading, and Formats

Next.js Image Optimization API serves WebP/AVIF when the browser supports them.

| Behavior | Default |
| -------- | ------- |
| Lazy loading | Yes — except when `priority` is set |
| Responsive srcset | Generated from `width`/`sizes` |
| Format | WebP or AVIF automatically |

### sizes attribute explained

```tsx
<Image
  src="/product.jpg"
  alt="Product"
  width={800}
  height={800}
  sizes="(max-width: 640px) 100vw, (max-width: 1024px) 50vw, 400px"
/>
```

Browser picks image width based on viewport — avoids downloading 800px image on mobile when 400px suffices.

### Static import for blur placeholder

```tsx
import heroImage from '@/public/hero.jpg'

<Image
  src={heroImage}
  alt="Hero"
  placeholder="blur"
  priority
/>
```

Static imports give automatic `width`, `height`, and `blurDataURL`.

# 4. next/font for Font Optimization

External font `<link>` tags cause layout shift and extra network requests. `next/font` self-hosts fonts at build time.

### Google Font

```tsx
// app/layout.tsx
import { Inter, Roboto_Mono } from 'next/font/google'

const inter = Inter({
  subsets: ['latin'],
  display: 'swap',
  variable: '--font-inter',
})

const robotoMono = Roboto_Mono({
  subsets: ['latin'],
  variable: '--font-mono',
})

export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <html lang="en" className={`${inter.variable} ${robotoMono.variable}`}>
      <body className="font-sans antialiased">{children}</body>
    </html>
  )
}
```

```css
/* globals.css with Tailwind */
body {
  font-family: var(--font-inter), system-ui, sans-serif;
}
```

### Local font

```tsx
import localFont from 'next/font/local'

const brandFont = localFont({
  src: '../public/fonts/Brand.woff2',
  variable: '--font-brand',
  display: 'swap',
})
```

| Benefit | Explanation |
| ------- | ------------- |
| No layout shift | `size-adjust` and `display: swap` |
| No external request | Font bundled with app |
| Privacy | No Google Fonts CDN call from client |

# 5. Code Splitting with Dynamic Imports

Load heavy components only when needed.

```tsx
import dynamic from 'next/dynamic'

const Chart = dynamic(() => import('@/components/dashboard-chart'), {
  loading: () => <p>Loading chart...</p>,
  ssr: false, // skip server render for browser-only chart library
})

export default function DashboardPage() {
  return (
    <main>
      <h1>Dashboard</h1>
      <Chart />
    </main>
  )
}
```

| Option | Use when |
| ------ | -------- |
| `loading` | Show skeleton while chunk loads |
| `ssr: false` | Component uses `window` or canvas |
| No options | Component can render on server too |

### Route-level splitting

Each page in `app/` is automatically a separate JavaScript chunk. Keep pages focused — avoid importing entire libraries in root layout.

Server Components use `dynamic()` from `next/dynamic`. Client subtrees use `React.lazy` with `Suspense` when needed.

# 7. Understanding Re-renders in Client Components

Unnecessary re-renders waste CPU and hurt INP (Interaction to Next Paint).

### Common causes

| Cause | Fix |
| ----- | --- |
| New object/array in props every render | `useMemo` for stable references |
| Inline functions passed to memoized children | `useCallback` |
| Context value recreated | Memoize context value |
| State too high in tree | Colocate state near usage |

```tsx
'use client'

import { useCallback, useMemo, memo } from 'react'

const ExpensiveList = memo(function ExpensiveList({
  items,
  onSelect,
}: {
  items: string[]
  onSelect: (item: string) => void
}) {
  return (
    <ul>
      {items.map((item) => (
        <li key={item} onClick={() => onSelect(item)}>
          {item}
        </li>
      ))}
    </ul>
  )
})

export function ProductPicker({ products }: { products: string[] }) {
  const [selected, setSelected] = useState<string | null>(null)

  const handleSelect = useCallback((item: string) => {
    setSelected(item)
  }, [])

  const sorted = useMemo(() => [...products].sort(), [products])

  return (
    <div>
      <ExpensiveList items={sorted} onSelect={handleSelect} />
      {selected && <p>Selected: {selected}</p>}
    </div>
  )
}
```

**Do not** memoize everything — profile first. Prefer Server Components to avoid shipping interactive JS.

# 8. Server Component Performance Habits

| Habit | Impact |
| ----- | ------ |
| Fetch in parallel | `Promise.all([getUser(), getPosts()])` |
| Stream with Suspense | Show shell while slow data loads |
| Minimize `'use client'` | Smaller JS bundle |
| Cache fetch results | Default caching + revalidation (Topic 24) |
| Avoid serial waterfalls | Don't await A before starting B if independent |

```tsx
export default async function DashboardPage() {
  const [user, stats, activity] = await Promise.all([
    getUser(),
    getStats(),
    getActivity(),
  ])

  return (
    <DashboardView user={user} stats={stats} activity={activity} />
  )
}
```

# 9. Core Web Vitals and Lighthouse

| Metric | Measures | Target |
| ------ | -------- | ------ |
| **LCP** (Largest Contentful Paint) | Main content load time | ≤ 2.5s |
| **INP** (Interaction to Next Paint) | Responsiveness to input | ≤ 200ms |
| **CLS** (Cumulative Layout Shift) | Visual stability | ≤ 0.1 |

### Lighthouse audit

1. Open Chrome DevTools → Lighthouse tab
2. Run audit for Performance, Accessibility, Best Practices, SEO
3. Fix flagged issues: unoptimized images, render-blocking fonts, large JS

### Next.js built-in analytics

```tsx
// app/layout.tsx
import { SpeedInsights } from '@vercel/speed-insights/next'
import { Analytics } from '@vercel/analytics/react'

export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <html lang="en">
      <body>
        {children}
        <Analytics />
        <SpeedInsights />
      </body>
    </html>
  )
}
```

Use Web Vitals data from production — lab scores differ from real user devices.

# 10. Performance Checklist

✔ Use `next/image` for all content images with `width`/`height` or `fill`

✔ Set `priority` on LCP hero image only

✔ Use `next/font` instead of external font CDN links

✔ Dynamic import for charts, maps, rich editors

✔ Parallel data fetching in Server Components

✔ Keep client components small — push `'use client'` down

✔ Run Lighthouse before major releases

✔ Monitor Core Web Vitals in production

# 12. Summary

| Tool | Optimization |
| ---- | ------------ |
| `next/image` | Lazy load, responsive sizes, WebP/AVIF |
| `next/font` | Self-hosted fonts, reduced CLS |
| `dynamic()` | Route and component code splitting |
| Server Components | Less client JavaScript |
| Parallel fetch | Faster server render |
| Core Web Vitals | LCP, INP, CLS — measure and improve |

Performance is a feature. Use Next.js built-ins before reaching for custom solutions.
