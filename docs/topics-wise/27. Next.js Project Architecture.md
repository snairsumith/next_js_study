# Next.js Project Architecture - Teaching Notes

## Topic 27: Scalable Folder Structure and Code Organization

---

# 1. Why This Topic Matters

Small tutorials fit everything in `app/page.tsx`. Real apps grow to hundreds of files. Without structure, teams waste time finding code, duplicate logic, and mix server/client concerns.

Good architecture in Next.js means:

* Predictable file locations
* Clear server vs client boundaries
* Reusable utilities and types
* Features that can be owned by one developer or squad

This lesson covers folder conventions, feature-based organization, service layers, and when to use (or avoid) barrel exports.

---

# 2. Default create-next-app Structure

```text
my-app/
├── app/                  # Routes, layouts, pages (App Router)
│   ├── layout.tsx
│   ├── page.tsx
│   └── globals.css
├── public/               # Static assets (images, favicon)
├── components/           # Shared UI components
├── lib/                  # Utilities, DB clients, helpers
├── next.config.ts
├── package.json
└── tsconfig.json
```

| Folder | Purpose |
| ------ | ------- |
| `app/` | Routing only — pages, layouts, route handlers |
| `components/` | Reusable UI not tied to one route |
| `lib/` | Server-safe utilities, database, auth helpers |
| `public/` | Files served at root URL (`/logo.png`) |

With `src/` directory enabled:

```text
src/
├── app/
├── components/
└── lib/
```

Both layouts are valid. Pick one and stay consistent.

# 3. Feature-Based Organization

As apps grow, organize by **feature** instead of only by file type.

```text
src/
├── app/
│   ├── (marketing)/
│   │   ├── page.tsx
│   │   └── pricing/page.tsx
│   ├── (dashboard)/
│   │   ├── layout.tsx
│   │   └── dashboard/
│   │       ├── page.tsx
│   │       └── settings/page.tsx
│   └── api/
├── features/
│   ├── auth/
│   │   ├── components/login-form.tsx
│   │   ├── actions/login-action.ts
│   │   └── lib/session.ts
│   ├── posts/
│   │   ├── components/post-card.tsx
│   │   ├── actions/create-post.ts
│   │   └── lib/posts-service.ts
│   └── billing/
│       ├── components/plan-selector.tsx
│       └── lib/stripe.ts
├── components/           # Truly shared UI (Button, Modal)
├── hooks/                # Client hooks only
├── types/                # Shared TypeScript types
└── lib/                  # Cross-feature utilities
```

| Approach | Pros | Cons |
| -------- | ---- | ---- |
| Type-based (`components/`, `hooks/`) | Simple for small apps | Hard to navigate at scale |
| Feature-based (`features/posts/`) | Colocation, clear ownership | Some duplication across features |
| Hybrid | Shared UI global; feature logic local | Best for most production apps |

**Rule of thumb:** If only one feature uses it, keep it inside that feature folder.

# 4. The components/ Folder

Split components by scope and server/client needs.

```text
components/
├── ui/                   # Primitives: Button, Input, Card
│   ├── button.tsx
│   └── input.tsx
├── layout/               # Header, Footer, Sidebar
│   └── site-header.tsx
└── forms/                # Reusable form pieces
    └── form-field.tsx
```

### Naming conventions

| Pattern | Example | Notes |
| ------- | ------- | ----- |
| kebab-case files | `post-card.tsx` | Matches Next.js community style |
| PascalCase export | `export function PostCard()` | Component name |
| Client suffix (optional) | `search-bar-client.tsx` | When server wrapper exists |

### Server vs client file placement

```text
features/posts/
├── components/
│   ├── post-list.tsx          # Server Component (default)
│   └── post-like-button.tsx   # 'use client'
```

Keep `'use client'` files as **small leaf components**. Import client components into server parents — never the reverse at the top level.

# 5. The lib/ Folder

`lib/` holds server-safe utilities, database clients, and third-party wrappers.

```text
lib/
├── db/
│   ├── client.ts           # Prisma or Drizzle client
│   └── users.ts            # User queries
├── auth/
│   └── session.ts
├── utils/
│   ├── format-date.ts
│   └── cn.ts               # Tailwind class merge helper
└── constants/
    └── routes.ts
```

```typescript
// lib/db/client.ts
import { PrismaClient } from '@prisma/client'

const globalForPrisma = globalThis as unknown as { prisma: PrismaClient }

export const prisma = globalForPrisma.prisma || new PrismaClient()

if (process.env.NODE_ENV !== 'production') {
  globalForPrisma.prisma = prisma
}
```

| Belongs in lib/ | Does not belong in lib/ |
| --------------- | ----------------------- |
| Database queries | React components |
| Auth session helpers | `'use client'` hooks |
| Pure formatting functions | JSX or browser APIs |

# 6. hooks/, types/, and constants/

### hooks/ — client only

```typescript
// hooks/use-debounce.ts
'use client'

import { useState, useEffect } from 'react'

export function useDebounce<T>(value: T, delay = 300): T {
  const [debounced, setDebounced] = useState(value)

  useEffect(() => {
    const timer = setTimeout(() => setDebounced(value), delay)
    return () => clearTimeout(timer)
  }, [value, delay])

  return debounced
}
```

Every file in `hooks/` should start with `'use client'`.

### types/ — shared interfaces

```typescript
// types/post.ts
export interface Post {
  id: string
  title: string
  content: string
  authorId: string
  createdAt: Date
}

export interface CreatePostInput {
  title: string
  content: string
}
```

Share types between Server Actions, Route Handlers, and client components. Avoid importing server-only modules into type files.

### constants/

```typescript
// lib/constants/routes.ts
export const ROUTES = {
  home: '/',
  login: '/login',
  dashboard: '/dashboard',
  admin: '/admin',
} as const

export const POSTS_PER_PAGE = 10
```

# 7. API Service Layer

Do not put raw database calls inside every page. Use a **service layer**.

```typescript
// features/posts/lib/posts-service.ts
import { prisma } from '@/lib/db/client'
import type { CreatePostInput } from '@/types/post'

export async function getPostsByUser(userId: string) {
  return prisma.post.findMany({
    where: { authorId: userId },
    orderBy: { createdAt: 'desc' },
  })
}

export async function createPost(userId: string, input: CreatePostInput) {
  return prisma.post.create({
    data: {
      ...input,
      authorId: userId,
    },
  })
}
```

```typescript
// features/posts/actions/create-post.ts
'use server'

import { auth } from '@/auth'
import { createPost } from '@/features/posts/lib/posts-service'
import { revalidatePath } from 'next/cache'

export async function createPostAction(formData: FormData) {
  const session = await auth()
  if (!session?.user?.id) throw new Error('Unauthorized')

  const title = formData.get('title') as string
  const content = formData.get('content') as string

  await createPost(session.user.id, { title, content })
  revalidatePath('/dashboard/posts')
}
```

| Layer | Responsibility |
| ----- | -------------- |
| Page (Server Component) | Fetch data, compose UI |
| Service (`lib/*-service.ts`) | Business logic, DB queries |
| Action (`actions/*.ts`) | Auth, validation, revalidation |
| Component | Display and user interaction |

# 8. Separating Server and Client Components

```text
Wrong boundary:
─────────────────
app/dashboard/page.tsx  ('use client' — entire page client-side)

Better boundary:
──────────────────
app/dashboard/page.tsx          (Server — fetches data)
  └── DashboardStats.tsx        (Server — static stats)
  └── ActivityFeed.tsx            (Server — async list)
        └── LikeButton.tsx        (Client — onClick handler)
```

### Composition pattern

```tsx
// app/products/page.tsx — Server Component
import { getProducts } from '@/features/products/lib/products-service'
import { AddToCartButton } from '@/features/products/components/add-to-cart-button'

export default async function ProductsPage() {
  const products = await getProducts()

  return (
    <ul>
      {products.map((product) => (
        <li key={product.id}>
          {product.name} — ${product.price}
          <AddToCartButton productId={product.id} />
        </li>
      ))}
    </ul>
  )
}
```

Push `'use client'` as far down the tree as possible to minimize JavaScript sent to the browser.

# 9. Route Groups and Colocation

Route groups `(folder)` organize routes without affecting URLs.

```text
app/
├── (auth)/
│   ├── login/page.tsx       → /login
│   └── register/page.tsx    → /register
├── (shop)/
│   ├── products/page.tsx    → /products
│   └── cart/page.tsx        → /cart
└── (admin)/
    ├── layout.tsx           # Admin shell
    └── admin/page.tsx       → /admin
```

Colocate route-specific components next to the route when they are not reused elsewhere:

```text
app/dashboard/
├── page.tsx
├── loading.tsx
└── _components/
    └── dashboard-chart.tsx
```

Prefix with `_` or use a private folder pattern so Next.js does not treat it as a route.

---

# 10. Best Practices

✔ Keep `app/` focused on routing — move logic to `features/` or `lib/`

✔ One main component per file; named exports for utilities

✔ Use `@/` path alias consistently (`@/components`, `@/lib`)

✔ Colocate feature code; extract to shared folders only when reused twice+

✔ Service layer between pages and database

✔ Minimize `'use client'` — leaf components only

✔ Document folder conventions in project README for new teammates

✔ Avoid deep barrel exports in performance-critical paths

# 12. Summary

| Concept | Key takeaway |
| ------- | ------------ |
| Folder structure | `app/` for routes; `features/` for domain logic |
| components/ | Shared UI; split server vs client files |
| lib/ | Server utilities, DB, auth — no JSX |
| Service layer | Pages call services; actions handle auth + revalidation |
| Server/client split | Push `'use client'` to leaf components |
| Barrel exports | Useful for UI kits; avoid in hot paths |

Good architecture makes Topics 28–36 (TypeScript, security, testing, deployment) easier because boundaries are already clear.
