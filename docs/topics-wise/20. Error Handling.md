# Error Handling - Teaching Notes

## Topic 20: Graceful Failures, Error Boundaries, and notFound

---

# 1. Why This Topic Matters

Apps fail — networks go down, databases timeout, users hit invalid URLs. How your app handles errors separates a polished product from a broken experience.

Next.js App Router provides built-in conventions for errors: `error.tsx` for runtime failures, `not-found.tsx` for missing resources, and `global-error.tsx` for root-level crashes. Learning these patterns means users always see helpful messages instead of blank screens.

---

# 2. Error Handling Layers in Next.js

```text
Request
   │
   ▼
page.tsx ── throws error ──► error.tsx (route-level boundary)
   │
   ├── notFound() ──► not-found.tsx
   │
   └── Root crash ──► global-error.tsx
```

| File | Scope | Trigger |
| ---- | ----- | ------- |
| `error.tsx` | Route segment + children | Uncaught error in Server/Client Component |
| `not-found.tsx` | Route segment | `notFound()` called or no matching route |
| `global-error.tsx` | Entire app (root layout) | Error in root `layout.tsx` |
| `try/catch` | Inside your code | Expected errors you handle manually |

---

# 3. error.tsx — Route Error Boundaries

`error.tsx` must be a **Client Component** — it uses React error boundary APIs internally.

```tsx
// app/products/error.tsx
'use client'

interface ErrorPageProps {
  error: Error & { digest?: string }
  reset: () => void
}

export default function ProductsError({ error, reset }: ErrorPageProps) {
  return (
    <main role="alert">
      <h1>Something went wrong</h1>
      <p>We could not load the products page. Please try again.</p>

      {/* Show details only in development */}
      {process.env.NODE_ENV === 'development' && (
        <pre>{error.message}</pre>
      )}

      <button type="button" onClick={reset}>
        Try again
      </button>
    </main>
  )
}
```

| Prop | Type | Purpose |
| ---- | ---- | ------- |
| `error` | `Error` | The thrown error object |
| `error.digest` | `string?` | Server error hash (safe to show users) |
| `reset` | `() => void` | Re-render the segment — retry without full reload |

**Key rules:**
* Must have `'use client'` at the top
* Wraps the route segment where it's placed — not sibling routes
* Does not catch errors in `layout.tsx` at the same level (needs `layout` error boundary or parent `error.tsx`)

# 4. Nested Error Boundaries

Place `error.tsx` at different levels for granular error UI.

```text
app/
  error.tsx              ← catches errors in all app routes
  products/
    error.tsx            ← catches errors only in /products/*
    [id]/
      error.tsx          ← catches errors only in /products/[id]
      page.tsx
```

```tsx
// app/products/[id]/error.tsx
'use client'

interface ProductErrorProps {
  error: Error & { digest?: string }
  reset: () => void
}

export default function ProductError({ error, reset }: ProductErrorProps) {
  return (
    <article role="alert">
      <h2>Could not load this product</h2>
      <p>The product details are temporarily unavailable.</p>
      <button type="button" onClick={reset}>Retry</button>
      <a href="/products">Back to all products</a>
    </article>
  )
}
```

| Granularity | User experience |
| ----------- | --------------- |
| Root `error.tsx` | Generic app error page |
| Feature `error.tsx` | Feature-specific message + recovery |
| Page `error.tsx` | Most specific — best UX |

# 5. global-error.tsx

Catches errors in the **root layout** itself. Must include `<html>` and `<body>` tags.

```tsx
// app/global-error.tsx
'use client'

interface GlobalErrorProps {
  error: Error & { digest?: string }
  reset: () => void
}

export default function GlobalError({ error, reset }: GlobalErrorProps) {
  return (
    <html lang="en">
      <body>
        <main role="alert">
          <h1>Application Error</h1>
          <p>A critical error occurred. Please refresh the page.</p>
          {process.env.NODE_ENV === 'development' && (
            <pre>{error.message}</pre>
          )}
          <button type="button" onClick={reset}>
            Try again
          </button>
        </main>
      </body>
    </html>
  )
}
```

| `error.tsx` | `global-error.tsx` |
| ----------- | ------------------ |
| Route segment errors | Root layout errors |
| Renders inside layout | Must include own `<html>` and `<body>` |
| More common | Rare — last resort |

# 6. not-found.tsx and notFound()

For missing resources — not server crashes.

```tsx
// app/products/[id]/page.tsx
import { notFound } from 'next/navigation'
import { getProductById } from '@/lib/products'

interface ProductPageProps {
  params: Promise<{ id: string }>
}

export default async function ProductPage({ params }: ProductPageProps) {
  const { id } = await params
  const product = await getProductById(id)

  if (!product) {
    notFound() // Renders nearest not-found.tsx — returns 404 status
  }

  return (
    <main>
      <h1>{product.name}</h1>
      <p>Rs. {product.price}</p>
    </main>
  )
}
```

```tsx
// app/products/[id]/not-found.tsx
export default function ProductNotFound() {
  return (
    <main>
      <h1>Product Not Found</h1>
      <p>This product does not exist or may have been removed.</p>
      <a href="/products">Browse all products</a>
    </main>
  )
}
```

```tsx
// app/not-found.tsx — global 404
export default function NotFound() {
  return (
    <main>
      <h1>404 — Page Not Found</h1>
      <p>The page you requested does not exist.</p>
      <a href="/">Go home</a>
    </main>
  )
}
```

| Function / file | HTTP status | Use when |
| --------------- | ----------- | -------- |
| `notFound()` | 404 | Resource doesn't exist |
| `not-found.tsx` | 404 UI | Custom 404 page design |
| `error.tsx` | 500 | Unexpected runtime error |

# 7. try/catch in Server Components

Handle **expected** errors inline without triggering error boundaries.

```tsx
// app/dashboard/page.tsx
import { getStats, getRecentActivity } from '@/lib/dashboard'

export default async function DashboardPage() {
  let stats = null
  let activity = []
  let statsError = false

  try {
    stats = await getStats()
  } catch (error) {
    console.error('Stats fetch failed:', error)
    statsError = true
  }

  try {
    activity = await getRecentActivity()
  } catch (error) {
    console.error('Activity fetch failed:', error)
    // activity stays [] — show empty state
  }

  return (
    <main>
      <h1>Dashboard</h1>

      {statsError ? (
        <p role="alert">Stats are temporarily unavailable.</p>
      ) : (
        <section>
          <p>Total users: {stats?.totalUsers}</p>
          <p>Revenue: Rs. {stats?.revenue}</p>
        </section>
      )}

      <section>
        <h2>Recent Activity</h2>
        {activity.length === 0 ? (
          <p>No recent activity.</p>
        ) : (
          <ul>
            {activity.map((item) => (
              <li key={item.id}>{item.description}</li>
            ))}
          </ul>
        )}
      </section>
    </main>
  )
}
```

| Strategy | When to use |
| -------- | ----------- |
| `try/catch` + inline UI | Expected failures, partial page failures |
| `throw error` → `error.tsx` | Unexpected failures, full page broken |
| `notFound()` | Resource doesn't exist (404) |
| Return fallback value | Non-critical data (empty array, null) |

# 8. Error Handling in Server Actions

Server Actions should return errors as values — not throw — for expected validation failures.

```tsx
// app/actions/contact.ts
'use server'

interface ContactResult {
  success: boolean
  error?: string
}

export async function submitContactForm(formData: FormData): Promise<ContactResult> {
  const name = formData.get('name') as string
  const email = formData.get('email') as string
  const message = formData.get('message') as string

  if (!name?.trim() || !email?.trim() || !message?.trim()) {
    return { success: false, error: 'All fields are required.' }
  }

  if (!email.includes('@')) {
    return { success: false, error: 'Please enter a valid email address.' }
  }

  try {
    await sendEmail({ name, email, message })
    return { success: true }
  } catch (error) {
    console.error('Contact form failed:', error)
    return { success: false, error: 'Could not send message. Please try again later.' }
  }
}
```

```tsx
// components/contact-form.tsx
'use client'

import { useActionState } from 'react'
import { submitContactForm } from '@/app/actions/contact'

export function ContactForm() {
  const [state, action, isPending] = useActionState(
    async (_prev: { success: boolean; error?: string }, formData: FormData) => {
      return submitContactForm(formData)
    },
    { success: false }
  )

  if (state.success) {
    return <p role="status">Message sent successfully!</p>
  }

  return (
    <form action={action}>
      {state.error && <p role="alert">{state.error}</p>}
      <input name="name" placeholder="Your name" required />
      <input name="email" type="email" placeholder="Email" required />
      <textarea name="message" placeholder="Message" required />
      <button type="submit" disabled={isPending}>
        {isPending ? 'Sending...' : 'Send Message'}
      </button>
    </form>
  )
}
```

| Error type | Handling |
| ---------- | -------- |
| Validation error | Return `{ success: false, error: '...' }` |
| Expected failure (email send fails) | Return error message to user |
| Unexpected crash | Let it throw → caught by error boundary |

# 9. Error Handling in Route Handlers

Return proper HTTP status codes with consistent JSON error shapes.

```tsx
// app/api/users/[id]/route.ts
import { NextRequest, NextResponse } from 'next/server'

interface RouteContext {
  params: Promise<{ id: string }>
}

export async function GET(_request: NextRequest, context: RouteContext) {
  try {
    const { id } = await context.params

    if (!id || id.length === 0) {
      return NextResponse.json({ error: 'Invalid user ID' }, { status: 400 })
    }

    const user = await db.user.findUnique({ where: { id } })

    if (!user) {
      return NextResponse.json({ error: 'User not found' }, { status: 404 })
    }

    return NextResponse.json(user)
  } catch (error) {
    console.error('GET /api/users/[id] failed:', error)
    return NextResponse.json(
      { error: 'Internal server error' },
      { status: 500 }
    )
  }
}
```

| Status | Error type | User-facing message |
| ------ | ---------- | ------------------- |
| 400 | Bad input | Specific validation message |
| 401 | Not authenticated | "Please sign in" |
| 403 | Forbidden | "You don't have access" |
| 404 | Not found | "Resource not found" |
| 500 | Server error | Generic — never expose stack traces |

# 10. Logging Without Leaking Secrets

Log details server-side; show safe messages to users.

```tsx
// lib/logger.ts
export function logError(context: string, error: unknown) {
  const message = error instanceof Error ? error.message : 'Unknown error'
  const stack = error instanceof Error ? error.stack : undefined

  // Server-side only — never sent to client
  console.error(`[${context}] ${message}`, { stack, timestamp: new Date().toISOString() })

  // In production, send to monitoring service:
  // Sentry.captureException(error, { tags: { context } })
}

export function getUserFriendlyMessage(error: unknown): string {
  // Never expose internal details to users
  if (error instanceof Error && error.message.includes('ECONNREFUSED')) {
    return 'Service temporarily unavailable. Please try again.'
  }
  return 'Something went wrong. Please try again later.'
}
```

```tsx
// Usage in Server Component
try {
  const data = await fetchCriticalData()
} catch (error) {
  logError('DashboardPage.fetchCriticalData', error)
  throw new Error(getUserFriendlyMessage(error)) // error.tsx shows safe message
}
```

| Log (server) | Show (client) |
| ------------ | ------------- |
| Full stack trace | "Something went wrong" |
| Database connection string | Never |
| User ID, request path | Generic retry message |
| `error.digest` | Safe to show — no internals |

# 11. Best Practices

✔ Use `error.tsx` for unexpected runtime errors — one per feature route

✔ Use `notFound()` + `not-found.tsx` for missing resources (404)

✔ Use `try/catch` for expected errors with inline fallback UI

✔ Return error objects from Server Actions — don't throw for validation

✔ Show user-friendly messages — log details server-side only

✔ Never expose stack traces, DB errors, or secrets in client UI

✔ Provide recovery actions: retry button, link back, contact support
