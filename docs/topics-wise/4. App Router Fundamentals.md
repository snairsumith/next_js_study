# App Router Fundamentals - Teaching Notes

## Topic 4: The app Directory and Special Files

---

# 1. Why This Topic Matters

The **App Router** is the heart of modern Next.js. Every page, layout, loading state, and error boundary is defined by **files in the `app/` folder** — not by a central router configuration.

Students who understand `page.tsx`, `layout.tsx`, and the other special files can:

* Create new routes by adding folders
* Share UI across pages with layouts
* Handle loading and errors per route segment
* Colocate related code next to the routes that use it

Without this foundation, Server Components, data fetching, and navigation will feel arbitrary.

---

# 2. The app Directory Structure

The `app/` directory is where all routes live. Each folder represents a URL segment.

```text
app/
├── layout.tsx           # Root layout — wraps entire app
├── page.tsx             # Home page → /
├── globals.css          # Global styles
├── favicon.ico
├── about/
│   └── page.tsx         # → /about
├── products/
│   ├── layout.tsx       # Layout for /products/*
│   ├── page.tsx         # → /products
│   ├── loading.tsx      # Loading UI for /products/*
│   ├── error.tsx        # Error boundary for /products/*
│   └── [id]/
│       └── page.tsx     # → /products/42
└── not-found.tsx        # Custom 404 for this segment
```

| Rule | Meaning |
| ---- | ------- |
| Folder name = URL segment | `app/about/` → `/about` |
| `page.tsx` makes route public | Folder without `page.tsx` is not a route |
| Files are colocated | UI, loading, error live next to each other |
| Nested folders = nested URLs | `app/blog/[slug]/page.tsx` → `/blog/hello-world` |

Compare with react-project-study where routes are declared manually:

```tsx
// react-project-study/src/App.tsx
<Routes>
  <Route path="/" element={<Home />} />
  <Route path="/products" element={<Product />} />
  <Route path="/data-fetch" element={<DataFetch />} />
</Routes>
```

In Next.js, **creating `app/data-fetch/page.tsx` automatically adds the route** — no edit to a central router file.

# 3. page.tsx — Route UI

A **`page.tsx`** file is the UI rendered for a specific URL. It is the leaf of the route tree.

```tsx
// app/page.tsx — home route (/)
export default function HomePage() {
  return (
    <main>
      <h1>Next.js Training Portal</h1>
      <p>Welcome to the home page.</p>
    </main>
  )
}
```

```tsx
// app/about/page.tsx — /about
export default function AboutPage() {
  return (
    <main>
      <h1>About Us</h1>
      <p>We teach modern web development.</p>
    </main>
  )
}
```

| Property | Detail |
| -------- | ------ |
| Export | Must be a **default export** |
| Name | Function name is for debugging only — not used in routing |
| Async | Can be `async` for Server Component data fetching |
| One per route | Each route segment has at most one `page.tsx` |

### Page vs folder without page

```text
app/
├── dashboard/
│   ├── layout.tsx      # ✅ Exists — provides shell
│   ├── settings/
│   │   └── page.tsx    # ✅ /dashboard/settings
│   └── analytics/
│       └── components/ # ❌ No page.tsx — NOT a route
│           └── Chart.tsx
```

# 4. layout.tsx — Shared UI Wrapper

A **`layout.tsx`** wraps child routes and **persists across navigation** within its segment. Layout state is not reset when navigating between sibling pages.

```tsx
// app/layout.tsx — root layout (required)
import type { Metadata } from 'next'
import './globals.css'

export const metadata: Metadata = {
  title: 'Training App',
  description: 'Next.js course project',
}

export default function RootLayout({
  children,
}: {
  children: React.ReactNode
}) {
  return (
    <html lang="en">
      <body>
        <header>
          <nav>Training Portal</nav>
        </header>
        {children}
        <footer>© 2026 Training Team</footer>
      </body>
    </html>
  )
}
```

```tsx
// app/products/layout.tsx — nested layout
export default function ProductsLayout({
  children,
}: {
  children: React.ReactNode
}) {
  return (
    <section className="products-section">
      <aside>
        <h2>Product Categories</h2>
        <ul>
          <li>Electronics</li>
          <li>Books</li>
        </ul>
      </aside>
      <div>{children}</div>
    </section>
  )
}
```

| Layout behavior | Detail |
| --------------- | ------ |
| Persists on navigation | Header/sidebar stays mounted when moving between child pages |
| Accepts `children` | Required prop — the child page or nested layout |
| Can be nested | Root → section → page layouts stack together |
| Must include `<html>` and `<body>` | Only in the **root** layout |

### Layout nesting visual

```text
RootLayout (app/layout.tsx)
  └── ProductsLayout (app/products/layout.tsx)
        └── ProductsPage (app/products/page.tsx)
```

When visiting `/products`, all three render. When navigating to `/products/123`, the root and products layouts stay mounted; only the page content changes.

# 5. loading.tsx — Loading UI

A **`loading.tsx`** file automatically wraps the route segment in a **React Suspense boundary**. It shows while the page or nested layouts are loading.

```tsx
// app/products/loading.tsx
export default function ProductsLoading() {
  return (
    <div className="loading-skeleton">
      <div className="skeleton-card" />
      <div className="skeleton-card" />
      <div className="skeleton-card" />
      <p>Loading products...</p>
    </div>
  )
}
```

```tsx
// app/products/page.tsx — slow async page
export default async function ProductsPage() {
  const products = await fetchProducts() // may take time
  return (
    <ul>
      {products.map((p) => (
        <li key={p.id}>{p.name}</li>
      ))}
    </ul>
  )
}
```

| Aspect | Detail |
| ------ | ------ |
| Automatic Suspense | Next.js wraps `page.tsx` in Suspense when `loading.tsx` exists |
| Instant navigation | Layout shows immediately; loading UI fills the page slot |
| Streaming | Server can stream HTML as data arrives |
| Scope | Applies to the route segment and its children |

Compare with react-project-study where loading is manual:

```tsx
// react-project-study pattern — manual loading state
const [isLoading, setIsLoading] = useState(true)

useEffect(() => {
  fetchData().then(() => setIsLoading(false))
}, [])

if (isLoading) return <Loading />
```

Next.js convention removes boilerplate for route-level loading.

# 6. error.tsx — Error UI

An **`error.tsx`** file creates an **error boundary** for that route segment. It catches runtime errors in child components.

```tsx
// app/products/error.tsx
'use client' // Error boundaries must be Client Components

import { useEffect } from 'react'

interface ErrorProps {
  error: Error & { digest?: string }
  reset: () => void
}

export default function ProductsError({ error, reset }: ErrorProps) {
  useEffect(() => {
    console.error('Products page error:', error)
  }, [error])

  return (
    <div role="alert">
      <h2>Something went wrong loading products</h2>
      <p>{error.message}</p>
      <button onClick={reset}>Try again</button>
    </div>
  )
}
```

| Aspect | Detail |
| ------ | ------ |
| Must be Client Component | `'use client'` required — uses React error boundary API |
| Receives `error` and `reset` | `reset()` re-attempts rendering the segment |
| Scoped to segment | Errors in `/products` do not crash the root layout |
| Does not catch layout errors | Layout errors need a layout-level error boundary |

### Triggering an error (for testing)

```tsx
// app/products/page.tsx
export default async function ProductsPage() {
  const res = await fetch('https://api.example.com/products')

  if (!res.ok) {
    throw new Error('Failed to fetch products')
  }

  const products = await res.json()
  return <ProductList items={products} />
}
```

# 7. not-found.tsx — 404 UI

A **`not-found.tsx`** file defines custom UI when a route segment is not found.

```tsx
// app/not-found.tsx — app-wide 404
import Link from 'next/link'

export default function NotFound() {
  return (
    <main>
      <h1>404 — Page Not Found</h1>
      <p>The page you requested does not exist.</p>
      <Link href="/">Go home</Link>
    </main>
  )
}
```

Call `notFound()` from a Server Component to trigger it programmatically:

```tsx
// app/products/[id]/page.tsx
import { notFound } from 'next/navigation'

interface PageProps {
  params: Promise<{ id: string }>
}

export default async function ProductDetailPage({ params }: PageProps) {
  const { id } = await params
  const product = await getProductById(id)

  if (!product) {
    notFound() // renders not-found.tsx
  }

  return <h1>{product.name}</h1>
}
```

Compare with react-project-study:

```tsx
// react-project-study/src/pages/404.tsx
<Route path="*" element={<NotFoundPage />} />
```

Next.js uses both a **file convention** and a **function** (`notFound()`).

# 8. Nested Routes and Folder Conventions

Nested folders create nested URLs automatically.

```text
app/
├── layout.tsx
├── page.tsx                    → /
└── dashboard/
    ├── layout.tsx
    ├── page.tsx                → /dashboard
    ├── settings/
    │   └── page.tsx            → /dashboard/settings
    └── analytics/
        └── page.tsx            → /dashboard/analytics
```

| URL | Files rendered (in order) |
| --- | ------------------------- |
| `/` | `app/layout.tsx` → `app/page.tsx` |
| `/dashboard` | `app/layout.tsx` → `app/dashboard/layout.tsx` → `app/dashboard/page.tsx` |
| `/dashboard/settings` | root layout → dashboard layout → settings page |

### Private folders (underscore prefix)

Folders starting with `_` are **not** route segments — useful for colocating non-route code:

```text
app/
├── dashboard/
│   ├── _components/       # Not a route — helper components
│   │   └── StatsCard.tsx
│   └── page.tsx
```

# 9. Colocation of Route Files

Next.js encourages placing everything a route needs **next to** that route.

```text
app/
└── products/
    ├── page.tsx           # Route UI
    ├── layout.tsx         # Route layout
    ├── loading.tsx        # Loading state
    ├── error.tsx          # Error boundary
    ├── not-found.tsx      # 404 for this segment
    ├── _components/       # Private components for this route
    │   ├── ProductCard.tsx
    │   └── ProductFilter.tsx
    └── _lib/
        └── getProducts.ts # Data helper for this route
```

| Colocation benefit | Explanation |
| ------------------ | ----------- |
| Easy to find code | Everything for `/products` lives under `app/products/` |
| Safe deletion | Remove the folder — route and helpers gone together |
| Clear ownership | No giant global `components/` dump |

Shared components used across many routes still belong in a top-level `components/` folder:

```text
my-next-app/
├── app/
│   └── products/
│       └── page.tsx
└── components/
    └── ui/
        └── Button.tsx     # Used by many routes
```

# 10. Special Files Reference Table

| File | Required? | Purpose | Server / Client |
| ---- | --------- | ------- | --------------- |
| `layout.tsx` | Root only | Shared UI wrapper | Server (default) |
| `page.tsx` | Per route | Route UI | Server (default) |
| `loading.tsx` | Optional | Loading fallback | Server (default) |
| `error.tsx` | Optional | Error boundary | **Client** (`'use client'`) |
| `not-found.tsx` | Optional | 404 UI | Server (default) |
| `route.ts` | Optional | API endpoint | Server |
| `template.tsx` | Optional | Re-mounting layout variant | Server (default) |
| `default.tsx` | Optional | Parallel route fallback | Server (default) |

Only **`page.tsx`** and **root `layout.tsx`** are required to start an app.

# 11. Route Segment Config (Preview)

Pages and layouts can export configuration constants:

```tsx
// app/products/page.tsx
export const dynamic = 'force-dynamic' // always SSR
export const revalidate = 60           // ISR — revalidate every 60 seconds

export default async function ProductsPage() {
  // ...
}
```

| Export | Effect |
| ------ | ------ |
| `dynamic = 'force-dynamic'` | Never cache — render on every request |
| `dynamic = 'force-static'` | Force static at build time |
| `revalidate = 60` | Regenerate page at most every 60 seconds |
| `runtime = 'edge'` | Run on Edge runtime instead of Node |

Deep caching rules are covered in Topic 24. For now, know these exports exist per route.

# 12. Complete Example: Products Route Segment

```text
app/products/
├── layout.tsx
├── loading.tsx
├── error.tsx
├── page.tsx
└── [id]/
    └── page.tsx
```

```tsx
// app/products/layout.tsx
export default function ProductsLayout({
  children,
}: {
  children: React.ReactNode
}) {
  return (
    <div className="flex gap-6">
      <aside className="w-48">Filters sidebar</aside>
      <main className="flex-1">{children}</main>
    </div>
  )
}
```

```tsx
// app/products/loading.tsx
export default function Loading() {
  return <p className="animate-pulse">Loading products...</p>
}
```

```tsx
// app/products/page.tsx
interface Product {
  id: number
  name: string
  price: number
}

async function getProducts(): Promise<Product[]> {
  const res = await fetch('https://fakestoreapi.com/products')
  return res.json()
}

export default async function ProductsPage() {
  const products = await getProducts()

  return (
    <ul>
      {products.map((product) => (
        <li key={product.id}>
          {product.name} — Rs. {product.price}
        </li>
      ))}
    </ul>
  )
}
```

When a user visits `/products`:

1. Root layout renders (header, footer)
2. Products layout renders (sidebar)
3. `loading.tsx` shows while `page.tsx` fetches data
4. Product list replaces loading UI

# 13. Best Practices

✔ One **`page.tsx`** per route — default export only

✔ Put shared chrome (nav, footer) in **`layout.tsx`**, not duplicated in every page

✔ Add **`loading.tsx`** for routes with slow data fetching

✔ Add **`error.tsx`** for routes that call external APIs

✔ Use **`notFound()`** for missing resources — not generic error throws

✔ Colocate route-specific components in `_components/` or `_lib/` folders

✔ Do not put business logic in `layout.tsx` if it slows every child navigation

✔ Remember **`error.tsx` requires `'use client'`**

# 15. Summary

| File | Purpose |
| ---- | ------- |
| `page.tsx` | UI for a URL — makes route publicly accessible |
| `layout.tsx` | Shared wrapper — persists across child navigation |
| `loading.tsx` | Automatic Suspense loading UI |

| `not-found.tsx` | Custom 404 UI |

| Colocation | Keep route files and helpers together |
