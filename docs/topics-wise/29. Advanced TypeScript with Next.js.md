## Topic 29: Generics, Zod, and End-to-End Type Safety

# 1. Why This Topic Matters

Basic TypeScript stops syntax errors. Advanced patterns stop **logic errors**:

* API returns unexpected shape → app crashes
* UI renders wrong state because loading/error/success are not discriminated
* Server Action accepts invalid input
* Generic list components lose type information

This lesson builds on Topic 28 with generics, utility types, discriminated unions, Zod validation, and patterns that keep types consistent from database to UI.

# 2. Generic Components

Generics let components work with multiple data types while preserving type information.

```tsx
interface ListProps<T> {
  items: T[]
  renderItem: (item: T) => React.ReactNode
  keyExtractor: (item: T) => string
}

export function List<T>({ items, renderItem, keyExtractor }: ListProps<T>) {
  return (
    <ul>
      {items.map((item) => (
        <li key={keyExtractor(item)}>{renderItem(item)}</li>
      ))}
    </ul>
  )
}
```

```tsx
// Usage — TypeScript infers T as Post
<List
  items={posts}
  keyExtractor={(post) => post.id}
  renderItem={(post) => <span>{post.title}</span>}
/>
```

| Without generics | With generics |
| ---------------- | ------------- |
| `items: unknown[]` | `items: T[]` |
| Manual casts in renderItem | Full autocomplete on `item` |
| Easy runtime bugs | Compile-time safety |

### Generic async data wrapper

```tsx
interface AsyncDataProps<T> {
  data: T | null
  isLoading: boolean
  error: string | null
  render: (data: T) => React.ReactNode
}

export function AsyncData<T>({
  data,
  isLoading,
  error,
  render,
}: AsyncDataProps<T>) {
  if (isLoading) return <p>Loading...</p>
  if (error) return <p role="alert">{error}</p>
  if (!data) return <p>No data</p>
  return <>{render(data)}</>
}
```

# 3. Utility Types for API Responses

TypeScript built-in utilities transform existing types.

```typescript
// types/api.ts
export interface User {
  id: string
  email: string
  passwordHash: string
  role: 'user' | 'admin'
}

// Omit sensitive fields for client
export type PublicUser = Omit<User, 'passwordHash'>

// Pick specific fields
export type UserSummary = Pick<User, 'id' | 'email'>

// Partial for update payloads
export type UpdateUserInput = Partial<Pick<User, 'email' | 'role'>>

// Readonly for config
export type AppConfig = Readonly<{
  apiUrl: string
  maxUploadSize: number
}>
```

### API response wrapper

```typescript
export interface ApiSuccess<T> {
  ok: true
  data: T
}

export interface ApiError {
  ok: false
  error: string
  code?: string
}

export type ApiResponse<T> = ApiSuccess<T> | ApiError
```

```typescript
async function fetchUser(id: string): Promise<ApiResponse<PublicUser>> {
  try {
    const res = await fetch(`/api/users/${id}`)
    if (!res.ok) {
      return { ok: false, error: 'User not found', code: 'NOT_FOUND' }
    }
    const data = (await res.json()) as PublicUser
    return { ok: true, data }
  } catch {
    return { ok: false, error: 'Network error' }
  }
}
```

# 4. Discriminated Unions for UI States

Model async UI as a union with a **discriminator** field (`status`).

```typescript
type PostListState =
  | { status: 'idle' }
  | { status: 'loading' }
  | { status: 'success'; posts: Post[] }
  | { status: 'error'; message: string }
```

```tsx
'use client'

function PostListView({ state }: { state: PostListState }) {
  switch (state.status) {
    case 'idle':
      return <p>Select a category</p>
    case 'loading':
      return <p>Loading posts...</p>
    case 'success':
      return (
        <ul>
          {state.posts.map((post) => (
            <li key={post.id}>{post.title}</li>
          ))}
        </ul>
      )
    case 'error':
      return <p role="alert">{state.message}</p>
  }
}
```

TypeScript narrows types inside each `case` — no optional chaining on possibly missing fields.

| Pattern | Benefit |
| ------- | ------- |
| `{ status: 'loading' }` | Exhaustive switch checks |
| Separate booleans (`isLoading`, `error`) | Harder to represent impossible states |
| Discriminated union | Invalid states unrepresentable |

# 5. Type-Safe Server Actions with Zod

Zod validates runtime data and infers TypeScript types from schemas.

```bash
npm install zod
```

```typescript
// lib/validations/post.ts
import { z } from 'zod'

export const createPostSchema = z.object({
  title: z.string().min(1, 'Title is required').max(200),
  content: z.string().min(10, 'Content too short'),
  published: z.boolean().default(false),
})

export type CreatePostInput = z.infer<typeof createPostSchema>
```

```typescript
'use server'

import { createPostSchema } from '@/lib/validations/post'
import { createPost } from '@/lib/db/posts'
import { auth } from '@/auth'

export interface ActionResult {
  success?: boolean
  error?: string
  fieldErrors?: Record<string, string[]>
}

export async function createPostAction(formData: FormData): Promise<ActionResult> {
  const session = await auth()
  if (!session?.user?.id) {
    return { error: 'Unauthorized' }
  }

  const raw = {
    title: formData.get('title'),
    content: formData.get('content'),
    published: formData.get('published') === 'on',
  }

  const parsed = createPostSchema.safeParse(raw)

  if (!parsed.success) {
    return {
      fieldErrors: parsed.error.flatten().fieldErrors,
    }
  }

  await createPost(session.user.id, parsed.data)
  return { success: true }
}
```

| Step | Tool |
| ---- | ---- |
| Parse input | `schema.safeParse()` |
| TypeScript type | `z.infer<typeof schema>` |
| Field errors | `parsed.error.flatten().fieldErrors` |

Never use `as CreatePostInput` on unvalidated FormData.

# 6. Type-Safe Route Handlers

Apply the same Zod pattern to API routes.

```typescript
// app/api/posts/route.ts
import { NextRequest, NextResponse } from 'next/server'
import { createPostSchema } from '@/lib/validations/post'
import { auth } from '@/auth'

export async function POST(request: NextRequest) {
  const session = await auth()
  if (!session?.user?.id) {
    return NextResponse.json({ error: 'Unauthorized' }, { status: 401 })
  }

  let body: unknown
  try {
    body = await request.json()
  } catch {
    return NextResponse.json({ error: 'Invalid JSON' }, { status: 400 })
  }

  const parsed = createPostSchema.safeParse(body)
  if (!parsed.success) {
    return NextResponse.json(
      { errors: parsed.error.flatten().fieldErrors },
      { status: 422 }
    )
  }

  const post = await createPost(session.user.id, parsed.data)
  return NextResponse.json(post, { status: 201 })
}
```

Reuse the **same schema** in Server Actions and Route Handlers to avoid validation drift.

# 8. End-to-End Type Safety Patterns

### Single source of truth for entities

```text
types/post.ts          → Post interface
lib/validations/post.ts → Zod schemas
lib/db/posts.ts        → Functions return Post
features/posts/        → UI consumes Post
```

### Const assertions for routes and enums

```typescript
export const USER_ROLES = ['user', 'admin', 'editor'] as const
export type UserRole = (typeof USER_ROLES)[number]

export function isAdmin(role: UserRole): boolean {
  return role === 'admin'
}
```

Prefer const arrays over TypeScript `enum` — better tree-shaking and JS interop.

### Satisfies operator

```typescript
const navItems = [
  { href: '/dashboard', label: 'Dashboard' },
  { href: '/settings', label: 'Settings' },
] as const satisfies ReadonlyArray<{ href: string; label: string }>
```

# 10. Best Practices

✔ Reuse Zod schemas across actions, APIs, and forms

✔ Use discriminated unions for loading/error/success UI

✔ Generic components when the same structure serves multiple entity types

✔ `z.infer<>` — never duplicate schema as manual interface

✔ Use `safeParse`, not `parse`, in user-facing code paths

✔ Map Zod field errors to form inputs for accessibility

✔ Avoid TypeScript enums; use `as const` maps

✔ Run `tsc --noEmit` in CI alongside `next build`

# 12. Summary

| Pattern | Purpose |
| ------- | ------- |
| Generic components | Reusable lists, tables, async wrappers |
| Utility types | `Omit`, `Pick`, `Partial` for API layers |
| Discriminated unions | Type-safe UI state machines |
| Zod + `z.infer` | Runtime validation + TypeScript types |
| Shared schemas | One validation source for actions and APIs |
| `as const` | Type-safe string unions without enums |

Advanced typing reduces production bugs and makes refactors safer across the full Next.js stack.
