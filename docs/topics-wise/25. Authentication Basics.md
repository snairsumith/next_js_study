# Authentication Basics - Teaching Notes

## Topic 25: Securing Routes, Sessions, and Server Actions

---

# 1. Why This Topic Matters

Authentication answers one question: **Who is this user?** Authorization answers: **What are they allowed to do?**

In Next.js, auth touches every layer:

| Layer | Auth responsibility |
| ----- | ------------------- |
| Middleware | Block unauthenticated requests before pages load |
| Server Components | Read session and show personalized data |
| Server Actions | Verify user before mutating data |
| Route Handlers | Protect API endpoints |
| Client Components | Display login state — never enforce security |

A common beginner mistake is checking auth only in the browser (`if (user) show dashboard`). Attackers can bypass client checks. **Always verify on the server.**

This lesson covers session vs JWT, cookie-based sessions, login/logout flows, and protecting routes in the App Router.

---

# 2. Auth in Next.js Overview

Next.js does not ship a built-in auth system. You choose an approach:

| Approach | How it works | Good for |
| -------- | ------------ | -------- |
| Cookie session | Server stores session; cookie holds session ID | Traditional web apps |
| JWT in cookie | Signed token in httpOnly cookie | Stateless APIs |
| JWT in header | Token sent via `Authorization` header | Mobile / SPA clients |
| Auth library (Auth.js) | Handles providers, sessions, callbacks | Production apps (Topic 26) |

```text
Login Flow (cookie session)
───────────────────────────
User submits credentials
       │
       ▼
Server validates against database
       │
       ▼
Server creates session → stores in DB or encrypted cookie
       │
       ▼
Browser receives httpOnly session cookie
       │
       ▼
Every request sends cookie → server reads session
```

# 3. Session vs JWT

### Session-based auth

* Server stores session data (user id, role, expiry)
* Browser holds a small **session ID** in a cookie
* Server looks up session on each request
* Easy to revoke (delete session server-side)

### JWT (JSON Web Token)

* Self-contained token with encoded payload (user id, expiry)
* Server verifies signature — no database lookup required
* Harder to revoke before expiry (need blocklist or short TTL)
* Often stored in httpOnly cookie in Next.js (not localStorage)

| Factor | Session | JWT |
| ------ | ------- | --- |
| Revocation | Easy | Harder |
| DB lookup | Yes (usually) | No (stateless) |
| Size | Small cookie | Larger token |
| Next.js fit | Excellent with Server Components | Good with httpOnly cookies |

**Recommendation for beginners:** Start with cookie sessions (Auth.js handles this in Topic 26).

# 4. Cookie-Based Sessions Overview

Cookies are key-value pairs the browser sends automatically with every request to your domain.

### Important cookie flags

| Flag | Purpose |
| ---- | ------- |
| `httpOnly` | JavaScript cannot read it — prevents XSS token theft |
| `secure` | Sent only over HTTPS in production |
| `sameSite: 'lax'` | Reduces CSRF risk |
| `maxAge` / `expires` | Session lifetime |

```typescript
// Setting a session cookie in a Route Handler or Server Action
import { cookies } from 'next/headers'

export async function createSession(userId: string) {
  const cookieStore = await cookies()
  cookieStore.set('session', userId, {
    httpOnly: true,
    secure: process.env.NODE_ENV === 'production',
    sameSite: 'lax',
    maxAge: 60 * 60 * 24 * 7, // 7 days
    path: '/',
  })
}
```

### Reading the session on the server

```typescript
// lib/auth/session.ts
import { cookies } from 'next/headers'

export async function getSessionUserId(): Promise<string | null> {
  const cookieStore = await cookies()
  const session = cookieStore.get('session')
  return session?.value ?? null
}
```

Never store passwords in cookies. Store only a session identifier or signed token.

# 5. Login and Logout Flows

### Login with Server Action

```typescript
// app/actions/auth.ts
'use server'

import { redirect } from 'next/navigation'
import { cookies } from 'next/headers'
import { verifyCredentials } from '@/lib/auth/verify'

export async function loginAction(formData: FormData) {
  const email = formData.get('email') as string
  const password = formData.get('password') as string

  const user = await verifyCredentials(email, password)
  if (!user) {
    return { error: 'Invalid email or password' }
  }

  const cookieStore = await cookies()
  cookieStore.set('session', user.id, {
    httpOnly: true,
    secure: process.env.NODE_ENV === 'production',
    sameSite: 'lax',
    maxAge: 60 * 60 * 24,
    path: '/',
  })

  redirect('/dashboard')
}
```

### Login form (progressive enhancement)

```tsx
// app/login/page.tsx
import { loginAction } from '@/app/actions/auth'

export default function LoginPage() {
  return (
    <form action={loginAction} className="max-w-sm space-y-4">
      <input name="email" type="email" required placeholder="Email" />
      <input name="password" type="password" required placeholder="Password" />
      <button type="submit">Sign in</button>
    </form>
  )
}
```

### Logout

```typescript
'use server'

import { cookies } from 'next/headers'
import { redirect } from 'next/navigation'

export async function logoutAction() {
  const cookieStore = await cookies()
  cookieStore.delete('session')
  redirect('/login')
}
```

# 6. Protecting Routes with Middleware

Middleware runs **before** the page renders. Use it for auth redirects.

```typescript
// middleware.ts (project root)
import { NextResponse } from 'next/server'
import type { NextRequest } from 'next/server'

const protectedPaths = ['/dashboard', '/settings', '/admin']

export function middleware(request: NextRequest) {
  const session = request.cookies.get('session')?.value
  const { pathname } = request.nextUrl

  const isProtected = protectedPaths.some((path) =>
    pathname.startsWith(path)
  )

  if (isProtected && !session) {
    const loginUrl = new URL('/login', request.url)
    loginUrl.searchParams.set('callbackUrl', pathname)
    return NextResponse.redirect(loginUrl)
  }

  // Redirect logged-in users away from login page
  if (session && pathname === '/login') {
    return NextResponse.redirect(new URL('/dashboard', request.url))
  }

  return NextResponse.next()
}

export const config = {
  matcher: ['/dashboard/:path*', '/settings/:path*', '/admin/:path*', '/login'],
}
```

| Middleware can | Middleware cannot |
| -------------- | ----------------- |
| Read cookies and headers | Access database directly (limited runtime) |
| Redirect and rewrite | Replace full authorization logic |
| Run on Edge | Access Node.js-only APIs |

Middleware is a **gate**, not the only security layer. Always re-check auth in Server Actions and Route Handlers.

# 7. Protecting Server Components and Pages

Middleware may be bypassed in edge cases. Pages should verify session too.

```tsx
// app/dashboard/page.tsx
import { redirect } from 'next/navigation'
import { getSessionUserId } from '@/lib/auth/session'
import { getUserById } from '@/lib/db/users'

export default async function DashboardPage() {
  const userId = await getSessionUserId()
  if (!userId) redirect('/login')

  const user = await getUserById(userId)
  if (!user) redirect('/login')

  return (
    <main>
      <h1>Welcome, {user.name}</h1>
    </main>
  )
}
```

### Layout-level protection

Protect at the dashboard `layout.tsx` with `getSessionUserId()` and `redirect('/login')` — avoids repeating checks in every child page.

# 8. Protecting Server Actions

Server Actions are public endpoints. **Always validate auth inside the action.**

```typescript
'use server'

import { revalidatePath } from 'next/cache'
import { getSessionUserId } from '@/lib/auth/session'
import { deletePost } from '@/lib/db/posts'

export async function deletePostAction(postId: string) {
  const userId = await getSessionUserId()
  if (!userId) {
    throw new Error('Unauthorized')
  }

  const post = await getPostById(postId)
  if (!post || post.authorId !== userId) {
    throw new Error('Forbidden')
  }

  await deletePost(postId)
  revalidatePath('/dashboard/posts')
}
```

| Check | Where |
| ----- | ----- |
| Is user logged in? | Every mutating Server Action |
| Does user own this resource? | Before update/delete |
| Does user have role? | Admin-only actions |

# 9. Never Trust Client-Only Auth Checks

Client components may hide buttons for UX, but attackers can bypass them. Always verify on the server:

```tsx
// app/admin/page.tsx — Server Component
export default async function AdminPage() {
  const user = await getSessionUser()
  if (!user?.isAdmin) redirect('/unauthorized')
  return <AdminPanelClient userId={user.id} />
}
```

**Security decisions belong on the server.**

---

# 10. Best Practices

✔ Use `httpOnly`, `secure`, and `sameSite` cookies for sessions

✔ Hash passwords with bcrypt or Argon2 — never store plain text

✔ Protect routes in middleware **and** verify again in Server Actions

✔ Use `redirect('/login')` in Server Components when session is missing

✔ Return generic login errors ("Invalid credentials") — do not reveal which field failed

✔ Set reasonable session expiry and implement logout

✔ Never put secrets or session tokens in `NEXT_PUBLIC_` variables

✔ Use Auth.js (Topic 26) for OAuth and production-ready session handling

# 12. Summary

| Concept | Key takeaway |
| ------- | ------------ |
| Session vs JWT | Sessions are easier to revoke; JWTs are stateless |
| Cookies | Use httpOnly + secure + sameSite for session storage |
| Middleware | First gate — redirect before page load |
| Server Components | Re-verify session; fetch user-specific data |
| Server Actions | Always check auth before mutations |
| Client checks | UX only — never the security boundary |
