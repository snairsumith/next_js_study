# Data Fetching Patterns - Teaching Notes

## Topic 14: Fetching Data the Next.js Way

---

# 1. Why This Topic Matters

Data fetching is the heart of most web applications. Next.js changes **where**, **when**, and **how** you fetch compared to plain React.

In Vite/React you typically use `useEffect` + `fetch` in the browser. In Next.js App Router, the preferred pattern is fetching in **Server Components** on the server — closer to your database, cacheable, and invisible to users' DevTools.

This topic covers parallel vs sequential fetching, caching options, and when client-side fetching still makes sense.

---

# 2. Server Fetch vs Client Fetch

| Approach | Where it runs | When to use |
| -------- | ------------- | ----------- |
| Server Component `fetch` | Server | Default for page data, SEO content |
| Server Action | Server | Mutations (create, update, delete) |
| Route Handler | Server | REST API for external clients |
| Client `useEffect` + fetch | Browser | Real-time updates, user-specific polling |
| TanStack Query | Browser | Complex client cache, optimistic UI |

```tsx
// ✅ Preferred — Server Component (app/posts/page.tsx)
async function getPosts() {
  const res = await fetch('https://jsonplaceholder.typicode.com/posts')
  if (!res.ok) throw new Error('Failed to fetch posts')
  return res.json()
}

export default async function PostsPage() {
  const posts = await getPosts()
  return (
    <ul>
      {posts.map((post: { id: number; title: string }) => (
        <li key={post.id}>{post.title}</li>
      ))}
    </ul>
  )
}
```

```tsx
// ⚠️ Client fetch — use only when server cannot handle it
'use client'

import { useEffect, useState } from 'react'

interface Post {
  id: number
  title: string
}

export function LivePosts() {
  const [posts, setPosts] = useState<Post[]>([])

  useEffect(() => {
    fetch('/api/posts')
      .then((res) => res.json())
      .then(setPosts)
  }, [])

  return (
    <ul>
      {posts.map((post) => (
        <li key={post.id}>{post.title}</li>
      ))}
    </ul>
  )
}
```

# 3. Parallel Data Fetching

When sections of a page need **independent** data, fetch them in parallel with `Promise.all`.

```tsx
// lib/data.ts
export async function getUser(id: string) {
  const res = await fetch(`https://api.example.com/users/${id}`, {
    next: { revalidate: 3600 },
  })
  return res.json()
}

export async function getOrders(userId: string) {
  const res = await fetch(`https://api.example.com/users/${userId}/orders`, {
    cache: 'no-store',
  })
  return res.json()
}

export async function getRecommendations(userId: string) {
  const res = await fetch(`https://api.example.com/recommendations/${userId}`)
  return res.json()
}
```

```tsx
// app/profile/[id]/page.tsx
import { getUser, getOrders, getRecommendations } from '@/lib/data'

interface ProfilePageProps {
  params: Promise<{ id: string }>
}

export default async function ProfilePage({ params }: ProfilePageProps) {
  const { id } = await params

  // All three requests start at the same time
  const [user, orders, recommendations] = await Promise.all([
    getUser(id),
    getOrders(id),
    getRecommendations(id),
  ])

  return (
    <main>
      <h1>{user.name}</h1>
      <section>
        <h2>Orders ({orders.length})</h2>
        {/* render orders */}
      </section>
      <section>
        <h2>Recommended for you</h2>
        {/* render recommendations */}
      </section>
    </main>
  )
}
```

| Pattern | Total wait time (if each takes 1s) |
| ------- | ----------------------------------- |
| Sequential (`await` one by one) | ~3 seconds |
| Parallel (`Promise.all`) | ~1 second |

# 4. Sequential Fetching When Needed

Use sequential fetching when one request **depends on** the result of another.

```tsx
// app/team/[teamId]/members/page.tsx
interface TeamMembersPageProps {
  params: Promise<{ teamId: string }>
}

export default async function TeamMembersPage({ params }: TeamMembersPageProps) {
  const { teamId } = await params

  // Step 1: get team info first
  const teamRes = await fetch(`https://api.example.com/teams/${teamId}`)
  const team = await teamRes.json()

  // Step 2: use team.leaderId from step 1
  const leaderRes = await fetch(`https://api.example.com/users/${team.leaderId}`)
  const leader = await leaderRes.json()

  return (
    <main>
      <h1>{team.name}</h1>
      <p>Team Leader: {leader.name}</p>
    </main>
  )
}
```

| Use sequential when | Use parallel when |
| ------------------- | ----------------- |
| Second fetch needs data from first | Requests are independent |
| Waterfall is unavoidable | You want fastest page load |

**Tip:** Restructure APIs or use `Promise.all` at the data layer when possible to avoid unnecessary waterfalls.

# 5. Fetch Caching in Next.js

Next.js extends the native `fetch` API with caching options.

```tsx
// Default — cached indefinitely (static)
const res = await fetch('https://api.example.com/categories')

// Time-based revalidation — ISR (revalidate every 60 seconds)
const res = await fetch('https://api.example.com/products', {
  next: { revalidate: 60 },
})

// Always fresh — no cache (dynamic)
const res = await fetch('https://api.example.com/cart', {
  cache: 'no-store',
})

// Tag-based revalidation (on-demand)
const res = await fetch('https://api.example.com/posts', {
  next: { tags: ['posts'] },
})
```

| Option | Behavior | Typical use |
| ------ | -------- | ----------- |
| Default (cached) | Cached until manually revalidated | Static content, rarely changing data |
| `next: { revalidate: N }` | Re-fetch every N seconds | Product catalogs, news feeds |
| `cache: 'no-store'` | Never cache | User cart, personalized data |
| `next: { tags: ['x'] }` | Cache with tag for on-demand purge | CMS content, admin updates |

# 6. Colocating Fetch Functions

Keep fetch logic in dedicated modules — not scattered inside components.

```tsx
// lib/products.ts
export interface Product {
  id: string
  name: string
  price: number
  category: string
}

const API_BASE = 'https://api.example.com'

export async function getProducts(): Promise<Product[]> {
  const res = await fetch(`${API_BASE}/products`, {
    next: { revalidate: 300, tags: ['products'] },
  })

  if (!res.ok) {
    throw new Error(`Failed to fetch products: ${res.status}`)
  }

  return res.json()
}

export async function getProductById(id: string): Promise<Product | null> {
  const res = await fetch(`${API_BASE}/products/${id}`, {
    next: { tags: ['products', `product-${id}`] },
  })

  if (res.status === 404) return null
  if (!res.ok) throw new Error(`Failed to fetch product ${id}`)

  return res.json()
}
```

```tsx
// app/products/page.tsx
import { getProducts } from '@/lib/products'

export default async function ProductsPage() {
  const products = await getProducts()

  return (
    <ul>
      {products.map((product) => (
        <li key={product.id}>
          {product.name} — Rs. {product.price}
        </li>
      ))}
    </ul>
  )
}
```

| Benefit | Explanation |
| ------- | ----------- |
| Reusability | Same function in page, layout, and Server Actions |
| Testability | Test fetch logic without rendering components |
| Caching control | Central place for `revalidate` and tags |

# 7. Request Memoization

Next.js automatically **deduplicates** identical `fetch` requests within a single render pass.

```tsx
// Both calls share the same in-memory result during one request
async function Header() {
  const user = await getUser('123') // fetch #1 — hits network
  return <nav>Hello, {user.name}</nav>
}

async function Sidebar() {
  const user = await getUser('123') // fetch #2 — deduplicated, no second network call
  return <aside>{user.role}</aside>
}

export default async function Layout({ children }: { children: React.ReactNode }) {
  return (
    <>
      <Header />
      <Sidebar />
      {children}
    </>
  )
}
```

| Layer | Scope | Duration |
| ----- | ----- | -------- |
| Request memoization | Single server render | One HTTP request lifecycle |
| Data Cache | Across requests | Until revalidation or expiry |
| Full Route Cache | Entire rendered page | Until revalidation |

# 8. Database Access (Beyond fetch)

Server Components can query databases directly — often better than wrapping DB in an API route.

```tsx
// lib/db/products.ts
import { db } from '@/lib/db'

export async function getFeaturedProducts() {
  return db.product.findMany({
    where: { isFeatured: true },
    take: 6,
    orderBy: { createdAt: 'desc' },
  })
}
```

```tsx
// app/page.tsx
import { getFeaturedProducts } from '@/lib/db/products'

export default async function HomePage() {
  const products = await getFeaturedProducts()

  return (
    <section>
      <h1>Featured Products</h1>
      <div className="grid grid-cols-3 gap-4">
        {products.map((product) => (
          <article key={product.id}>
            <h2>{product.name}</h2>
            <p>Rs. {product.price}</p>
          </article>
        ))}
      </div>
    </section>
  )
}
```

| fetch to external API | Direct DB query |
| --------------------- | --------------- |
| Good for third-party services | Good for your own database |
| HTTP overhead | Faster — no HTTP round-trip |
| Cacheable with Next.js fetch options | Use `unstable_cache` or ORM caching |

# 9. TanStack Query with Next.js (Client-Side)

Use TanStack Query when you need client-side caching, polling, or optimistic updates.

```tsx
'use client'

import { useQuery } from '@tanstack/react-query'

interface Notification {
  id: string
  message: string
  read: boolean
}

async function fetchNotifications(): Promise<Notification[]> {
  const res = await fetch('/api/notifications')
  if (!res.ok) throw new Error('Failed to load notifications')
  return res.json()
}

export function NotificationBell() {
  const { data, isLoading, error } = useQuery({
    queryKey: ['notifications'],
    queryFn: fetchNotifications,
    refetchInterval: 30_000, // poll every 30 seconds
  })

  if (isLoading) return <span>Loading...</span>
  if (error) return <span>Could not load notifications</span>

  const unreadCount = data?.filter((n) => !n.read).length ?? 0

  return (
    <button type="button" aria-label={`${unreadCount} unread notifications`}>
      🔔 {unreadCount > 0 && <span>{unreadCount}</span>}
    </button>
  )
}
```

| Server fetch | TanStack Query |
| ------------ | -------------- |
| Initial page load | After hydration, live updates |
| SEO-critical content | Dashboard widgets, notifications |
| No client JS cost | Adds client bundle, powerful cache |

# 10. Error Handling in Fetch

Always handle failed requests in Server Components.

```tsx
// app/products/page.tsx
import { getProducts } from '@/lib/products'

export default async function ProductsPage() {
  try {
    const products = await getProducts()

    if (products.length === 0) {
      return <p>No products available right now.</p>
    }

    return (
      <ul>
        {products.map((product) => (
          <li key={product.id}>{product.name}</li>
        ))}
      </ul>
    )
  } catch (error) {
    console.error('Products fetch failed:', error)
    throw error // Let error.tsx boundary handle UI
  }
}
```

| Strategy | When |
| -------- | ---- |
| `try/catch` + fallback UI | Expected failures (empty data) |
| `throw error` → `error.tsx` | Unexpected failures |
| Return `null` + `notFound()` | Resource doesn't exist |

# 11. Best Practices

✔ Fetch in Server Components by default — not in `useEffect`

✔ Use `Promise.all` for independent requests

✔ Colocate fetch functions in `lib/` modules

✔ Set explicit cache options — don't rely on defaults blindly

✔ Use `cache: 'no-store'` for personalized or real-time data

✔ Handle errors with `try/catch` or error boundaries

✔ Use TanStack Query only when client-side cache adds real value

# 13. Summary

| Pattern | Use case |
| ------- | -------- |
| Server Component fetch | Default for page and layout data |
| `Promise.all` | Independent parallel requests |
| Sequential `await` | Dependent data chains |
| `cache: 'no-store'` | Dynamic, user-specific data |
| `next: { revalidate: N }` | Time-based ISR |
| `next: { tags: [...] }` | On-demand revalidation |
| Direct DB access | Your own database — skip HTTP |
| TanStack Query | Client polling, optimistic UI |
